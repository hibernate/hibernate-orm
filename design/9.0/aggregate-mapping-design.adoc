= Aggregate Mapping Design
:toc:

This note describes a possible ORM 9 design direction for aggregate-valued
embeddable mappings such as SQL `STRUCT` / UDT, JSON, XML, and aggregate arrays.

The immediate trigger is `@Struct` parity in the boot-model pipeline, but the
larger goal is broader: mapping-model, runtime-model, binder, and extractor
changes are valid design tools in the explicit pipeline.  The goal should not
be to reproduce every historical
internal shape when a cleaner model would make aggregate mappings easier to bind,
validate, and execute.

== Problem Statement

The current support works, but aggregate embeddables are not modeled as a
first-class storage shape.

Today an aggregate component is effectively represented as:

* a `Component` with normal child properties;
* aggregate-specific state on that component, such as struct name, struct
  column ordering, and aggregate column;
* an `AggregateColumn` standing in for the physical aggregate value;
* a `UserDefinedObjectType` or equivalent database object for SQL `STRUCT` /
  UDT metadata;
* second-pass logic that validates the component, initializes aggregate child
  columns, and removes flattened child columns from the owner table.

That shape is compatible with existing mapping code, but it makes the aggregate
storage mode implicit.  Binding first creates something close to a flattened
component and then repairs the table/database-object model later.  Runtime code
must infer aggregate behavior from a combination of component state, aggregate
columns, dialect support, and SQL expression helpers.

This is especially awkward for plural aggregate mappings, nested aggregate
components, arrays, and non-STRUCT aggregate formats such as JSON or XML.

== Design Goals

Aggregate mappings should be explicit in the boot mapping model and runtime
model.

The design should:

* distinguish flattened component storage from aggregate component storage early
  in binding;
* avoid binding child columns into a physical table only to remove or relocate
  them later;
* support singular and plural aggregate-valued embeddables through the same
  conceptual model;
* keep SQL `STRUCT`, JSON, XML, and aggregate arrays related, but not force them
  into an abstraction that hides important differences;
* centralize dialect-specific aggregate read, write, assignment, and extraction
  behavior;
* preserve compatibility where practical, while allowing ORM 9 mapping and
  runtime model changes when they remove real complexity.

== Current Short-Term Path

The current boot-model pipeline can close parity gaps by adapting the existing
mapping model:

[source]
----
ComponentSource
    -> Component
    -> AggregateColumn
    -> ComponentBindingPhase.AggregateFinalization
    -> UserDefinedObjectType / aggregate metadata
----

This is a useful bridge.  It lets the pipeline prove source discovery,
categorization, binding, UDT metadata, and runtime compatibility without waiting
for a larger model redesign.

However, this should be treated as transitional.  The bridge is allowed to use
existing `Component` and aggregate-column behavior, but new design work should
not assume that compatibility projection and table mutation are the desired
final shape.

The current compatibility policy is to keep materializing the mapping shape that
legacy boot and runtime consumers already understand.  The new pipeline may
collect richer aggregate intent internally, but it should project that intent
into `Component`, `AggregateColumn`, component aggregate finalization, and
existing database aggregate metadata for now.

Explicit storage descriptors should not be attached directly to shared
`org.hibernate.mapping` contracts as parallel state.  Introduce one only when
the relevant runtime and schema consumers can treat it as authoritative.

Current bridge status:

* singular embedded `@Struct` attributes materialize the legacy-compatible
  `Component` / `AggregateColumn` / aggregate finalization shape;
* element-collection `@Struct` values use the same bridge, with the aggregate
  column on the collection table and plural struct SQL type defaults derived
  from the preferred array strategy;
* element-collection map-value and map-key aggregate values use the same
  bridge.  Map values reuse the collection-element source role and read
  value-side `@JdbcTypeCode` for JSON/XML aggregates.  Map keys use the map-key
  source role, read the aggregate column from `@MapKeyColumn`, and read
  key-side `@MapKeyJdbcTypeCode` for JSON/XML aggregates;
* nested `@Struct` values use the same bridge.  Nested component binding runs
  aggregate processing after nested member binding so the outer aggregate can
  treat the nested aggregate column as one ordered member;
* Java array-valued `@Struct` attributes use the same bridge.  Categorization
  treats the array attribute as an aggregate component source, the component
  binds against the array element embeddable type, and the aggregate column uses
  the plural struct SQL type/default dialect array type;
* singular JSON/XML aggregate embeddables use the same bridge when an
  embeddable-valued member declares `@JdbcTypeCode(SqlTypes.JSON)` or
  `@JdbcTypeCode(SqlTypes.SQLXML)`.  The JDBC type code supplies the aggregate
  column type instead of UDT metadata;
* nested JSON/XML aggregate embeddables use the same bridge.  Nested component
  binding applies aggregate processing to the inner JSON/XML component and
  carries the outer aggregate column as the nested component's parent aggregate
  column;
* element-collection JSON/XML aggregate embeddables use the same bridge.  The
  aggregate column is bound on the collection table and the member
  `@JdbcTypeCode(SqlTypes.JSON)` / `@JdbcTypeCode(SqlTypes.SQLXML)` supplies the
  aggregate column type;
* Java array-valued JSON/XML aggregate embeddables use the same bridge when the
  array member declares `@JdbcTypeCode(SqlTypes.JSON_ARRAY)` or
  `@JdbcTypeCode(SqlTypes.XML_ARRAY)`.  Categorization treats the array as an
  aggregate component source, the component binds against the array element
  embeddable type, and the aggregate column carries the array aggregate SQL type.
  Runtime-model smoke coverage verifies these mappings cross the
  `buildSessionFactory()` boundary after the new pipeline registers the same
  fallback JSON/XML array JDBC type constructors that legacy boot registers.
  XML array annotation/runtime coverage uses an aggregate-support-capable test
  dialect because H2's production aggregate support only handles JSON assignment
  expressions in this path;
* element-collection JSON/XML aggregate embeddables currently have boot-model
  bridge coverage only.  A runtime-model smoke attempt reaches
  `buildSessionFactory()` and exposes a `CollectionTableGroup` /
  `StandardVirtualTableGroup` recursion for aggregate collection elements.
  A historical pre-removal bootstrap characterization using the same mapping shape
  reaches the same recursion, so this is not tracked as a new-pipeline
  binding/materialization parity gap.  Any follow-up belongs at the runtime
  collection table-group/model boundary, not in source discovery or aggregate
  column materialization;
* the bridge still intentionally binds component members first and lets
  component aggregate finalization validate, order, contribute UDT metadata, and
  remove flattened member columns from the owner or collection table.

== Proposed Mapping Model

An embeddable/component should have an explicit storage descriptor.

Conceptually:

[source,java]
----
interface ComponentStorage {
    ComponentStorageKind kind();
}

enum ComponentStorageKind {
    FLATTENED,
    AGGREGATE
}

interface AggregateComponentStorage extends ComponentStorage {
    AggregateStorageKind aggregateKind();
    Column aggregateColumn();
    QualifiedName aggregateTypeName();
    List<AggregateMember> members();
}

enum AggregateStorageKind {
    STRUCT,
    JSON,
    XML,
    ARRAY
}
----

The exact type names are placeholders, but the split is important.  A `Component`
should be able to answer directly:

* whether its attributes are flattened into the owner table;
* whether the component is represented by a single aggregate column;
* which database aggregate type or format backs that column;
* what ordered aggregate members exist inside the aggregate value;
* where nested aggregate components appear.

This could eventually be implemented as a descriptor attached to `Component`
rather than a replacement for `Component`.  Replacing `Component` wholesale would
be far more invasive and is not necessary to get a clearer model.

That attachment should wait until the descriptor can be authoritative for the
runtime and schema consumers which need aggregate state.  Before that point,
descriptor-like state should remain an explicit pipeline contribution or
resolved plan projected into the current mapping shape.

=== Flattened Storage

Flattened storage is the current normal embeddable shape:

[source]
----
owner table
    id
    street
    city
    zip
----

The component's child selectables are physical selectables in the owner table or
collection table.

=== Aggregate Storage

Aggregate storage is a different shape:

[source]
----
owner table
    id
    address

address aggregate metadata
    street
    city
    zip
----

The owner table has the aggregate column.  The component's child selectables
describe members of the aggregate value rather than independent columns in the
owner table.

For SQL `STRUCT`, those members may also define UDT metadata.  For JSON/XML,
they may define path-based access metadata rather than a database UDT.  For
arrays, the aggregate storage descriptor also needs element aggregate metadata.

== Proposed Binder Shape

The binder should determine component storage before binding the component's
child attributes.

Current transitional shape:

[source]
----
bind component child columns into owner table
    -> discover aggregate column
    -> second pass initializes aggregate metadata
    -> remove child columns from owner table
----

Desired shape:

[source]
----
resolve component storage
    -> bind aggregate column into owner table or collection table
    -> bind aggregate members into aggregate-member container
    -> validate aggregate storage
    -> contribute database aggregate metadata when needed
----

The key idea is that child binding should target the right container from the
start.

For flattened components, the target is the physical owner table or collection
table.

For aggregate components, the target is an aggregate member container associated
with the component storage descriptor.  That container may later contribute SQL
`STRUCT` / UDT metadata, JSON path metadata, XML path metadata, or array element
metadata.

This avoids table-column cleanup and makes plural aggregate binding less
special.  A collection element embeddable can still bind its aggregate column to
the collection table while binding aggregate members to the aggregate member
container.

== Source Roles

The source model should keep the declaration site and the component type
separate.

Aggregate support needs to work for at least:

* singular embedded attributes;
* element-collection embeddable values;
* map-value embeddables;
* map-key embeddables;
* nested aggregate embeddables;
* aggregate arrays.

These are not all the same source role.  For example, a plural attribute may be
the declaration site for overrides, conversions, collection-table details, and
aggregate column details, while the component type supplies the embeddable member
model.

The binder should therefore receive a component source that can answer:

* what kind of source role is being bound;
* which member declared the aggregate mapping;
* which class supplies the embeddable members;
* which physical table receives the aggregate column;
* which logical aggregate member container receives child selectables.

== Runtime Model Direction

The runtime model should expose aggregate-valued embeddables directly.

Possible shape:

[source,java]
----
interface AggregateEmbeddableValuedModelPart {
    AggregateStorageKind aggregateKind();
    SelectableMapping aggregateSelectable();
    EmbeddableMappingType embeddableType();
    AggregateValueAccessStrategy accessStrategy();
}
----

The exact type names are placeholders.  The important bit is that SQL AST,
result assembly, mutation generation, dirty checking, and query path resolution
should be able to ask directly whether an embeddable value is aggregate-backed.

Runtime should not have to reverse-engineer aggregate-ness from child column
placement or special cases around `AggregateColumn`.

== Aggregate Access Strategy

Dialect-specific behavior should be centralized behind an aggregate access
strategy produced from the mapping model and dialect support.

The strategy should cover:

* reading an aggregate member from the aggregate column;
* assigning an aggregate member during mutation;
* replacing the whole aggregate value;
* constructing aggregate values when the database requires explicit
  constructors;
* extracting aggregate values as one JDBC value, when supported;
* extracting aggregate members as separate projections, when required;
* handling nested aggregates and arrays.

Existing `AggregateSupport` methods are a good foundation, but call sites should
ideally consume a runtime strategy rather than repeatedly combining component
state, column state, and dialect hooks.

== Format-Specific Concerns

SQL `STRUCT`, JSON, XML, and arrays are related, but they should not be treated
as identical.

SQL `STRUCT` / UDT::
Usually needs database aggregate type metadata, ordered members, dialect-specific
member read and assignment expressions, and sometimes explicit constructors.

JSON::
Usually needs path expressions, serialization/deserialization decisions, and
potentially whole-value replacement rather than member assignment.

XML::
Similar to JSON in being path-oriented, but with different dialect functions,
typing, and mutation support.

Aggregate arrays::
Need element aggregate metadata and may require special handling for collection
table values, array element access, and dialect array constructors.

The common abstraction should be "aggregate-valued embeddable storage", not
"everything is a STRUCT".

== Migration Strategy

This should be incremental.

1. Keep closing parity gaps with the existing `Component` / `AggregateColumn` /
   second-pass bridge.
2. Add internal contribution/resolution state only when the bridge needs
   whole-shape reasoning before projecting into the legacy mapping model.
3. Use that internal state to characterize aggregate source roles, validation,
   aggregate column placement, database-object contribution, and unsupported
   combinations.
4. Introduce a storage descriptor on `Component` or beside it only when the
   relevant runtime and schema consumers can make it authoritative rather than
   a parallel representation.
5. Teach the new binder to resolve component storage before child attribute
   binding.
6. Add an aggregate member container so aggregate child selectables are never
   temporarily bound as owner-table columns.
7. Build runtime aggregate access strategies from the explicit storage
   descriptor.
8. Move SQL generation, extraction, and mutation code to consume the runtime
   aggregate model directly.
9. Retire second-pass table mutation once all aggregate paths bind into their
   final containers.

The trigger for steps 4 through 9 is not simply that the new pipeline has a
working aggregate bridge.  The trigger is one of:

* a supported aggregate shape cannot be represented through the bridge without
  excessive table mutation, duplicated runtime inference, or unmaintainable
  side channels;
* runtime and schema consumers are deliberately migrated to a shared aggregate
  descriptor.

== Open Questions

These are stabilization questions.  A question becomes PR-blocking only when a
focused runtime reproducer cannot be expressed through the existing aggregate
intent/member-container handoff.  See
link:design-doc-audit.adoc#_open_question_routing[
the centralized scope routing].

Should aggregate storage be represented as a descriptor attached to `Component`,
or as a distinct mapping value type?

This should remain an explicit pipeline design question rather than parallel
state attached to shared mapping contracts.  The safe near-term answer is to
project into the existing `Component` / `AggregateColumn` shape until runtime
and schema consumers can make a descriptor authoritative.

Can `UserDefinedObjectType` remain the right database metadata object for SQL
`STRUCT`, or should aggregate member metadata move behind a more general
database aggregate metadata contract?

How much aggregate finalization can become normal binder validation once source
roles and aggregate member containers are explicit?

Should JSON/XML aggregate mappings share the same component storage descriptor
as SQL `STRUCT`, or should the common contract be narrower with
format-specific subtypes?

What is the runtime boundary between aggregate value assembly and embeddable
value assembly?

How should partial aggregate mutation interact with dynamic update, dirty
tracking, generated columns, and optimistic locking?

How should nested aggregate values be represented in SQL AST paths?

== Follow-Up Work

The complete suite is green with the current compatibility bridge.  Future
aggregate work should stay explicit about what a change proves:

* singular embedded `@Struct` support: covered by the current bridge;
* plural `@Struct` support for element-collection values: covered by the
  current bridge for collection elements;
* map-value and map-key `@Struct` aggregate embeddables: covered by the current
  bridge for element-collection values and map keys;
* map-value and map-key JSON/XML aggregate embeddables: covered by the current
  bridge for element-collection values and map keys;
* nested `@Struct` aggregate embeddables: covered by the current bridge;
* Java array-valued `@Struct` aggregate embeddables: covered by the current
  bridge;
* singular JSON/XML aggregate embeddables: covered by the current bridge;
* nested JSON/XML aggregate embeddables: covered by the current bridge;
* plural JSON/XML aggregate embeddables: covered by the current bridge for
  element-collection values at the boot-model boundary; any future runtime
  collection element work should start from a fresh focused reproducer rather
  than reopening the old boot-pipeline parity bucket;
* Java array-valued JSON/XML aggregate embeddables: covered by the current
  bridge and by focused runtime-model smoke coverage;
* runtime mapping and SQL generation tests that assert observable behavior, not
  just successful metadata creation.

Each parity slice should also record whether it reinforces the current bridge or
exposes a need for the proposed first-class aggregate storage model.  Exposing
that need does not by itself mean attaching descriptors to `Component`; it may
first mean adding internal resolved aggregate intent that still materializes the
legacy-compatible mapping shape.

== Implementation Notes

An attempted shortcut of suppressing owner-table registration for new-pipeline
aggregate member columns while still projecting directly into the historical
`Component` / `AggregateColumn` shape was not viable.  Metadata-only aggregate
coverage still passed, but JSON aggregate-array runtime probes
(`JsonEmbeddableArrayTest` and `JsonWithArrayEmbeddableTest`) exposed
ClassCastException failures during value binding: converter paths saw a
`Gender[]` where the element converter expected `Gender`, and the aggregate
array path saw `EmbeddableAggregate` where an enum member was expected.  The
same focused run also showed JSON aggregate check rendering leaking the
historical `@` placeholder into H2 DDL.  That confirmed the bridge's table-column
side effects are entangled with aggregate-array type resolution, member check
expression rendering, custom read/write expression setup, and schema check
rendering.

That result led to the current internal aggregate member container and explicit
aggregate value/type-resolution plan.  Only after aggregate members have a real
non-table home should the new pipeline stop registering those members as
owner-table columns.  A callback/finalizer or a boolean table-registration flag
is just another second pass and does not solve the model problem.

Follow-up work on 2026-06-27 narrowed the JSON aggregate-array runtime bucket.
The schema/exporter side should use the physical `AggregateColumn` type to
detect aggregate arrays; resolving array-ness through the normalized aggregate
`BasicValue` can hide `JSON_ARRAY`/`XML_ARRAY` behind the scalar aggregate
descriptor used for member materialization.  Explicit converters on Java array
members must also leave the converter's element domain visible so the existing
`BasicPluralJavaType` element-converter branch can build the converted array
type.

The earlier `JsonEmbeddableArrayTest` failures were not another member-column
registration issue.  They showed that a Java array of aggregate embeddables
could not be fetched as a normal aggregate embeddable with nested member
fetches: the initializer treated the aggregate array as one component and
indexed array elements as if they were component member values.  The runtime
model needed an aggregate-array route that assembled the aggregate column as a
basic plural aggregate value, while still exposing component-member paths where
HQL supports aggregate array indexing.

The first explicit runtime handoff now follows that direction.  Entity-level
aggregate arrays use the normalized aggregate `BasicPluralType` and
`BasicPluralJavaType` for SQM/JPA metamodel typing, domain results, SQL
selection, and fetch assembly.  Scalar aggregate embeddables continue through
the decomposed aggregate embeddable initializer path.  The embedded component is
still retained as the aggregate array element descriptor, but the aggregate
array value itself is no longer forced through the historical `Object[].class`
double meaning.

Follow-up work introduced the first internal binding-side version of that same
idea: `AggregateValuePlan` resolves the aggregate column storage code, the
normalized aggregate value JDBC code, the explicit aggregate Java type decision,
and the logical `AggregateMemberContainer` before the binder projects those
facts into `Component`, `BasicValue`, `AggregateColumn`, and component aggregate
finalization.  This does not yet move child binding into a real non-table member
container; it names that target and removes another layer of duplicated inline
aggregate type-resolution logic from the binder.  The next design step is to
make component-member binding consume the logical container instead of
temporarily registering aggregate members as owner-table columns.

The next bridge slice introduced `ComponentMemberTarget` and threaded it through
component-member binding.  Normal components use a table target.  Aggregate
components use an aggregate-member target carrying the logical
`AggregateMemberContainer`, while still delegating physical member
materialization to the same table path.  `AggregateComponentBinder` now consumes
that target for both the aggregate member container and the table used by the
legacy aggregate bridge.  This is intentionally behavior-preserving groundwork
for making member materializers target-aware.

The basic-value member materializer now consumes the same
`ComponentMemberTarget` when creating component-member `BasicValue` instances
and resolving explicit/implicit member tables.  It still chooses the legacy
physical table for actual column binding; the important handoff change is that
materialization can now branch on the logical aggregate-member target without
changing the public `ComponentBinder` traversal shape again.

The next small step made that branch observable: aggregate-target basic member
columns are registered with `AggregateMemberContainer` as logical aggregate
members.  They are no longer registered as physical table columns in the new
pipeline.  Runtime/schema behavior is intentionally unchanged because
component aggregate finalization consumes the aggregate member list and still
handles bridge cleanup, but basic aggregate leaves no longer exist as temporary
table columns.

Aggregate finalization now accepts that container from the new binding pipeline.
For new-pipeline aggregates it uses the container-owned member columns when the
container is complete, while ordering them by the sorted
`Component#getAggregatedColumns()` view so the historical UDT/schema column
order is preserved.  The finalizer falls back to the component graph if the
container does not describe the same aggregate member columns.  Nested aggregate
columns are also kept in the aggregate-member container instead of being
registered with the physical table; only root aggregate columns still register
with the table.
