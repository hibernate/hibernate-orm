= Annotation Coverage Audit
:toc:

This document tracks runtime annotation coverage for annotations declared in
`jakarta.persistence` and `org.hibernate.annotations`.  It intentionally
excludes `org.hibernate.annotations.processing`, which belongs to compile-time
processor coverage rather than native bootstrap/model-building coverage.

The goal is to make the new categorization/binding pipeline auditable at the
annotation level: whether an annotation is visible to the source model, whether
categorization or binding consumes it, whether XML/source support exists where
relevant, whether `DialectOverride` is honored, and whether focused tests assert
the resulting `org.hibernate.mapping` state.

== Classification

`covered`::
The new pipeline has descriptor, categorization/binding support, and focused
tests.

`test-only gap`::
The annotation appears to be handled or partially handled, but still needs a
focused model-binding assertion before it can be closed.

`categorization gap`::
The annotation is visible to the source model, but categorization does not yet
route it into the right metadata shape.

`binding gap`::
The annotation is visible and may be categorized, but binding does not yet apply
the expected `org.hibernate.mapping` effect.

`XML/source gap`::
The annotation has a corresponding XML/source vocabulary item that is not yet
processed into the new annotation model.

`dialect-override gap`::
The override annotation has descriptor support, but the new pipeline has not
yet been verified to resolve the effective annotation for the active dialect.

`deferred`::
The annotation belongs to a larger feature area that should be handled in its
own slice rather than folded into low-risk binding parity.

`not applicable/nested`::
The annotation is a repeatable container, nested helper, meta-annotation, or
annotation member value consumed through another annotation.

== JPA Initial Findings

This audit is against the Jakarta Persistence API version used by the build:
`jakarta.persistence-api` `4.0.0-M4`.

* `JpaAnnotations` has descriptors for the JPA mapping annotations consumed by
  the model pipeline, including newer JPA 4.0 annotations such as
  `@CheckConstraint`, `@EnumeratedValue`, `@Fetch`, `@NamedStatement`,
  `@NamedNativeStatement`, `@EntityListener`, `@ExcludedFromVersioning`, and
  `@Discoverable`.
* `JpaAnnotations` is not exhaustive for every annotation type in the
  `jakarta.persistence-api` jar.  Comparing against the `4.0.0-M4` jar found
  100 registered JPA annotation descriptors and 18 additional annotation types
  not currently registered.  Of those descriptor gaps, only the new lifecycle
  callback annotations are in scope for the boot-model pipeline.
* The core mapping vocabulary has focused mapping-state assertions for the
  basic table, column, discriminator, identifier, secondary-table, LOB,
  inheritance, version, access, and cacheability surface.
* The JPA modeling annotations in scope for the boot-model pipeline have
  descriptor/source support and focused categorization/binding coverage.

== Hibernate Initial Findings

* `HibernateAnnotations` covers most runtime Hibernate annotations.
* `Audited`, `Audited.*`, `Changelog`, and `Changelog.*` are not registered in
  the new annotation descriptor registry.  Legacy binder support exists, but
  new-pipeline support should be deferred to a state-management/audit slice.
* `DialectOverrideAnnotations` registers the override annotation descriptors and
  `DialectOverrider` supplies matching logic.  The new pipeline has a shared
  effective-annotation lookup used by the supported override-capable binders.
* Several annotations have descriptor support and legacy binder support, but no
  verified new-pipeline binding assertion yet.  These are the best candidates
  for low-risk parity slices.

such as `@IdGeneratorType`, `@AttributeBinderType`, `@TypeBinderType`,
`@ValueGenerationType`, `@DialectOverride.Version`,
`@DialectOverride.OverridesAnnotation`
| Covered: verified through the owning annotation or extension mechanism rather
than as independent mapping annotations.  `@AttributeBinderType` and
`@TypeBinderType` have focused extension-mechanism tests in the new binding
pipeline, and `@IdGeneratorType` is verified through a custom
identifier-generator meta-annotation.

| @(Embeddable)Instantiator
| `@Instantiator`, `@EmbeddableInstantiator`
| Deferred until hibernate-models exposes constructor annotation targets
(https://github.com/hibernate/hibernate-models/issues/243[hibernate-models#243],
planned for 1.3.0).  `@EmbeddableInstantiator` has binding coverage today, but
should be revisited in this slice so both embeddable-instantiation annotations
are sourced through hibernate-models rather than reflection/ad hoc lookup.

| Struct/aggregate mapping
| `@Struct`
| Partially covered.  Singular embedded `@Struct` mappings are bound to an
aggregate column and UDT metadata in the new pipeline.  Plural, array, and
deeper nested aggregate parity is intentionally deferred until the aggregate
mapping model, runtime model, binder, and extractor design is revisited; see
link:aggregate-mapping-design.adoc[`aggregate-mapping-design.adoc`].

| State management
| `@Audited`, `@Audited.Excluded`, `@Audited.Table`,
`@Audited.SecondaryTable`, `@Audited.SecondaryTables`,
`@Audited.CollectionTable`, `@Changelog`, `@Changelog.ChangesetId`,
`@Changelog.Timestamp`, `@Changelog.ModifiedEntities`, `@Temporal`,
`@Temporal.HistoryPartitioning`, `@Temporal.HistoryTable`,
`@Temporal.Excluded`
| Deferred.  Audit/changelog and temporal history annotations need a dedicated
state-management design and binding slice.
