* `copyForSameApplication()` is only an alias of the same occurrence;
* property-selection views are transient, roleless, and must not be attached
  to the mapping graph;
* semantic correspondence must use roles, not mapping-object identity.

== Background Branch Theme

The branch moves ORM bootstrap away from direct, side-effect-heavy
`org.hibernate.mapping` construction and toward an explicit boot/mapping
pipeline:

* source discovery, XML overlay, categorization, binding, views, and
  materialization are separate phases;
* semantic binding facts are recorded before compatibility
  `org.hibernate.mapping` objects are populated;
* binding captures source-aware facts and resolved intents, while
  materialization owns `org.hibernate.mapping` creation and mutation;
* views over the binding model support mapping-object materialization and JPA
  static metamodel planning;
* hidden key creation and ordering side effects are being replaced by explicit

== Current BasicValue Resolution State

`BasicValue` is being turned back into a simple compatibility mapping object.
It no longer self-resolves through `InFlightMetadataCollector` and no longer
stores deferred resolution callbacks.

Current code direction:

* `InFlightMetadataCollector#registerValueMappingResolver(...)` and collector
  value-resolver execution were removed.
* `BasicValue#resolve()` now requires an already-applied
  `BasicValue.Resolution`; it no longer calculates one internally.
* `BasicValue#applyResolution(...)` applies and finalizes an externally built
  resolution.
* `BasicValue` no longer carries the old `Function<TypeConfiguration, ...>`
  hooks for explicit Java type, explicit JDBC type, explicit mutability plan,
  or implicit Java type.
* `BasicValueResolutionBuilder` is the materializer/binder-side resolution
  engine.  Its `Input` object carries explicit facts such as Java type, JDBC
  type, mutability plan, temporal precision, enum style, converter, configured
  JDBC type code, source kind, and source member.
* `BasicValueResolutionBuilder.BasicValueRole` drives role-specific annotation
  lookup such as standard attribute, map key, list index, collection id,
  any-key, discriminator, state-management, generic declaration, and version
  handling.
* `BasicValueSourceBinder`, `BasicValueMappingMaterializer`,
  `IdentifierMappingMaterializer`, `VersionMappingMaterializer`,
  `AttributeOptionsMappingMaterializer`, aggregate binding, temporal helper,
  audit helper, and soft-delete helper now build/apply explicit resolution
  input instead of asking `BasicValue` to self-resolve.
* `DependantBasicValue` follows the referenced value's applied resolution
  rather than depending on copied implicit type callbacks.

Remaining compatibility island:

* Older resolver internals such as `InferredBasicValueResolver` and
  `NamedConverterResolution` still use
  `Function<TypeConfiguration, MutabilityPlan<?>>` parameters internally.
  Those are local resolver mechanics, not stored `BasicValue` state.

Current status:

* Basic-value resolution determination is considered externalized.  Do not
  treat this as an open design bucket unless a concrete call path is found that
  still asks `BasicValue` to determine its own resolution.
* `BasicValue#resolve()` is now an enforcement/finalization bridge: it requires
  an already-applied `BasicValue.Resolution` and fails if one was not provided.

Suggested next step in this area:

* Avoid adding new `BasicValue` setters that encode resolution behavior.  New
  resolution decisions should be represented as
  `BasicValueResolutionBuilder.Input` facts or role-specific builder logic.

=== Context Taxonomy

Use this taxonomy when deciding whether a remaining dependency should stay,
move, or become a phase-specific contract:

`BootstrapContext`::
Compatibility root and temporary bootstrap resource holder.  It should own JPA
temporary class-loader lifecycle, bootstrap release hooks, and extension-facing
compatibility state until those contracts have replacements.

`MappingResolutionOptions`::
Resolved instructions for metadata building: mapping defaults, model
preferences, XML enablement, CDI-extension policy, naming/cache choices, and
other plan inputs.

`MappingResolutionServices`::
Named service access needed while building metadata: standard service registry
bridge, configuration service, class loading, temp-class-loader-aware
`ClassLoaderAccess`, managed beans, custom type production, `ModelsContext`,
`TypeConfiguration`, JDBC services, changeset coordination, and strategy
selection.

`MetadataBuildingContext`::
Current metadata-build operation state: contributor, effective defaults,
metadata collector, object-name normalizer, type-definition registry, JPA
compliance, and current build decisions such as audit and temporal-table
strategy.

Phase-specific contexts::
Narrow adapters for one phase, such as `BindingContext`,
`XmlDocumentContext`, discovery contexts, materialization contexts, and
runtime-model creation contexts.  These should expose only the inputs their
phase naturally consumes.

The intent is not to move `BootstrapContext`'s broad surface onto
`MetadataBuildingContext`.  If a dependency is only needed by XML, source
discovery, binding, materialization, or runtime-model creation, prefer a phase
context.  If it is a service needed across metadata building, prefer
`MappingResolutionServices`.  If it is current build state or a resolved build
decision, prefer `MetadataBuildingContext` or `MappingResolutionOptions`.

== Hidden Mapping Side-Effect Hook Audit

The focused design source for this topic is
link:mapping-key-creation-design.adoc[`mapping-key-creation-design.adoc`].
Its current status is: materializer bridge drain complete; deprecated
compatibility helpers retained.

The phrase "hidden side-effect hook" should mean concrete mapping-object
methods that create or mutate relational products indirectly from mutable
`org.hibernate.mapping` state.  It should not be used as vague shorthand for
unfinished bootstrap work.

Current retained candidates:

* `Collection#createPrimaryKeyIfNeeded()`;
* `Collection#createForeignKeys()`;
* `PersistentClass#createPrimaryKey()`;
* `Value#createForeignKey()`;
* `Value#createUniqueKey(MetadataBuildingContext)`;
* `KeyValue#createForeignKeyOfEntity(...)`;
* `Join#createForeignKey()`.

Related low-level table registry methods:

* `Table#createForeignKey(...)` remains the primitive used by
  `ForeignKeyMappingMaterializer` and should not be deleted as part of hidden
  helper cleanup.
* `Table#createUniqueKey(...)` is still a compatibility helper around naming
  and table unique-key registration; audit callers before deciding whether to
  delete, narrow, or keep it.
* `DenormalizedTable#createForeignKeys(...)` still copies included-table
  foreign keys during metadata finalization and should be evaluated separately
  from boot hidden-helper cleanup.
* `InFlightMetadataCollectorImpl` still calls `Table#createForeignKeys(...)`
  while finalizing table foreign keys.  That makes table-level FK finalization a
  live metadata-finalization concern, not the same thing as the deprecated
  value/collection/join helper hooks.

Next implementation direction:

* delete or sharply narrow deprecated hidden helpers whose ORM boot callers have
  been drained;
* keep low-level table registry primitives only where explicit materializers
  still need them;
* if a helper remains for external compatibility, mark the reason and verify it
  delegates to explicit materializer products rather than re-deriving mapping
  facts.

== Current Mapped-Superclass Handoff State

The active mapped-superclass/generic handoff shape is documented in
link:applied-mapped-superclass-handoff-design.adoc[`applied-mapped-superclass-handoff-design.adoc`]
and
link:boot-model-terminology.adoc[`boot-model-terminology.adoc`].

Current code direction:

* `AttributeUsageBinding` is the applied attribute usage object.  The removed
  `AppliedMappedSuperclassAttribute` wrapper should not be reintroduced unless
  it carries genuinely new semantics.
* `MappedSuperclassContribution` owns contribution provenance and the ordered
  list of applied usages.
* `BootBindingModel` owns the global `MappingRole -> AppliedAttributeMapping`
  index and keeps mapped-superclass contribution provenance as an ordered list
  of applied usages.
* `MappedSuperclassHandoffResolver` is exposed through
  `RuntimeModelCreationContext` and is the runtime/JPA metamodel bridge from
  legacy `Property` inputs to applied boot-model metadata.  It uses the
  property's intrinsic role or, for declaration-side properties, derives the
  applied role from the nearest entity consumer.
* The former `MappedSuperclassPropertyHandoff`, its `BindingState` identity
  map, and the parallel entity-alias/attribute-name index have been removed.
* `BootBindingModel` owns the applied embeddable mappings indexed by intrinsic
  `MappingRole`.  The former `Component` identity handoff and its transitional
  `BindingState` API have been removed.
* `EmbeddableHandoffResolver` is exposed through
  `RuntimeModelCreationContext` and is the runtime/JPA metamodel bridge from a
  legacy `Component + Property` pair to the applied `AttributeUsageBinding`.
* `Property#isGenericSpecialization()` is localized fallback bridge state.
  Runtime/JPA metamodel code should ask the mapped-superclass or embeddable
  handoff resolver for semantic member/type/classification decisions instead of
  checking the flag directly.  Resolver fallbacks should run only when no
  applied handoff metadata is available.
* `Property#getMemberDetails()` remains the source-member compatibility bridge,
  but applied usage metadata should win when available.

Known next modeling gap: embeddable concrete-generic classification now treats
the role-indexed applied `AttributeUsageBinding` as authoritative when an
applied embeddable mapping exists, but it still retains copied-property
fallbacks for compatibility paths not yet backed by applied mapping data.  The
next cleanup should
identify those remaining unbacked paths and either model them as applied
handoffs or keep them explicitly marked as legacy compatibility paths.

=== Generic Fallback Audit

The current fallback paths break down as follows:

* Ordinary mapped-superclass attributes are backed by
  `AttributeUsageBinding`.  `MappedSuperclassHandoffResolver` should treat the
  applied usage as authoritative when found and use
  `Property#isGenericSpecialization()` only when no applied handoff exists.
* Embeddable members are backed by role-indexed `AppliedEmbeddableMapping` and
  `AppliedAttributeMapping` products.  `EmbeddableHandoffResolver` should treat
  the applied usage as authoritative when found and use copied-property or
  mapped-superclass generic markers only when no applied mapping exists.
* Generic mapped-superclass identifiers are a distinct handoff path.  The
  legacy runtime metamodel receives identifiers through `PersistentClass`
  identifier slots rather than ordinary declared-property iteration.
  `IdentifierHandoffResolver` now centralizes the classification decision and
  prefers `EntityIdentifierBinding` metadata, including the top-level
  identifier member for basic and aggregated ids.  When a `BootBindingModel` is
  available, the resolver treats identifier binding metadata as authoritative.
  The resolver-local `mappedSuperclassIdentifier.isGeneric()` fallback is now
  limited to legacy SessionFactory construction paths that do not yet hand a
  `BootBindingModel` to runtime/JPA metamodel creation.
* Low-level mapping compatibility still uses
  `Property#isGenericSpecialization()` in `MappingHelper` to avoid duplicating
  columns/selectables for copied concrete generic properties.  That is not a
  runtime/JPA metamodel classification path.

Follow-up options:

* route the remaining legacy SessionFactory construction path through
  `SessionFactoryConstructionPlan` with a populated `BootBindingModel`
  so `IdentifierHandoffResolver` can remove its no-boot-model generic
  identifier `Property` fallback;
* keep `MappingHelper` as a documented legacy mapping-model compatibility use;
* avoid adding new direct `Property#isGenericSpecialization()` checks outside
  resolver or low-level mapping compatibility code.
