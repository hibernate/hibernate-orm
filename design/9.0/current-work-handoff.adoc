= ORM 9 Current Work Handoff

This note is a compact handoff for continuing the ORM 9 boot-model /
binding-model work in a fresh context.
It records the current branch intent, recently completed slices, deliberate
transitional bridges, useful guardrails, and the next likely migration slices.

Read this first when picking up the current branch, then follow the linked
design documents for detail.

== Current Branch Theme

The active work is moving ORM bootstrap away from direct, side-effect-heavy
`org.hibernate.mapping` construction and toward an explicit boot/mapping
pipeline:

* source discovery, XML overlay, categorization, binding, views, and
  materialization are separate phases;
* semantic binding facts are recorded before compatibility
  `org.hibernate.mapping` objects are populated;
* the contributor guide now defines a bootstrap phase contract: binding captures
  source-aware facts and resolved intents, while materialization owns
  `org.hibernate.mapping` creation/mutation;
* views over the binding model are intended to support more than one consumer,
  including mapping-object materialization and JPA static metamodel planning;
* hidden key creation and ordering side effects are being replaced by explicit
  materializers and correspondence products.

== Recently Completed Direction

SessionFactory and JPA bootstrap have already moved substantially toward the
new pipeline:

* `EntityManagerFactoryBuilder` and `SessionFactoryBuilder` public staged
  bootstrap surfaces were removed.
* `SessionFactoryProducer` / construction-request based custom factory
  production is the intended custom `SessionFactory` hook.
* `SessionFactoryImpl` construction now has internal preparation products, but
  much constructor work intentionally remains in place.
* `MetadataBootstrap` / `SessionFactoryBootstrap.resolveMetadata(...)` exist as
  transitional metadata-resolution products.

Binding/model work has also moved forward:

* the internal binding model package was moved under `org.hibernate.boot.mapping`;
* binding-model declarations/usages now distinguish direct attributes,
  mapped-superclass declarations, and embeddable-site usages;
* `ValueIntent` covers singular basic, embedded, to-one, any, and plural
  collection-valued attributes;
* identifier binding has an identifier materializer path for the important
  simple, embedded, IdClass, association-id, and derived-id shapes;
* JPA static metamodel work now has view-backed planning pieces, though
  injection is still partly transitional.

== Current Uncommitted Slice

The active uncommitted slice has moved from the original generic-inheritance
bucket into binding/materialization parity buckets found by broader suite
sampling.  The most recent completed bucket is schema/exportables parity:
secondary-table and collection-table key details, joined-subclass/table
validation, check/unique/index/comment/options metadata, derived-id column
metadata, and quoted-name behavior.

Important recent code changes in this slice:

* Entity-level Hibernate `@Check` handling moved out of direct table binding and
  into `EntityTypeBinder` / `PersistentClass` check metadata, so check
  constraints can be routed later by referenced columns.  This fixes secondary
  table check placement.
* Column-level/member-level `@Check` metadata is carried through
  `BasicValueIntent` and applied during basic value materialization.
* JPA unique constraints with missing or empty column names now fail during table
  binding instead of being silently ignored.  The same validation exists for
  collection/association table keys.
* Explicit single-table discriminators are materialized even when the hierarchy
  has no subclasses, and `DiscriminatorType.CHAR` normalizes length to `1`.
* Implicit to-one join columns and derived-id `@MapsId` join columns copy
  relevant referenced-column metadata such as length/sql type while preserving
  local nullability.
* Derived-id primary-key column ordering now places derived association columns
  before remaining id columns, matching legacy DDL/PK ordering expectations.
* `AbstractDelegatingMetadata#orderColumns(boolean)` now passes the caller's
  `forceOrdering` flag to the delegate instead of always passing `false`.
* `@TimeZoneColumn` now carries `options` and `comment` into the materialized
  column metadata.
* Quoted JPA entity names are now rejected during entity skeleton binding.
  Backtick-quoted and escaped double-quoted `@Entity(name=...)` values are
  illegal; quoted database identifiers must be expressed through table/column
  annotations.
* `SchemaUpdateWithUseJdbcMetadataDefaultsSettingToFalseAndQuotedNameTest` now
  uses explicit `@Table(name = "`Another_Test_Entity`")` for the quoted table
  scenario and has explicit negative coverage for quoted entity names.
* SINGLE_TABLE subclasses with direct `@Table` now fail during
  `EntityTypeBinder#bindTables()`, before subclass table metadata can be
  silently ignored.

The generic-inheritance bridge work is still present and still transitional:

* `BasicValueSource` / `BasicValueMappingMaterializer` can carry resolved
  generic usage types for inherited basic attributes.
* `MappedSuperTypeBinder` and `EntityTypeBinder` still bridge legacy generic
  declaration/specialization behavior through compatibility mapping objects.
* link:generic-inheritance-model-design.adoc[`generic-inheritance-model-design.adoc`]
  remains the design boundary: avoid more ad hoc generic runtime behavior until
  a proper declaration/usage/contribution model or shared runtime adapter is
  available.

The latest focused aggregate/materialization bucket is green:

* plural JSON/XML/STRUCT aggregate embeddable attributes now hand off both facts
  needed by legacy runtime model creation: the owning property remains backed by
  the aggregate `BasicValue`, while the aggregate `AggregateColumn` retains the
  element `Component` used to build embeddable mapping metadata;
* plural basic aggregate attributes such as `List<Embeddable>` with
  `@JdbcTypeCode(JSON_ARRAY)` now create element component metadata instead of
  relying only on an `EmbeddableAggregateJavaType`;
* plural aggregate `BasicValue` resolution no longer forces the element Java
  type onto the owning plural property, allowing arrays/collections to resolve
  from the reflected member type and explicit aggregate JDBC code;
* entity basic-attribute materialization now uses the aggregate-column
  `BasicType` returned by `resolveAggregateColumnBasicType(...)` when available;
* the JPA metamodel aggregate-element path now checks that the owning selectable
  is an `AggregateColumn` before using its component metadata.

== Deliberate Transitional Bridges

Do not treat these as accidental leftovers:

* `org.hibernate.mapping` remains the compatibility materialization output for
  now.
* Binding-model contracts should stay mapping-model-free, but materializers and
  compatibility registries may still hold `PersistentClass`, `Component`,
  `Property`, `Value`, and `Column`.
* `BindingStateImpl` still owns compatibility registries for mapping objects and
  pending work.
* Mapped-superclass and embeddable contribution/view designs are not fully
  implemented.
* `Table#createForeignKey(...)` is not deprecated yet because it is still the
  low-level table registry primitive for explicit FK creation.
* `sortProperties()`, `sortColumns(...)`, and similar ordering APIs still exist;
  the target direction is to reduce destructive sorting after explicit
  correspondence/order products cover enough paths.

== Fresh-Context Cautions

* Do not "fix" runtime entity-key or initializer behavior to compensate for
  mapping shapes that the old binding process produced correctly.  Most recent
  id/derived-id regressions were binding/materialization shape issues.
* Be careful with quoted identifiers.  Use `Identifier#matches(...)` semantics
  where identifier-aware comparison is needed; do not introduce broad
  case-insensitive string matching.
* Quoted JPA entity names are illegal in this branch.  Use `@Table`, `@Column`,
  and other database-identifier annotations for quoted database names.
* Avoid reintroducing builder surfaces.  The design direction is explicit
  pipeline products, construction requests, producers, views, and
  materializers.
* Keep new binding-model contracts free of `org.hibernate.mapping` types unless
  the contract is explicitly a materializer or compatibility bridge.
* When adding new Java files, include `@author Steve Ebersole`.
