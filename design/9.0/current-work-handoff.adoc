== Current Branch Theme

The branch moves ORM bootstrap away from direct, side-effect-heavy
`org.hibernate.mapping` construction and toward an explicit boot/mapping
pipeline:

* source discovery, XML overlay, categorization, binding, views, and
  materialization are separate phases;
* semantic binding facts are recorded before compatibility
  `org.hibernate.mapping` objects are populated;
* binding captures source-aware facts and resolved intents, while
  materialization owns `org.hibernate.mapping` creation and mutation;
* views over the binding model support mapping-object materialization and JPA
  static metamodel planning;
* hidden key creation and ordering side effects are being replaced by explicit
  materializers and correspondence products.

== Current BasicValue Resolution State

`BasicValue` is being turned back into a simple compatibility mapping object.
It no longer self-resolves through `InFlightMetadataCollector` and no longer
stores deferred resolution callbacks.

Current code direction:

* `InFlightMetadataCollector#registerValueMappingResolver(...)` and collector
  value-resolver execution were removed.
* `BasicValue#resolve()` now requires an already-applied
  `BasicValue.Resolution`; it no longer calculates one internally.
* `BasicValue#applyResolution(...)` applies and finalizes an externally built
  resolution.
* `BasicValue` no longer carries the old `Function<TypeConfiguration, ...>`
  hooks for explicit Java type, explicit JDBC type, explicit mutability plan,
  or implicit Java type.
* `BasicValueResolutionBuilder` is the materializer/binder-side resolution
  engine.  Its `Input` object carries explicit facts such as Java type, JDBC
  type, mutability plan, temporal precision, enum style, converter, configured
  JDBC type code, source kind, and source member.
* `BasicValueResolutionBuilder.BasicValueRole` drives role-specific annotation
  lookup such as standard attribute, map key, list index, collection id,
  any-key, discriminator, state-management, generic declaration, and version
  handling.
* `BasicValueSourceBinder`, `BasicValueMappingMaterializer`,
  `IdentifierMappingMaterializer`, `VersionMappingMaterializer`,
  `AttributeOptionsMappingMaterializer`, aggregate binding, temporal helper,
  audit helper, and soft-delete helper now build/apply explicit resolution
  input instead of asking `BasicValue` to self-resolve.
* `DependantBasicValue` follows the referenced value's applied resolution
  rather than depending on copied implicit type callbacks.

Remaining compatibility island:

* Older resolver internals such as `InferredBasicValueResolver` and
  `NamedConverterResolution` still use
  `Function<TypeConfiguration, MutabilityPlan<?>>` parameters internally.
  Those are local resolver mechanics, not stored `BasicValue` state.

Suggested next step in this area:

* Continue retiring legacy basic-value/binder compatibility paths and move any
  remaining type-resolution decisions into `BasicValueResolutionBuilder` input
  facts or role methods.  Avoid adding new `BasicValue` setters that encode
  resolution behavior.

== Current Bootstrap / SessionFactory Construction State

SessionFactory construction has been partially split into named inputs and
products.  This work is intentionally still in flight: the current goal is to
make phase boundaries explicit enough that remaining constructor-time and
`BootstrapContext` dependencies are visible and can be narrowed.

Current code direction:

* `SessionFactoryAccess` / delayed SessionFactory reference work lets runtime
  products store a factory reference without forcing all preparation code to
  call back into an incomplete `SessionFactoryImpl`.
* `RuntimeModelCreationContext#getSessionFactory()` remains for true factory
  storage/runtime dependencies, but easy construction-time reads have been
  migrated to narrower accessors such as wrapper options, SQL string generation
  context, domain model, and changeset coordinator.
* `FetchProfileRegistry` replaced the earlier ambiguous in-flight fetch-profile
  map passed through SQL translation setup.
* `SqlAstCreationContext#getSessionFactory()` was removed in favor of narrower
  SQL translation/context access.
* `SessionFactoryServiceComponents` was renamed/split toward
  `StandardServiceComponents`; the intent is to pass grouped standard-service
  access during bootstrap rather than repeatedly threading raw service
  registries.
* `MappingSourceContributionContext` was renamed toward
  `ContributionDiscoveryContext` for APIs that specifically perform
  scanning/resource-discovery.
* `MetadataBuildingOptions` has been replaced conceptually/API-wise by
  `MappingResolutionOptions` and `MappingPreferences`; no-arg preference methods were
  moved off `MetadataBuildingContext` where practical.
* `BindingContext` has been narrowed so it holds explicit bootstrap capabilities
  instead of directly exposing `BootstrapContext`.
* `MappingResolutionServices` now groups metadata-building service
  dependencies, while `JpaCompliance` remains a metadata-building-context
  concern rather than a service-component concern.

Suggested next step in this area:

* Keep thinning `MetadataBuildingContext#getBootstrapContext()` and raw
  service-registry usage where the needed capability is obvious, but avoid a
  broad rename/delete pass until the current mapping-object simplification work
  has settled.

== Current Mapped-Superclass Handoff State

The active mapped-superclass/generic handoff shape is documented in
link:applied-mapped-superclass-handoff-design.adoc[`applied-mapped-superclass-handoff-design.adoc`]
and
link:boot-model-terminology.adoc[`boot-model-terminology.adoc`].

Current code direction:

* `AttributeUsageBinding` is the applied attribute usage object.  The removed
  `AppliedMappedSuperclassAttribute` wrapper should not be reintroduced unless
  it carries genuinely new semantics.
* `MappedSuperclassContribution` owns contribution provenance and the ordered
  list of applied usages.
* `BootBindingModel` owns indexed lookup for applied mapped-superclass usages.
  Register usages through
  `BootBindingModel#addAppliedMappedSuperclassAttributeUsage(...)` so the
  contribution list and lookup index stay consistent.
* `MappedSuperclassPropertyHandoff` links the copied compatibility `Property`
  to the applied `AttributeUsageBinding` and contribution provenance.
* `MappedSuperclassHandoffResolver` is exposed through
  `RuntimeModelCreationContext` and is the runtime/JPA metamodel bridge from
  legacy `Property` inputs to applied boot-model metadata.
* `BootBindingModel` also owns an initial `Component -> EmbeddableContribution`
  handoff index.  `BindingState#addEmbeddableComponentHandoff(...)` still
  preserves the transitional binding-state API, but it now feeds the boot-model
  index used after binding.
* `EmbeddableHandoffResolver` is exposed through
  `RuntimeModelCreationContext` and is the runtime/JPA metamodel bridge from a
  legacy `Component + Property` pair to the applied `ComponentMemberBinding`.
* `Property#isGenericSpecialization()` is localized fallback bridge state.
  Runtime/JPA metamodel code should ask the mapped-superclass or embeddable
  handoff resolver for semantic member/type/classification decisions instead of
  checking the flag directly.  Resolver fallbacks should run only when no
  applied handoff metadata is available.
* `Property#getMemberDetails()` remains the source-member compatibility bridge,
  but applied usage metadata should win when available.

Known next modeling gap: embeddable concrete-generic classification now treats
applied `ComponentMemberBinding` metadata as authoritative when the component
handoff exists, but it still retains copied-property fallbacks for compatibility
paths not yet backed by applied contribution data.  The next cleanup should
identify those remaining unbacked paths and either model them as applied
handoffs or keep them explicitly marked as legacy compatibility paths.

=== Generic Fallback Audit

The current fallback paths break down as follows:

* Ordinary mapped-superclass attributes are backed by
  `AttributeUsageBinding`.  `MappedSuperclassHandoffResolver` should treat the
  applied usage as authoritative when found and use
  `Property#isGenericSpecialization()` only when no applied handoff exists.
* Embeddable members are backed by `ComponentMemberBinding` through the
  `Component -> EmbeddableContribution` index.  `EmbeddableHandoffResolver`
  should treat the applied member binding as authoritative when found and use
  copied-property/mapped-superclass generic markers only when no component
  member handoff exists.
* Generic mapped-superclass identifiers are a distinct handoff path.  The
  legacy runtime metamodel receives identifiers through `PersistentClass`
  identifier slots rather than ordinary declared-property iteration.
  `IdentifierHandoffResolver` now centralizes the classification decision and
  prefers `EntityIdentifierBinding` metadata, including the top-level
  identifier member for basic and aggregated ids.  When a `BootBindingModel` is
  available, the resolver treats identifier binding metadata as authoritative.
  The resolver-local `mappedSuperclassIdentifier.isGeneric()` fallback is now
  limited to legacy SessionFactory construction paths that do not yet hand a
  `BootBindingModel` to runtime/JPA metamodel creation.
* Low-level mapping compatibility still uses
  `Property#isGenericSpecialization()` in `MappingHelper` to avoid duplicating
  columns/selectables for copied concrete generic properties.  That is not a
  runtime/JPA metamodel classification path.

Follow-up options:

* route the remaining legacy SessionFactory construction path through
  `SessionFactoryConstructionPlan` with a populated `BootBindingModel`
  so `IdentifierHandoffResolver` can remove its no-boot-model generic
  identifier `Property` fallback;
* keep `MappingHelper` as a documented legacy mapping-model compatibility use;
* avoid adding new direct `Property#isGenericSpecialization()` checks outside
  resolver or low-level mapping compatibility code.

== Next Phase

The next pass should shift from failure-bucket triage to implementing the
design work captured in the 9.0 design documents.  Treat the green suite as the
baseline and use focused tests as guardrails for intentional design changes, not
as a stale backlog-mining source.

Prefer work that reduces transitional bridges and makes the pipeline shape more
explicit:

* move sidecar facts toward the applied/resolved contribution model;
* advance mapped-superclass, embeddable contribution, and generic-inheritance
  modeling as one coordinated design surface;
* keep extracting finalization responsibilities from legacy second-pass names
  into explicit binding/materialization/finalization phases;
* clarify aggregate, state-management, and additional-mapping-contributor
  handoffs using the documented designs;
* replace positional assumptions with correspondence products where they make a
  real handoff safer or clearer.

Do not restart the old binding-gap or full-suite census workflow unless a fresh
suite run produces new failures.  If a regression appears, reproduce it
directly, fix the boot/model or settings deviation, and return to the design-led
queue.

== Active Selection Rules

Use these rules before choosing any new work:

* Prefer selecting work from the design documents listed above.  Use old
  failure-bucket language only as historical context.
* Do not choose work from archived refresh notes or old census buckets unless a
  fresh focused test or current XML report reproduces the failure.
* Treat closed buckets as closed.  Reopen one only with a current failing test
  name, stack, or boot-model assertion.
* Prefer direct boot/model signals: source overlay, categorization, binding,
  materialization, settings normalization, metadata finalization, and
  compatibility handoff.
* Keep criteria, proxy, bytecode, mutation, and SQL-AST/runtime symptoms behind
  boot/model failures unless the mapping handoff is clearly missing a fact.
* For generics, use hibernate-models metadata.  Do not add Java reflection
  generic parsing.
* Keep fixes in source/categorization, binding, materialization, settings
  resolution, or handoff metadata unless the legacy mapping handoff cannot
  express the needed fact.

== Compatibility Handoffs To Keep Visible

* `Property#getMemberDetails()` is the source-member bridge.
* `AttributeTypeCorrespondence` centralizes hibernate-models declaration/usage
  type facts for JPA-domain/SQM metamodel creation.
* `RelationalModelCorrespondences` owns new-pipeline relational correspondence
  facts.  Its first product, `ColumnNameCorrespondence`, records table-local
  logical-to-physical column pairs and is projected to legacy collector column
  bindings for compatibility.
* Phase 1 sidecars currently exist for attribute type correspondence,
  mapped-superclass properties, embeddable components, entity identifiers,
  table references, natural-id, collation, tenant-id, and version properties.
* State-management binding has an early root phase and late compatibility
  projection for property/collection temporal metadata.
* Aggregate work has `AggregateMappingIntent`, `AggregateValuePlan`,
  `AggregateMemberContainer`, and `ComponentMemberTarget` as binding-side
  handoff points before projection into `Component`, `AggregateColumn`, and
  compatibility second-pass structures.

== Fresh-Context Cautions

* Do not "fix" runtime entity-key, SQM, SQL-AST, or initializer behavior to
  compensate for mapping shapes that the boot model should express.
* Be careful with quoted identifiers.  Use `Identifier#matches(...)` semantics
  where identifier-aware comparison is needed.
* Quoted JPA entity names are illegal in this branch.  Use `@Table`, `@Column`,
  and other database-identifier annotations for quoted database names.
* Avoid reintroducing builder surfaces.  The design direction is explicit
  pipeline products, construction requests, producers, views, and
  materializers.
* Keep new binding-model contracts free of `org.hibernate.mapping` types unless
  the contract is explicitly a materializer or compatibility bridge.
* When adding new Java files, include `@since 9.0`.
* When adding new Java files, include `@author Steve Ebersole`.
