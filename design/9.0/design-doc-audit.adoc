= ORM 9 Design Document Audit
:toc:

This document audits the design notes in `design/9.0`.

The immediate goal is to reduce stale and duplicated information.  The longer
term goal is to make each remaining document have one clear job:

* durable target design and rationale;
* contributor orientation;
* completed audit/reference data;
* historical context.

Current branch status, test status, and "what next?" guidance should live only
in link:current-work-handoff.adoc[`current-work-handoff.adoc`].  Other design
documents should not track branch health or old failure buckets.

== Classification

Active::
Still describes a live design problem or cleanup target.

Reference::
Useful durable explanation or contributor orientation, but not a status tracker.

Finished reference::
Completed reference or audit data that remains useful but is no longer an
active design note.  Keep under `finished/` when it is large enough to distract
from active routing.

Resolved::
The main design work is implemented or the audit is mostly complete.  Keep only
if it remains useful as a concise final-state reference.

Historical::
Rationale for an old PoC or transition.  Keep only if it prevents repeating an
old mistake; otherwise move to git history.

Archive candidate::
Mostly status log, stale implementation narrative, or duplicate material that
should be folded into another document before deletion.

== Policy

* `current-work-handoff.adoc` is the only current-status document.
* Design documents may say whether a design is active, resolved, or historical,
  but should not track current test health.
* Prefer one canonical document per design area.
* Prefer deleting old work logs once their final decisions are captured.
* If two documents explain the same concept, keep the one that states the
  target model most clearly and fold unique durable rationale into it.
* Completed audits should either become concise reference tables or be archived.

execution scope is assigned here.  This table is authoritative when an older
question section does not yet carry an inline disposition.

|===
| Area
| Scope
| Questions To Resolve

| Bootstrap customizations
| Blocking
| Product ownership and deterministic ordering for type, function, naming,
mapping, cache, and factory customizations; removal of broad context transport.

| Additional mapping contribution
| Blocking / Triggered follow-up
| Blocking: processed-model view for the retained late contributor.
Follow-up: early source contribution, recategorization, ordering, and fixed
points, triggered by a concrete pre-binding use case.

| Binding extensions
| Stabilization / Triggered follow-up
| Decide the ORM 9 status of legacy `AttributeBinder` and `TypeBinder`.

| JPA static metamodel
| Blocking
| Materialization-time plan production, embeddables, inherited generic
attributes, explicit field roles, field hiding, and retirement of duplicate
runtime traversal.

| Aggregate mappings
| Stabilization
| Verify remaining runtime consumers with focused reproducers.  A new mapping
product is blocking only if observable runtime behavior cannot be expressed by
the existing aggregate intent/member-container handoff.

| Generators
| Blocking
| Choose descriptor versus source shape; preserve exportables, CDI, archive
restoration, and identifier/value lifecycle separation.

| State management
| Stabilization
| Verify runtime/settings handoff.  A first-class descriptor is a triggered
follow-up unless whole-model reasoning proves necessary.

| Declaration / usage / applied mapping
| Resolved foundation
| No umbrella redesign.  Route remaining behavior to the mapped-superclass,
embeddable, generic, identifier, or runtime consumer which exposes it.

| Blocking
| Finish stable role-based application and runtime/JPA handoffs, including
mixed components and generic specialization; then consolidate overlapping
notes.

| Identifier correspondence
| Blocking only with a concrete consumer
| Implement identifier-part correspondence where IdClass, derived-id, or
association-valued-id runtime behavior cannot use existing role/selectable
correspondence.  Otherwise keep as a follow-up runtime-model improvement.

| Deterministic ordering
| Resolved
| Preserve phase-owned sorting and focused guardrails; no remaining design
question blocks the PR.

| Mapping-key creation
| Stabilization
| Narrow remaining low-level compatibility/naming helpers without reopening
resolved key products and materializers.

| `MetadataBuildingContext`
| Cross-cutting stabilization
| Remove stored broad context and phase-ownership violations by capability.
Total reference count and compatibility constructor parameters are not
completion metrics.

| SessionFactory bootstrap/construction
| Blocking
| Prepare no-factory values, narrow runtime model creation context, make
service/integrator phases explicit, remove broad contexts from the final
construction boundary, and define the retained `SessionFactoryOptions` view.

| Metadata snapshot/serialization
| Resolved / Triggered follow-up
| The factory-ready archive is complete.  Other general-purpose snapshot layers
are follow-ups triggered by a concrete tooling or startup requirement.

| Bootstrap API compatibility
| Stabilization
| Formal `Configuration` deprecation policy, replacement coverage for removed
builder contributors, and the minimum public custom-producer settings view.

== Resolved Or Mostly Resolved Items

These should not remain as active status narratives:

* removal of `MetadataSources`, `MetadataBuilder`, `MetadataBuilderImpl`,
  `HibernateBootstrap`, `SessionFactoryBuilder`, and
  `Metadata#buildSessionFactory()`;
* replacement of the old public staged builder path with
  `HibernatePersistenceConfiguration` / JPA bootstrap and internal
  `BootstrapPipeline`;
* externalization of `BasicValue.Resolution` determination;
* removal of the old collector-driven basic-value resolver registration path;
* materializer bridge drain for key creation, except for retained deprecated
  compatibility helpers;
* removal of `MetadataBuildingProcess` as a production path;
* old broad failure-bucket and green-suite closeout narratives.
