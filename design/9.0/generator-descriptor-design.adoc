= Generator Descriptor Design
:toc:

Status: Deferred. Identifier-generator descriptor/source and UUID-interpretation
decisions are intentionally postponed while boot-model serialization work
continues in other areas.

This note defines the ORM 9 direction for representing bound generator
declarations and creating runtime `Generator` instances.

The immediate goal is to replace opaque, serializable `GeneratorCreator`
closures with explicit declarative descriptors. The descriptor is a
boot/runtime handoff product: source binding creates it, mapping objects retain
it, schema tooling consumes its exportable contribution, and runtime-model
construction asks it to create the runtime generator.

== Post-Rebase Baseline

The deprecated `ForeignGenerator`, `UUIDHexGenerator`, `GUIDGenerator`, and
`org.hibernate.id.UUIDGenerator` classes have been removed.

The surviving generator paths are now materially clearer:

* `GenerationType.UUID` and implicit UUID generation are handled directly by
  `GeneratorAnnotationHelper#handleUuidStrategy(...)` using
  `org.hibernate.id.uuid.UuidGenerator`. They do not pass through a legacy
  strategy string.
* Derived identifiers are mapping semantics. `@MapsId` and related derived-id
  bindings install an internal insert-only `BeforeExecutionGenerator` which
  obtains the associated identifier through
  `IdentifierGeneratorHelper#getForeignId(...)`. They do not use a `foreign`
  strategy.
* `@GenericGenerator` is no longer a named legacy strategy declaration. It is
  an `@IdGeneratorType(GenericGeneratorGeneration.class)` annotation with a
  required generator `type` and declarative parameters. It is a migrational
  bridge to a dedicated user-defined `@IdGeneratorType` annotation.
* Sequence, table, identity, UUID, annotation-based custom, and derived-id
  generation all have direct semantic interpretations.
* XML generic-generator registrations and some programmatic/named-definition
  paths still use `IdentifierGeneratorDefinition`, a strategy class name, and
  a parameter map.

`GeneratorStrategies` has been reduced accordingly. Its remaining named
compatibility inputs are `native`, `assigned`, `enhanced-sequence`, `sequence`,
`enhanced-table`, `table`, `identity`, `increment`, and `select`, plus an
arbitrary generator class name. It also still converts some JPA generation
types to class-name strings for existing definition-based paths. It is now
clearly compatibility machinery, not the architecture for new binding code.

== Current Runtime Shape

The mapping graph already avoids retaining runtime generator instances:

* `SimpleValue` retains a `GeneratorCreator` for identifier generation;
* `Property` retains a `GeneratorCreator` for version and ordinary generated
  values;
* metadata finalization creates identifier generators to discover and register
  schema exportables;
* `SessionFactory` construction creates the runtime identifier and value
  generators from the retained creators.

This phase split is directionally correct, but `GeneratorCreator` is too opaque
to be the durable representation. Most creator instances are lambdas which
capture annotations, class names, configuration maps, or derived-id facts.
Archive support made the problem visible by requiring these closures to be
serializable and, for annotation-based identifiers, by requiring a transient
`ModelsContext` to be reattached after restoration.

The descriptor work should preserve the phase split while making the retained
state explicit and inspectable.

== Decisions

=== Descriptors are declarative boot state

A descriptor records the already-interpreted generator declaration. It may
retain:

* a localized dynamic annotation implementation;
* a generator class or declarative class name;
* immutable configuration parameters;
* built-in semantic facts such as identity, UUID, sequence, table, assigned,
  or derived-id generation;
* stable role information needed for diagnostics.

A descriptor must not retain a `MetadataBuildingContext`, `BootstrapContext`,
`ModelsContext`, bean container, service registry, mapping graph object,
runtime `Generator`, or other service-backed instance. The existing
`GeneratorCreationContext` supplies the mapping value, root entity, property,
member, database, services, and SQL-generation context when creation is
requested.

Annotation selection and localization, including localization of a
`@GenericGenerator` declared on a generic mapped superclass, must finish during
binding. Runtime creation must not repeat source discovery or require
`GeneratorCreator#reattachModelsContext(...)`.

=== Identifier and value descriptors are separate contracts

Identifier generators have schema-exportable and identifier-specific timing
responsibilities which ordinary value generators do not. The initial design
should therefore use separate contracts rather than forcing both through one
large interface:

[source,java]
----
interface IdentifierGeneratorDescriptor {
    Generator createGenerator(GeneratorCreationContext context);
    void contributeExportables(GeneratorCreationContext context);
}

interface ValueGeneratorDescriptor {
    Generator createGenerator(GeneratorCreationContext context);
}
----

The exact names may be shortened if implementation experience suggests a
better vocabulary, but the role split is intentional. Shared instantiation,
initialization, configuration, CDI, and validation code belongs in internal
support, not in a public common supertype.

These are internal contracts. Implementations belong under an internal boot
generator package; they are not new public generator SPIs.

=== Use the existing creation context first

`GeneratorCreationContext` already carries the runtime inputs used by current
generator creation. The first descriptor implementation should consume it
directly. Introducing another runtime creation context before a concrete
missing input is identified would add vocabulary without improving the phase
boundary.

The context may be narrowed later if the descriptor implementations reveal a
stable smaller service bundle.

=== Preserve current CDI behavior

Annotation-based and named custom generators currently use the configured bean
container when one is available. The descriptor refactoring should preserve
that behavior. Disallowing CDI-managed identifier generators would be a
separate compatibility decision and is not required by this design.

== Bound Descriptor Forms

Source binding should normalize declarations into a small set of explicit
descriptor forms.

Annotation-based identifier descriptor::
Retains the localized generator annotation, its annotation type, and the class
declared by `@IdGeneratorType`. This covers dedicated custom generator
annotations and `@GenericGenerator`; the latter retains its delegate class and
parameters in the localized annotation.

Sequence and table identifier descriptors::
Retain the normalized JPA declaration and fallback allocation facts. They
create `SequenceStyleGenerator` or `org.hibernate.id.enhanced.TableGenerator`
without converting the declaration to an arbitrary strategy string.

Identity and UUID identifier descriptors::
Represent these built-in semantics directly. UUID generation uses
`org.hibernate.id.uuid.UuidGenerator` and its localized configuration.

Assigned identifier descriptor::
Represents application-assigned identifiers explicitly rather than by testing
a creator implementation or string name.

Derived identifier descriptor::
Retains the declaring entity and association attribute needed by
`IdentifierGeneratorHelper#getForeignId(...)`. It remains internal mapping
semantics and is never interpreted as a named generator strategy.

Legacy named-strategy descriptor::
Retains the surviving strategy class name and immutable parameters for XML,
programmatic, or compatibility definition paths. Only this descriptor uses
`GeneratorStrategies` and class-name loading.

Annotation-based value descriptor::
Retains the localized value-generation annotation, annotation type, and class
declared by `@ValueGenerationType`. Runtime creation performs the existing
constructor selection, initialization, configuration, and version-timing
validation.

== Phase Ownership

Mapping-model binding owns::
Selecting and localizing the declaration, validating mutually exclusive
annotations, normalizing parameters, choosing the descriptor form, and storing
it on the mapped value or property.

Metadata finalization and schema tooling own::
Asking identifier descriptors to contribute schema exportables. They must not
rediscover source annotations or strategy declarations.

Runtime-model construction owns::
Creating runtime generator instances from descriptors and an explicit
`GeneratorCreationContext`, then validating runtime-dependent generator facts.

Mapping objects are descriptor carriers. They do not create generators and do
not recover creation services from retained boot contexts.

== Identifier Exportables

Identifier generation remains the harder case because `Configurable` and
`ExportableProducer` define an ordered lifecycle:

[source]
----
instantiate -> configure -> register exportables -> initialize
----

The initial descriptor implementation should preserve this lifecycle exactly.
For a custom or legacy generator, `contributeExportables(...)` may instantiate
and configure a temporary generator, register its exportables, and discard it.
Runtime construction creates a separate runtime instance, as it does today.

Built-in sequence and table descriptors should ultimately contribute their
known database objects directly from the bound recipe. That avoids temporary
runtime-generator creation during metadata finalization. It is an optimization
and ownership improvement, not a prerequisite for replacing creator closures.

Archive restoration does not rerun source binding or metadata finalization.
The restored `Database` already contains the exported relational objects;
runtime generator creation proceeds from the restored descriptor and mapping
graph.

== Relationship to `IdentifierGeneratorDefinition`

`IdentifierGeneratorDefinition` remains the string-and-parameter compatibility
representation for named definitions. It should not become the descriptor
supertype.

As direct sequence, table, and annotation-based binding moves to descriptors,
uses of `IdentifierGeneratorDefinition` should contract toward the remaining
XML/programmatic named-definition boundary. A definition is interpreted once
into a descriptor; runtime construction should not interpret it again.

Similarly, `GeneratorStrategies#generatorStrategy(...)` should disappear from
direct semantic binding paths. `GeneratorStrategies#generatorClass(...)` and
`mapLegacyNamedGenerator(...)` remain available only to the legacy
named-strategy descriptor while those inputs are supported.

== Alternative: An Identifier Generator Source Fact

The descriptor design above treats the retained object primarily as a recipe
for creating and preparing a `Generator`. A stronger alternative is to make
the retained object the fully interpreted source fact for identifier
generation. For discussion, call this object `IdentifierGeneratorSource`.

[source]
----
declaration -> IdentifierGeneratorSource -> schema export
                                      \-> runtime Generator
----

Under this model, annotation, XML, and programmatic declarations are
interpreted once during mapping-model binding. The resulting source manages
the later phase boundary:

. metadata binding creates the source;
. schema tooling asks the source to contribute exportables, if any; and
. `SessionFactory` construction asks the source to create and initialize the
  runtime generator, possibly obtaining that generator from CDI.

The source is still declarative boot state. It may retain the resolved
generator class, immutable custom parameters, resolved sequence or table
names, allocation and optimizer facts, identifier type and timing facts, and
other stable configuration. It must not retain boot contexts, model contexts,
services, mapping graph objects, bean instances, or runtime generators.

=== Possible contract shape

A minimal behavioral contract could combine export contribution and runtime
creation:

[source,java]
----
interface IdentifierGeneratorSource extends Serializable {
    void contributeExportables(GeneratorExportContext context);
    Generator createGenerator(GeneratorCreationContext context);
}
----

However, CDI lifecycle should preferably remain owned by Hibernate. A source
which obtains CDI beans itself would have to understand bean-container
lifecycle and release semantics. A more constrained contract would instead
describe the runtime class and initialize the instance supplied by Hibernate:

[source,java]
----
interface IdentifierGeneratorSource<G extends Generator> extends Serializable {
    Class<G> getGeneratorClass();
    void contributeExportables(GeneratorExportContext context);
    void initializeGenerator(G generator, GeneratorCreationContext context);
}
----

The exact contract requires more work. In particular, on-execution generators,
assigned identifiers, and built-in generators which need no instantiated
runtime object during schema tooling might not all benefit from a single
generic shape. The important distinction is that the source owns interpreted
configuration while Hibernate owns runtime instance acquisition.

Creating a custom source from an annotation would also require a source
factory contract, for example:

[source,java]
----
interface AnnotationBasedIdentifierGeneratorSource<A extends Annotation> {
    IdentifierGeneratorSource<?> createSource(
            A annotation,
            IdentifierGeneratorSourceCreationContext context);
}
----

The creation context must provide enough mapping, dialect, naming, and type
information to interpret the declaration. The returned source must copy the
required facts and not retain the context.

=== Lifecycle improvement

For a source-aware generator, exportable registration no longer requires a
temporary runtime generator:

[source]
----
metadata:        interpret declaration -> source
schema tooling:  source -> register exportables
runtime:         source -> obtain generator through Hibernate/CDI
                        -> initialize generator
                        -> retain generator
----

This removes duplicate generator construction and avoids replaying
configuration merely to discover relational objects. It also ensures that CDI
is a runtime concern unless source creation itself explicitly requires a
separate boot-time service.

The source/runtime boundary must nevertheless preserve any ordering required
by a runtime implementation. A source-aware runtime generator must receive all
state it previously derived from `configure()` and `registerExportables()`
before `Configurable#initialize(...)` or first use. Moving export registration
to the source must not silently omit instance initialization formerly
performed as a side effect of `registerExportables(...)`.

=== Compatibility with existing custom generators

Existing custom generators cannot in general be converted automatically into
pure source facts. Their exportable structure may be calculated by constructors,
`AnnotationBasedGenerator#initialize(...)`, `Configurable#configure(...)`, or
`ExportableProducer#registerExportables(...)` operating on the generator
instance.

Existing `@IdGeneratorType` and named-strategy generators should therefore be
adapted by an internal legacy source. It retains the localized annotation or
class name and immutable parameters, and preserves the current lifecycle:

* during schema tooling it may instantiate, initialize, configure, register
  exportables, and discard a temporary generator;
* during runtime construction it creates a separate instance and performs the
  complete required lifecycle before retaining it.

The lifecycle of one generator instance must never be divided between the two
phases. In particular, Hibernate must not invoke `registerExportables(...)` on
a temporary instance and assume that a different runtime instance can proceed
directly to `Configurable#initialize(...)`.

This adapter gives existing custom generator implementations zero mandatory
changes. They do not need to be serializable and do not implement the source
contract, but they do not receive the cleaner source-aware lifecycle.

=== Opt-in custom source SPI

Source-aware custom generation should initially be opt-in rather than changing
the meaning of `@IdGeneratorType`. A possible incubating meta-annotation is:

[source,java]
----
@IdGeneratorSourceType(CustomSequenceSource.class)
@Retention(RUNTIME)
@Target(ANNOTATION_TYPE)
public @interface CustomSequence {
    // declarative configuration
}
----

The resulting compatibility model is:

* built-in identifier generation uses internal source implementations;
* a new source-aware annotation uses its declared source factory;
* existing `@IdGeneratorType` annotations use the legacy source adapter;
* `@GenericGenerator` uses the legacy source adapter because it declares a
  runtime generator class and generic parameters; and
* named XML and programmatic strategies use the legacy named-strategy source.

This avoids forcing source/archive concerns into the existing custom
`Generator` SPI. If experience with the opt-in contract is successful, ORM 9
may later reconsider whether `@IdGeneratorType` itself should select a source
instead of a runtime generator.

=== Benefits and costs

Compared with a construction descriptor, the source-fact model:

* interprets declarations once and makes the result inspectable;
* provides a natural archive product;
* allows schema tooling to operate without runtime or CDI instances;
* avoids duplicate construction and configuration for source-aware generators;
* exposes timing, identity, and exportable facts without executing user
  generator code; and
* gives built-in sequence and table generation a cleaner ownership boundary.

Its costs are:

* a new custom extension contract alongside the existing generator SPI;
* an archive-compatibility obligation for user-defined source state;
* careful definition of which state a source may retain;
* possible duplication between source configuration and runtime generator
  initialization;
* difficulty supporting generators whose exportable structure depends on CDI
  injection or other runtime services; and
* permanent compatibility machinery for arbitrary existing generators.

The source-fact model is highly feasible for built-in generators and as an
opt-in custom SPI. It is not feasible as a transparent pure-source replacement
for every existing custom generator. A hybrid architecture using native
sources plus a legacy adapter is therefore the realistic form of this
alternative.

== Implementation Sequence

Before beginning this sequence, choose whether the retained identifier contract
is the internal construction descriptor described first or the hybrid
`IdentifierGeneratorSource` model. The first three steps otherwise risk
creating a short-lived descriptor API which is immediately replaced by the
source SPI. The value-generator work is independent and may continue to use a
construction descriptor even if identifier generation adopts sources.

. Introduce the internal identifier and value descriptor contracts and change
  `SimpleValue` / `Property` to retain them instead of `GeneratorCreator`.
. Replace `GeneratorBinder.IdentifierGeneratorCreator` with an explicit
  annotation-based identifier descriptor. Complete annotation localization at
  binding time and remove generator-related `ModelsContext` reattachment from
  metadata restoration. This slice covers all `@IdGeneratorType` annotations,
  including the reworked `@GenericGenerator`.
. Replace annotation-based value-generator lambdas with an explicit value
  descriptor using the same declarative annotation approach.
. Add direct descriptors for UUID, identity, assigned, and derived identifiers.
. Add sequence and table descriptors, initially preserving the existing
  generator-based exportable lifecycle.
. Confine remaining class-name and parameter-map handling to a legacy
  named-strategy descriptor, then remove `GeneratorStrategies` calls from
  direct JPA and annotation binding paths.
. Move built-in sequence/table exportable contribution into their descriptors
  once the descriptor boundary is stable.

Each step must preserve both normal factory construction and the metadata
archive-to-factory round trip.

== Verification

Descriptor coverage should include:

* `GenerationType.IDENTITY`, `SEQUENCE`, `TABLE`, `UUID`, and UUID selected by
  `AUTO` identifier type;
* local, type-level, package-level, inherited, and generic-mapped-superclass
  `@IdGeneratorType` declarations;
* `@GenericGenerator` with a custom generator class and parameters;
* an annotation-based ordinary value generator and version generator;
* assigned and derived identifiers;
* named sequence/table definitions and the surviving legacy named strategies;
* a custom `ExportableProducer` generator;
* CDI and non-CDI instantiation paths;
* live metadata-to-factory and serialized metadata-to-factory construction.

Archive inspection should assert that descriptors retain only declarative
state and require no post-restoration `ModelsContext` attachment.

== Remaining Questions

* Should identifier descriptors remain internal construction recipes, or
  should `IdentifierGeneratorSource` become the primary retained boot fact?
* If source-aware custom generation is exposed, should it use a new
  `@IdGeneratorSourceType` meta-annotation or evolve `@IdGeneratorType` in ORM
  9?
* Should a source create the runtime generator, or should Hibernate acquire it
  through CDI and pass it back to the source for initialization?
* Should the descriptor methods retain the name `createGenerator`, or should
  identifier creation distinguish schema-discovery and runtime purposes more
  explicitly?
* Which generator timing facts can be validated entirely during binding and
  stored on the descriptor without instantiating user code?
* Which programmatic APIs still require arbitrary strategy class names after
  direct annotation and JPA binding no longer use them?
* Can built-in sequence/table descriptors contribute exportables without any
  compatibility-visible change in naming or optimizer behavior?

== Guardrails

* Descriptors retain declarations, never live generator instances or service
  objects.
* Mapping objects carry descriptors but do not instantiate generators.
* Binding completes annotation selection and localization once.
* Runtime creation receives context explicitly.
* Identifier and value descriptors remain separate where their lifecycle
  responsibilities differ.
* Schema exportable discovery remains available for custom generators.
* `@GenericGenerator` follows the `@IdGeneratorType` descriptor path; it is not
  reintroduced as a named strategy string.
* Derived identifiers remain derived-id mapping semantics.
* `GeneratorStrategies` is restricted to surviving compatibility inputs.
