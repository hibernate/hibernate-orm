= Boot Mapping Finalization Design
:toc:

This note describes the finalization phase for the ORM 9 boot mapping pipeline.

== Goal

New-pipeline binders should not use legacy `SecondPass` callbacks for work that
is not unresolved binding.  Some mapping work legitimately needs the complete
bound model, but that work should be named as finalization:

* all source facts have been categorized;
* mapping objects and binding sidecars have been created;
* identifiers, members, table keys, inverse associations, and foreign keys have
  been resolved;
* final derived mapping side effects can be applied deterministically.

Finalization is not a retry mechanism.  If a binder is missing a semantic fact,
the fix belongs in source, categorization, binding, materialization, or a typed
pending binding product.  Finalization should only consume already-bound facts.

== Initial Shape

`TypeBindingPhase.Finalization` is the first narrow hook.  `BindingCoordinator`
runs it after the foreign-key and state-management phases.

The first migrated finalizer is discriminator column finalization for the new
pipeline.  `EntityTypeBinder` now finalizes discriminator column nullability and
check constraints directly through `DiscriminatorColumnFinalizer` instead of
scheduling legacy `DiscriminatorColumnSecondPass`.

The legacy `DiscriminatorColumnSecondPass` remains for the old
`boot.model.internal.EntityBinder` path.

== Remaining New-Pipeline Callback Uses

As of the green-suite close-out, the new pipeline still schedules callback-style
work for:

* aggregate compatibility projection from `AggregateComponentBinder` through
  `AggregateComponentSecondPass`;
* entity constraint creation from `EntityTypeBinder` through an anonymous
  second-pass callback.

Aggregate work has already moved important facts into binding-side products:
`AggregateMappingIntent`, `AggregateValuePlan`, `AggregateMemberContainer`, and
`ComponentMemberTarget`.  The remaining compatibility callback still performs
component validation, aggregate column initialization, UDT registration, dialect
aggregate support validation, auxiliary database object creation, aggregate
read/write expression setup, and ordering/projection into the historical
`Component` / `AggregateColumn` shape.

The next aggregate finalization step is therefore not a mechanical callback
rename.  It should make aggregate finalization consume those binding-side
products through a named new-pipeline finalizer, while preserving the legacy
`AggregateComponentSecondPass` for the old binder path until both paths share an
authoritative aggregate storage model.

Entity constraint creation is smaller and can follow once finalization ordering
is stable.

== Rules

* Prefer a typed finalization class over anonymous callbacks.
* Keep finalizers deterministic and idempotent with respect to one bootstrap run.
* Do not use finalization to search for facts that should have been recorded by
  binding.
* Leave old-binder `SecondPass` users alone unless the old path is being retired
  or explicitly bridged to the new finalization phase.
