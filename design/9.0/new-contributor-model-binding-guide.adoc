= New Contributor Guide: Model Binding Pipeline
:toc:

This guide explains the prototype boot flow from source collection through
binding-model population and `org.hibernate.mapping` materialization.

The goal is to give new contributors a mental model before they start changing
annotation, XML, categorization, or binding code.


== Before You Start

This guide assumes you are familiar with:

* JPA annotations such as `@Entity`, `@MappedSuperclass`, `@Embeddable`, and `@ElementCollection`
* The idea that Hibernate needs to process your domain classes and turn them into
an internal model before it can generate SQL.  This is the "boot" process.
* `org.hibernate.mapping` is Hibernate's internal boot-time representation of your
domain model: tables, columns, joins, and foreign keys.  In the ORM 9 pipeline it
is the materialization target, not the semantic binding model itself.  It is not
the runtime metamodel; it exists during startup to build the SessionFactory.

If any of these are unfamiliar, start with the Hibernate ORM documentation on
bootstrapping and the JPA specification chapter on metadata.


== Big Picture

The pipeline is:

[source,text]
----
Persistence unit / programmatic config / tests
        |
        v
AvailableResources
        |
        v
DomainModelCategorizer.categorize(...)
        |
        v
CategorizedDomainModel
        |
        v
BindingCoordinator.coordinateBinding(...)
        |
        v
BindingState / binding products / pending work
        |
        v
materializers
        |
        v
org.hibernate.mapping
----

Source collection is the input-assembly step.  It gathers explicit persistence
unit declarations, programmatic configuration, XML mapping bindings, and any
archive-scanning contributions into
link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources`].

link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources`]
answers: "What source material should this boot process consider?"

link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`]
answers: "After interpreting that source material, what persistent types,
attributes, hierarchies, and global registrations does binding need?"

link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingCoordinator.java[`BindingCoordinator`]
answers: "How do we turn that categorized model into ordered binding facts,
pending work, and materialization requests in a dependency-safe order?"

Materializers answer: "Given finalized binding facts or resolved products, how
do we create or mutate Hibernate's boot-time `org.hibernate.mapping` objects?"

The categorizer should be the last place that works with raw source material
(classes, package metadata, XML documents, global annotations).  Later binding
code should consume categorized contracts (typed Java interfaces that expose
already-interpreted facts, rather than raw annotation or XML data) instead of
re-examining that source material directly.


[TIP]
.Example: A Simple @Entity
====
Imagine a persistence unit with one class, `Customer`, annotated with `@Entity`.

1. link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources`]
receives `Customer` as a managed class detail.
2. link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/DomainModelCategorizer.java[`DomainModelCategorizer`]
sees `@Entity` on `Customer` and records it as a persistent entity type.  It
also collects any global registrations declared on the class, such as
`@NamedQuery`.
3. link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/EntityHierarchyBuilder.java[`EntityHierarchyBuilder`]
creates a single-class hierarchy with `Customer` as the root.
4. The result, a
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`],
now contains one entity hierarchy, zero mapped superclasses, and zero
embeddables.
5. link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingCoordinator.java[`BindingCoordinator`]
walks that hierarchy and records the ordered binding facts needed for tables,
identifiers, attributes, and constraints.
6. Materializers consume those binding facts and resolved products, creating or
mutating the `PersistentClass`, `Table`, `Property`, `Value`, `PrimaryKey`,
`ForeignKey`, `UniqueKey`, and `Index` objects needed by the rest of boot.

At the end, `org.hibernate.mapping` contains the materialized boot mapping model
Hibernate needs to generate the `customer` table DDL and build the runtime
SessionFactory.
====

== AvailableResources

`AvailableResources` is a small source inventory: the raw inputs represented as
Java class details, `package-info` class details, and XML mapping documents.  It
has three buckets:

* managed class details
* package details, represented by `package-info` class details
* already-bound XML mapping documents

Touchpoint:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources`]

The record has convenience factories for the source forms used by the PoC:

* `AvailableResources.from(PersistenceUnitDescriptor, AvailableResourcesContext)`
* `AvailableResources.from(PersistenceUnitDescriptor, AvailableResourcesContext, ResolvedMappingSettings)`
* `AvailableResources.from(PersistenceConfiguration, AvailableResourcesContext)`
* `AvailableResources.from(HibernatePersistenceConfiguration, AvailableResourcesContext)`
* `AvailableResources.from(HibernatePersistenceConfiguration, AvailableResourcesContext, ResolvedMappingSettings)`
* `AvailableResources.from(MappingSourceContributions, AvailableResourcesContext, ResolvedMappingSettings)`
* `AvailableResources.from(MetadataSources, AvailableResourcesContext)`

link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResourcesContext.java[`AvailableResourcesContext`]
carries the early bootstrap inputs needed to build the inventory:
`ModelsContext` and `ServiceRegistry`.  This keeps source collection independent
of `MetadataBuildingContext`, which is created later when available resources
are categorized and bound.

The factories resolve class names through the Hibernate Models
`ClassDetailsRegistry`, Hibernate Models' registry for reflective class
metadata.  They also bind mapping files immediately with `MappingBinder`, so by
the time categorization starts XML mappings are JAXB bindings, Java objects
parsed from XML, rather than unresolved resource names or DOM models.

The `HibernatePersistenceConfiguration` factory additionally applies archive
scanning results from
link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/HibernatePersistenceConfigurationScanner.java[`HibernatePersistenceConfigurationScanner`].

Tests sometimes construct `AvailableResources` directly.  That is fine when a
test wants tight control over the source set.


== Package Metadata

Package annotations enter through `package-info` class details.  The helper
method
link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources#applyClassDetails`]
separates these from ordinary managed classes.

That distinction matters because package metadata can contribute defaults or
global declarations, but it is not itself a persistent type.  New code should not
put `package-info` into the same mental bucket as an entity, mapped superclass,
or embeddable.


== XML Pre-Processing

Categorization starts with XML that has been pre-processed from
`AvailableResources`:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/xml/spi/XmlPreProcessor.java[`XmlPreProcessor`]

This happens before annotation processing because XML can affect what is visible
to Hibernate Models.  For example, XML can contribute mapped classes, dynamic
model names, and metadata-complete annotation overlays.  Metadata-complete is
the XML flag that tells Hibernate to ignore annotations on a class and use only
the XML mapping.

`DomainModelCategorizer` keeps XML pre-processing in the categorization phase by
calling `XmlPreProcessor` with `AvailableResources#xmlMappings` directly, so this
path no longer needs ORM's historical `ManagedResources` source shape.

After pre-processing, `DomainModelCategorizer` builds an `allKnownClassNames`
list from:

* mapped classes contributed by XML
* explicitly supplied managed class details
* explicitly supplied package details


== Categorization

`DomainModelCategorizer.categorize(...)` owns the transition from source
inventory to categorized model.

Touchpoint:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/DomainModelCategorizer.java[`DomainModelCategorizer`]

The high-level sequence is:

1. Pre-process XML into ORM's `XmlPreProcessingResult`.
2. Build the complete list of known class and package names.
3. Create a
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/DomainModelCategorizationCollector.java[`DomainModelCategorizationCollector`].
4. Process XML through ORM's `XmlProcessor`.
5. Apply each known `ClassDetails` to the collector.
6. Apply deferred XML overlays.
7. Build inheritance state from the visible persistent types.
8. Build entity hierarchies.
9. Create the
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`].

The categorizer uses the existing bootstrap `ModelsContext`; it does not create a
separate source model registry.


== Categorization Collector

`DomainModelCategorizationCollector` is the in-flight builder for the categorized
model.

Touchpoint:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/DomainModelCategorizationCollector.java[`DomainModelCategorizationCollector`]

It records:

* source persistent types, meaning visible entities and mapped superclasses
* root entity candidates
* mapped superclasses keyed by class name
* embeddables keyed by class name
* persistence-unit scoped global registrations

When it sees a class, it first collects global registrations from that class.
Then it classifies the class as one of:

* `@MappedSuperclass`
* `@Entity`
* `@Embeddable`
* none of the above

Only entities and mapped superclasses are added to the source persistent type
set used for inheritance.  Embeddables are tracked separately because they are
managed types, but they do not participate in entity inheritance.


== Global Registrations

Global registrations are persistence-unit scoped things collected while
categorizing.  They are exposed through:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/GlobalRegistrations.java[`GlobalRegistrations`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/GlobalRegistrationsImpl.java[`GlobalRegistrationsImpl`]

Examples include:

* identifier generators
* Java and JDBC type registrations
* converters
* user types and collection types
* embeddable instantiators
* filter definitions
* named queries
* named entity graphs
* JPA event listeners

If a new annotation or XML element affects the whole persistence unit rather than
one mapped attribute or entity binding, it probably belongs in global
registrations.  Categorization should collect it, and the binding coordinator
should have a small processing method that applies it to the appropriate ORM
boot collector or registry.


== Entity Hierarchies

After all visible persistent types are collected, categorization builds
inheritance state:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/ManagedTypeInheritanceState.java[`ManagedTypeInheritanceState`]

This state is scoped to the visible source set.  It does not scan every class
reachable from the classpath.  If a persistent superclass exists in Java but was
not included in `AvailableResources`, the inheritance state decides how to handle
that missing type.

Then
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/EntityHierarchyBuilder.java[`EntityHierarchyBuilder`]
creates hierarchy objects:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/EntityHierarchyBuilder.java[`EntityHierarchyBuilder`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/EntityHierarchy.java[`EntityHierarchy`]

An `EntityHierarchy` starts from a root entity.  Its absolute root may be a
mapped superclass above that root entity.  Each hierarchy also records facts that
the binder needs later, such as default access type, identifier mapping, version
attribute, tenant-id attribute, inheritance style, caching, and optimistic
locking.


== CategorizedDomainModel

`CategorizedDomainModel` is the result of categorization.

Touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/CategorizedDomainModelImpl.java[`CategorizedDomainModelImpl`]

It exposes:

* global registrations
* entity hierarchies
* mapped superclasses
* embeddables

It deliberately does not expose categorization-time infrastructure such as the
`ClassDetailsRegistry` or `AnnotationDescriptorRegistry`.  If later binding code
needs services, it should get them from binding/categorization context objects,
not by treating
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`]
as a service locator.


== BindingCoordinator

`BindingCoordinator` takes the categorized model and coordinates binding phases.
The phases populate binding state, create transitional mapping shells where the
current compatibility model still requires them, and schedule or invoke explicit
materializers for mapping-object population.

Touchpoint:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingCoordinator.java[`BindingCoordinator`]

`BindingCoordinator.coordinateBinding(...)` receives:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/CategorizedDomainModel.java[`CategorizedDomainModel`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingState.java[`BindingState`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingOptions.java[`BindingOptions`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingContext.java[`BindingContext`]

The categorized model describes what should be bound.  The binding state is the
mutable place where semantic binding products, compatibility mapping shells, and
pending bindings are recorded.  Options and context provide boot-time services
and configuration.


== Binding Steps

`BindingCoordinator` does two things:

1. `coordinateGlobalBindings()`
2. `coordinateModelBindings()`

Global bindings apply persistence-unit scoped registrations to the metadata
collector.  Model bindings record type/member/table/value facts, create
transitional compatibility shells where needed, and call named materializers for
the parts of `org.hibernate.mapping` that are now phase-owned products.  Primary
keys, dependent table keys, collection-table keys, foreign keys, unique keys,
and indexes are examples of work that should flow through explicit
materialization products instead of hidden side effects on mapping values.

The current global binding step processes:

* identifier generators
* converters
* named queries
* named entity graphs
* Java type registrations
* JDBC type registrations
* user type registrations
* composite user type registrations
* collection type registrations
* embeddable instantiator registrations
* filter definitions

JPA event listeners are different: they are consumed while binding entity
metadata through
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/EntityTypeMetadata.java[`EntityTypeMetadata#getCompleteJpaEventListeners()`].


[[_bootstrap_phase_contract]]
== Bootstrap Phase Contract

The pipeline has one guiding rule:

Binding records source-aware facts and resolved intents.
Materialization creates or mutates `org.hibernate.mapping` objects based on
those facts and intents.

Do not add a new phase just because a feature feels complicated.  Add a phase
only when some work must wait until a clearly defined dependency is available
for every managed type, collection, or table.  Otherwise, prefer a typed pending
binding or a resolved product consumed by an existing phase.

[cols="1,2,2", options="header"]
|===
| Step
| Owns
| Must not own

| Source collection
| Persistence-unit resources, mapping files, package details, scanner results
| Annotation interpretation, binding decisions, mapping-object mutation

| Categorization
| Source interpretation into typed persistent-type, member, hierarchy, XML, and
global-registration facts
| `org.hibernate.mapping` object construction or physical constraint creation

| Binding intent capture
| Binding products, source descriptors, value shapes, selectable
correspondence, pending work, and compatibility lookup shells
| Opportunistic physical `PrimaryKey`, `ForeignKey`, `UniqueKey`, or `Index`
creation

| Value/table materialization
| `Table`, `Column`, `Value`, `Property`, `PersistentClass`, `Join`, and
collection-table mapping objects needed as compatibility anchors or later
materialization targets
| Final physical keys and constraints unless the current phase explicitly owns
that artifact

| Key/constraint materialization
| Physical `PrimaryKey`, `ForeignKey`, `UniqueKey`, `Index`, and table-level
constraint objects from resolved products
| Re-reading raw annotations or inventing source semantics from mutable mapping
objects

| Cross-reference finalization
| Property-ref associations, inverse associations, derived identifiers,
synthetic properties, and other work that requires both sides to be bound
| Creating new side channels when typed pending state or resolved products can
carry the dependency

| Validation and compatibility finalization
| Mapping consistency checks and transitional compatibility behavior still
needed by old mapping consumers
| New semantic binding decisions
|===

Materialization therefore has a phase concept, but it is an ownership and
ordering concept first, not necessarily a separate coordinator framework.
Today, materializers are invoked from the binding phase that owns their
dependencies: table-key materializers run after values and tables are available,
foreign-key materializers run after association targets and inverse structures
are stable, and collection/index materializers run after collection key, element,
and index columns are known.

If materialization ordering becomes too large for the binding phase interfaces
to express clearly, introduce a small materialization coordinator with the same
contract: each materialization phase consumes resolved products, creates or
mutates mapping artifacts, and does not rediscover source metadata.


== Type Binders

Model binding starts by walking every categorized entity hierarchy.  For each
entity or mapped superclass in hierarchy order, the coordinator creates a
`ManagedTypeBinder`:

* entities use
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/EntityTypeBinder.java[`EntityTypeBinder`]
* mapped superclasses use
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/MappedSuperTypeBinder.java[`MappedSuperTypeBinder`]

Touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/ManagedTypeBinder.java[`ManagedTypeBinder`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/EntityTypeBinder.java[`EntityTypeBinder`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/MappedSuperTypeBinder.java[`MappedSuperTypeBinder`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/ModelBinders.java[`ModelBinders`]

While each binder is created, the coordinator immediately calls
`bindTypeSkeleton()`.  The skeleton is a compatibility anchor with just enough
identity and table/type structure for other binders to reference it.  In the
current transition this may still include partially populated
`org.hibernate.mapping` objects, but new binding contracts should treat those as
materialization targets or lookup anchors, not as the place where semantic facts
are rediscovered.


== Binding Phases

After binder creation, the coordinator runs a fixed sequence of narrow phase
contracts.

The phases exist because binding facts and materialized mapping objects have
circular dependencies.  For example, a foreign key cannot be materialized until
both sides of an association have tables and columns.  Rather than using
callbacks or lazy resolution, the coordinator runs a fixed sequence of narrow
phases.  Each phase can assume that all prior phases have completed for all
types in all hierarchies.

Treat these as dependency boundaries, not as buckets for arbitrary work.  A
phase may collect binding facts, resolve pending work, or invoke a materializer,
but it should do so only for artifacts whose dependencies are guaranteed by that
point in the sequence.  If a materializer needs more information than the phase
has, carry that information forward as typed pending state instead of looking
back at raw source metadata or mutable mapping objects.

The phase contracts live here:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/binders/TypeBindingPhase.java[`TypeBindingPhase`]

The order is:

[cols="1,3", options="header"]
|===
| Phase
| Purpose

| `Tables`
| Create primary, secondary, joined-subclass, collection, and association table shells.

| `SuperType`
| Connect mapping objects to their already-created super type skeletons.

| `EntityMetadata`
| Apply entity-level metadata that does not depend on member value binding.

| `Identifiers`
| Create root identifier value shapes.

| `AssociationIdentifiers`
| Complete identifier attributes that are associations, after all root identifier shapes exist.

| `Members`
| Bind discriminator, version, tenant id, and persistent attributes.

| `CollectionIndexes`
| Resolve collection index/key values that depend on member bindings, such as `@MapKey(name)`.

| `AssociationTargets`
| Resolve associations that reference non-primary-key target properties.

| `DerivedIdentifiers`
| Resolve `@MapsId` associations after owner identifiers and association values exist.

| `TableKeys`
| Resolve and materialize dependent table keys for joined subclasses, secondary tables, and collection/association tables.

| `InverseAssociations`
| Copy or reconcile owning-side state for `mappedBy` associations.

| `ForeignKeys`
| Resolve and materialize physical foreign-key constraints after values, table keys, and inverse structures are available.
|===

Binders implement only the phases they participate in.  The coordinator checks
whether each binder implements the phase interface before invoking it.

Materialization does not currently require a separate public phase interface for
each artifact family.  The important contract is that the binding phase that
invokes a materializer owns that artifact's dependency boundary and passes a
resolved product such as `ResolvedForeignKey`, `ResolvedUniqueKey`, or
`ResolvedIndex`.


== Pending State

The prototype avoids callback-style second passes by recording typed pending
state in
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingState.java[`BindingState`].

Examples:

* association targets that need non-primary-key property resolution
* association-valued identifiers
* derived identifiers
* table-key foreign keys
* inverse to-one and inverse plural associations

This pattern keeps dependencies explicit:

[source,text]
----
early phase creates mapping shape
        |
        v
register typed pending binding
        |
        v
later phase consumes pending binding
        |
        v
resolved product is materialized into org.hibernate.mapping
----

If a new feature cannot be completed in the member phase because another type,
table, identifier, or association is not ready yet, add typed pending state and
consume it in the phase where the dependency is guaranteed to exist.

After binding and materialization complete, `org.hibernate.mapping` contains a
full description of every table, column, identifier, association, and constraint
in the persistence unit.  The SessionFactory builder consumes these mapping
objects to produce the runtime metamodel and SQL generators, then discards them.


== Source Descriptors

Most attribute binders do not read raw annotations directly everywhere.  Instead,
they use small source descriptors that normalize source facts.

Examples:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/ColumnSource.java[`ColumnSource`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/TableSource.java[`TableSource`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/BasicValueSource.java[`BasicValueSource`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/ToOneSource.java[`ToOneSource`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/CollectionSource.java[`CollectionSource`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/internal/sources/ComponentSource.java[`ComponentSource`]

These descriptors are important because one Java member can produce multiple
mapping values.  A map-valued member, for example, has collection structure,
element value metadata, and key value metadata.  Each value may use a different
annotation vocabulary.

When adding support for an annotation family, first check whether the relevant
source descriptor should expose a normalized form of that information.


== Adding New Support

When adding support for a new annotation or XML construct, first decide what kind
of thing it is:

* A persistence-unit scoped registration belongs in
  link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/internal/GlobalRegistrationsImpl.java[`GlobalRegistrationsImpl`].
* A class-level persistent-type fact usually belongs in entity, mapped-superclass,
  embeddable, or hierarchy metadata.
* A member-level fact usually belongs in
  link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/AttributeMetadata.java[`AttributeMetadata`],
  link:../../hibernate-core/src/main/java/org/hibernate/boot/models/categorize/spi/KeyMapping.java[`KeyMapping`],
  or a source descriptor consumed later by binding.
* XML that changes annotation visibility or metadata-complete behavior must be
  handled before ordinary annotation categorization.

A useful workflow:

1. Identify where the source fact should be categorized.
2. Add or extend a source descriptor if the binder needs normalized annotation or
   XML facts.
3. Decide which phase contract owns the work: binding intent capture,
   value/table materialization, key/constraint materialization, cross-reference
   finalization, or validation.
4. Record the earliest binding fact or compatibility shell in the earliest safe
   phase.
5. If anything depends on later state, register typed pending state in
   link:../../hibernate-core/src/main/java/org/hibernate/boot/models/bind/spi/BindingState.java[`BindingState`].
6. Resolve the pending state in the narrowest phase that has the dependency
   available.
7. Materialize mapping objects through an existing materializer, or add a narrow
   resolved product plus materializer when the work creates physical mapping
   artifacts such as keys, constraints, indexes, or compatibility values.
8. Add focused tests for both the produced mapping object and the phase-ordering
   edge case.

Useful questions:

* Does binding need this information before any `org.hibernate.mapping` object
  exists, or can it be carried as a binding product until materialization?
* Is this a global declaration, or is it tied to one managed type/member?
* Does XML need to override annotations, replace annotations, or simply add more
  source facts?
* Can the binder consume a typed categorized contract, or would it need to
  re-read raw annotations?
* Is an ordering problem better modeled as a narrow phase or typed pending
  binding?
* Is this materialization ordering already owned by an existing phase, or does
  the feature expose a genuinely new cross-type dependency boundary?
* Is this code creating a physical table artifact?  If so, should it use a
  resolved product and a materializer instead of mutating a `Value`, `Table`, or
  `Collection` opportunistically?

Prefer adding a categorized contract over making the binder search the source
model opportunistically.


== Common Traps

Do not assume every class in
link:../../hibernate-core/src/main/java/org/hibernate/boot/pipeline/internal/source/AvailableResources.java[`AvailableResources`]
is persistent.  Package-info, plain converter classes, listener classes, and
non-persistent support classes may all appear in the visible source set.

*Do instead:* check the class's categorization role: entity, mapped superclass,
embeddable, or none.

Do not assume embeddables are part of entity inheritance.  They are tracked as
managed types, but entity hierarchy construction uses entities and mapped
superclasses.

*Do instead:* use the entity hierarchy APIs for inheritance questions and the
embeddable registry for embeddable lookups.  Keep the two paths separate.

Do not skip XML pre-processing when debugging annotation behavior.  XML can add
mapped classes and can change the annotation model that categorization sees.

*Do instead:* run the full categorization pipeline in your test or debug
scenario, including
link:../../hibernate-core/src/main/java/org/hibernate/boot/models/xml/spi/XmlPreProcessor.java[`XmlPreProcessor`],
even if the bug looks purely annotation-related.

Do not resolve non-primary-key association targets during ordinary member
binding.  Other members of the target type may not be bound yet.

*Do instead:* register a typed pending binding during the `Members` phase and
resolve it in the `AssociationTargets` phase, where all member bindings are
guaranteed to exist.

Do not create physical foreign keys as soon as a value is created.  FK creation
belongs after table keys, association targets, derived identifiers, and inverse
associations have done their work.

*Do instead:* record the FK requirement as pending state and create the physical
foreign key in the `ForeignKeys` phase through `ResolvedForeignKey` and
`ForeignKeyMappingMaterializer`.

Do not create primary keys, unique keys, indexes, or collection-table keys by
calling hidden mapping helper methods from whichever binder happens to have the
columns in hand.

*Do instead:* use the appropriate resolved product and materializer, such as
`ResolvedPrimaryTableKey`, `ResolvedDependentTableKey`,
`ResolvedCollectionTableKey`, `ResolvedUniqueKey`, or `ResolvedIndex`.  If the
artifact needs a new source of semantic information, add that information to the
binding product first and let the materializer perform the mapping mutation.

Do not assume collection indexes are simple columns.  Property map keys and
entity-valued map keys depend on element or target bindings.

*Do instead:* defer index resolution to the `CollectionIndexes` phase, which runs
after member bindings are complete.

Do not treat inverse associations as ordinary owning associations with a flag
flipped.  Inverse bindings often need to copy or reconcile state from an
owning-side mapping that is only complete in a later phase.

*Do instead:* handle inverse associations in the `InverseAssociations` phase,
copying or reconciling state from the owning-side mapping that is already
complete by that point.

Do not add binder-only side channels for source facts that are naturally known
during categorization.  If a later binding phase needs the fact, categorize it
once and carry it forward explicitly.

*Do instead:* add the fact to the appropriate categorized contract or metadata
object during categorization, so binders can consume it without re-reading raw
annotations or XML.
