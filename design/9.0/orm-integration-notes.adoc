= ORM Integration Notes
:toc:

These are concrete `org.hibernate.mapping` changes suggested by the phased binder prototype.
They are separate from prototype implementation details: the point is to remember which pieces should move into ORM proper when integrating this work.


== Represent custom SQL mutations as one mapping value

Custom SQL mutation details are represented by
link:../../hibernate-core/src/main/java/org/hibernate/mapping/CustomSqlMapping.java[`CustomSqlMapping`].
The value carries:

* the custom SQL string
* whether the statement is callable
* the `Supplier<? extends Expectation>` used to verify the JDBC outcome

`PersistentClass`, `Collection`, and `Join` store one `CustomSqlMapping` for each
custom mutation operation and expose value-object accessors such as
`getCustomSqlInsert()` / `setCustomSqlInsert(CustomSqlMapping)`.

The legacy `getCustomSQLInsert()`, `isCustomInsertCallable()`, and
`getInsertExpectation()` style accessors still delegate to the value object so
existing persister code can keep using the older shape while call sites migrate.
This keeps annotation-sourced `verify()` expectations, XML-sourced result-check
styles, and binder-created mapping state aligned around the actual mapping-model
shape.

NOTE: Decide if we want to carry this "one value type" idea into the runtime model (EntityPersister, CollectionPersister).

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/mapping/CustomSqlMapping.java[`CustomSqlMapping`] is the mapping-level value object.
* link:../../hibernate-core/src/main/java/org/hibernate/mapping/PersistentClass.java[`PersistentClass`], link:../../hibernate-core/src/main/java/org/hibernate/mapping/Collection.java[`Collection`], and link:../../hibernate-core/src/main/java/org/hibernate/mapping/Join.java[`Join`] store custom SQL mutations as `CustomSqlMapping` values.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/categorize/EntityTypeMetadata.java[`EntityTypeMetadata`] exposes custom insert, update, and delete mappings with this type.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/categorize/EntityTypeMetadataImpl.java[`EntityTypeMetadataImpl`] creates the value from `@SQLInsert`, `@SQLUpdate`, and `@SQLDelete`.


== Separate collection key, primary-key, and foreign-key creation

`org.hibernate.mapping.Collection#createAllKeys()` historically did too much for a phased binder.
It creates collection foreign keys and then creates the collection table primary key.

That makes it awkward to enforce a clean phase order:

* bind collection key value
* bind collection index and element values
* bind collection table primary key
* create/customize physical foreign-key constraints

The prototype can centralize most foreign-key creation in a `ForeignKeys` phase.
Collection tables use `Collection#createPrimaryKeyIfNeeded()` so the table-key
phase can create only the collection table primary key while leaving FK creation
to the explicit late phase.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TableKeyBinder.java[`TableKeyBinder`] calls `collection.createPrimaryKeyIfNeeded()` after setting the collection key.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ForeignKeyBinder.java[`ForeignKeyBinder`] owns the explicit late FK phase for association values and table-key FKs.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TypeBindingPhase.java[`TypeBindingPhase.ForeignKeys`] and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/BindingCoordinator.java[`BindingCoordinator`] show where the FK phase runs after collection indexes, association targets, table keys, and inverse associations.

When integrating upstream, keep this API split explicit, for example:

[source,java]
----
collection.createPrimaryKeyIfNeeded();
collection.createForeignKeys();
----

The important part is that physical FK creation/customization should happen in one explicit phase after table keys, association targets, collection indexes, and inverse structures are ready.

== Make table-key foreign keys first-class

`Join#createForeignKey()` hides the actual key value and target entity used to create the FK.
For phased binding, it is easier to record:

* the table key value
* the referenced entity name
* the source `@ForeignKey` details

and create/customize the FK later.

The prototype models this as typed pending state.
ORM proper could support this either by exposing more direct key-level APIs or by making FK metadata a first-class part of `Join` / secondary-table / association-table mapping.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TableForeignKeyBinding.java[`TableForeignKeyBinding`] is the local typed state for table-key FKs.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TableKeyBinder.java[`TableKeyBinder`] registers joined-subclass, secondary-table, association-table, and collection owner-key FKs instead of creating them immediately.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ForeignKeyBinder.java[`ForeignKeyBinder`] consumes both association-value FK bindings and table-key FK bindings in the `ForeignKeys` phase.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/context/BindingState.java[`BindingState`] and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/context/BindingStateImpl.java[`BindingStateImpl`] expose `addTableForeignKeyBinding` / `forEachTableForeignKeyBinding`.

== Retain source-model facts on mapping values

The prototype has a family of source descriptors that preserve Hibernate Models source facts long enough for later binding phases to make decisions without re-reading annotations or relying on captured lambdas.
This suggests that upstream `org.hibernate.mapping` values should retain more source-level context directly, especially now that this work is not constrained by legacy `hbm.xml` equivalence.

Examples of useful source facts:

* source `MemberDetails`
* effective `TypeDetails`
* source role/kind, such as attribute, collection element, map key, list index, embedded attribute, embedded identifier, or collection-element component
* source annotation vocabulary, or an equivalent role descriptor, that tells value resolution whether to read annotations such as `@JdbcType`, `@MapKeyJdbcType`, `@ListIndexJdbcType`, `@AnyKeyJdbcType`, or `@JdbcType`-as-`@Any`-discriminator metadata
* source association role, such as many-to-one versus owning one-to-one
* explicit Java type or converter source selected for this value
* effective target entity details for associations

Without this, later phases need binder-local side channels for type resolution, converter handling, map-key handling, association target resolution, and FK customization.
`MemberDetails` and `TypeDetails` are necessary but not sufficient: a single annotated member can contribute multiple mapping values, and each value may use a different annotation family.
For example, a map-valued member can contribute an element value that reads ordinary `@JdbcType` / `@JdbcTypeCode` annotations and a map-key value that reads `@MapKeyJdbcType` / `@MapKeyJdbcTypeCode`.
A Hibernate `@Any` member contributes both a discriminator value, controlled by ordinary `@JdbcType` / `@JdbcTypeCode`, and a key value, controlled by `@AnyKeyJdbcType` / `@AnyKeyJdbcTypeCode`.

The prototype currently solves this by carrying `BasicValueSource.Kind` and having `BasicValueBinder` register the appropriate explicit Java/JDBC type access functions on `BasicValue`.
When integrating upstream, either `BasicValue` needs to retain the source role and know which annotation vocabulary applies, or the binder needs to normalize these role-specific annotations into mapping-model metadata before later type resolution.
The important point is that storing only the source member/type still leaves `BasicValue` unable to distinguish `@JdbcType` from `@MapKeyJdbcType`, `@ListIndexJdbcType`, `@AnyKeyJdbcType`, and similar role-specific controls.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/BasicValueSource.java[`BasicValueSource`] records the source member, effective value type, basic-value role, explicit Java type override, and selected converter.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/BasicValueBinder.java[`BasicValueBinder`] uses that role to choose the correct annotation family and to install explicit Java/JDBC type access functions on `BasicValue`.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ToOneSource.java[`ToOneSource`] records the association member, logical association role, target type, optionality/fetch, join columns, join table, and association override.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/CollectionSource.java[`CollectionSource`] records the plural member, element type, map-key type, collection classification, table source, list-index source, and map-key source.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ComponentSource.java[`ComponentSource`] records component role, source member, component class, and path-aware adjustment state.

== Preserve component path-adjustment state

Embeddable binding is path-sensitive.
`@AttributeOverride`, `@AssociationOverride`, and `@Convert(attributeName = ...)` are declared on the owning member, but they apply to nested component members by relative path.

Upstream `org.hibernate.mapping.Component` should probably retain enough source/path context to resolve these adjustments consistently, instead of requiring each binder to pass parallel override/converter lookup functions.
This becomes more important for nested embeddables and embeddable collection elements, where the same Java embeddable class can be bound under different source paths with different adjustments.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ComponentSource.java[`ComponentSource`] exposes path-aware `columnSource`, `conversion`, and `associationOverride` lookups.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/PathAdjustmentCollector.java[`PathAdjustmentCollector`] collects path-keyed attribute overrides, association overrides, and conversion overrides from the source member.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/EmbeddableAttributeBinder.java[`EmbeddableAttributeBinder`] and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ComponentBinder.java[`ComponentBinder`] consume this source state while binding component members.

== Keep plural source classification and element/key type facts

A plural Java member contributes several mapping values: the collection container, the element value, and sometimes an index/key value.
Those values may use different source types and annotation vocabularies even though all annotations are declared on the same member.

Upstream collection mappings should retain the semantic source classification and the effective element/key type facts.
This avoids repeatedly recomputing whether a `List` is an indexed list or a bag, whether a map key is basic/entity/property-based, and which annotations apply to the element versus key.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/CollectionSource.java[`CollectionSource`] centralizes the collection classification rules, including `List` as indexed list unless Hibernate `@Bag` is present.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/BasicValueSource.java[`BasicValueSource`] distinguishes collection element, map key, and list-index basic values from ordinary attribute values.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/CollectionIndexBinder.java[`CollectionIndexBinder`] depends on collection source facts for list indexes, basic map keys, property map keys, and entity map keys.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ElementCollectionAttributeBinder.java[`ElementCollectionAttributeBinder`] and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/PluralAssociationAttributeBinder.java[`PluralAssociationAttributeBinder`] use `CollectionSource` to bind structurally different collection shapes from one source abstraction.

== Unify table and column source descriptors

The prototype uses small source interfaces to normalize annotation families that describe the same physical concept.
This reduces binder branching and keeps defaulting/quoting/naming logic closer to the physical concept being bound.

Upstream mapping objects should either keep similar source descriptors or expose equivalent normalized source metadata for table-like and column-like mappings.
This is especially useful when the same physical concept can come from different annotations, such as `@Column`, `@JoinColumn`, `@MapKeyColumn`, `@MapKeyJoinColumn`, `@OrderColumn`, `@Table`, `@JoinTable`, `@CollectionTable`, and `@SecondaryTable`.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ColumnSource.java[`ColumnSource`] normalizes column-like annotations for name, uniqueness, nullability, DDL type, length/precision/scale, options, and table.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/TableSource.java[`TableSource`] normalizes table-like annotations for name, schema, catalog, comment, and options.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ColumnBinder.java[`ColumnBinder`] consumes `ColumnSource`.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TableBinder.java[`TableBinder`] consumes `TableSource`.

== Preserve source foreign-key metadata on mapping structures

Some source-level FK details are currently only available while binding annotations:

* `@SecondaryTable#foreignKey`
* `@JoinTable#foreignKey`
* `@JoinTable#inverseForeignKey`
* `@CollectionTable#foreignKey`
* `@JoinColumn#foreignKey`
* `@MapKeyJoinColumn#foreignKey`

The phased binder needs those details after values and keys are structurally bound.
Upstream integration should either store this source metadata directly on the relevant mapping objects or expose source descriptors from them.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ForeignKeySource.java[`ForeignKeySource`] is the source descriptor used to carry FK names, constraint modes, definitions, and options across phases.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/relational/SecondaryTable.java[`SecondaryTable`] currently carries the secondary-table FK source so the late FK phase can customize the key FK.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationTableBinding.java[`AssociationTableBinding`] carries `@JoinTable#foreignKey` for association-table owner-key FKs.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/CollectionTableBinding.java[`CollectionTableBinding`] carries `@CollectionTable#foreignKey` for collection owner-key FKs.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/sources/ToOneSource.java[`ToOneSource#valueForeignKeySource`] preserves value-side FK metadata for owning to-one associations, including join-table inverse FK metadata.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/CollectionIndexBinder.java[`CollectionIndexBinder`] preserves `@MapKeyJoinColumn#foreignKey` for entity-valued map-key FKs.

== Property-ref association targets

Associations whose `referencedColumnName` points at a non-identifier property need all target members to be bound before the referenced property can be resolved.
This applies to normal to-one associations, plural association element FKs, and entity-valued map keys.

The prototype resolves this in an association-target phase and later creates the physical FK in the foreign-key phase.
ORM proper should avoid resolving these target properties during attribute/member binding.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationTargetBinding.java[`AssociationTargetBinding`] records association values whose join columns reference non-identifier target columns.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationTargetBinder.java[`AssociationTargetBinder`] resolves the referenced target property after all members have been bound.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TypeBindingPhase.java[`TypeBindingPhase.AssociationTargets`] and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/BindingCoordinator.java[`BindingCoordinator`] place this resolution after `Members` and `CollectionIndexes`, before `TableKeys` / `ForeignKeys`.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ToOneAttributeBinder.java[`ToOneAttributeBinder`], link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/PluralAssociationAttributeBinder.java[`PluralAssociationAttributeBinder`], and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/CollectionIndexBinder.java[`CollectionIndexBinder`] register these pending bindings for to-one associations, plural association elements, and entity-valued map keys.

== Unique property references

`InFlightMetadataCollector#addUniquePropertyReference` is still used as a side channel for property-ref associations.
When moving this work upstream, consider whether unique property references should be represented directly on the association mapping or folded into the same association-target resolution model.

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/UniquePropertyReferenceBinder.java[`UniquePropertyReferenceBinder`] centralizes the local `addUniquePropertyReference` side channel.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationTargetBinder.java[`AssociationTargetBinder`] uses it after resolving non-PK association targets.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/InverseToOneAssociationBinder.java[`InverseToOneAssociationBinder`] uses the same helper while resolving inverse one-to-one mappings whose value references the owning-side property.

The collector call is still a side channel, but it is now isolated in one prototype touchpoint.
When integrating upstream, consider whether this should be represented directly as association target metadata rather than as separate collector state.

== Derived identifiers

`@MapsId` needs dedicated typed state and phase ordering.
It connects identifier binding with association binding, which means it should not be handled as an ordinary to-one member during the coarse member phase.

The prototype now has two related phase concepts:

* association-valued identifiers, such as `@IdClass` plus `@Id @ManyToOne`, which need all target identifiers to exist before their identifier columns can be resolved
* derived identifiers, such as `@EmbeddedId` plus `@MapsId`, which need both the owner identifier component and the association member value to exist before the association can share identifier columns

Prototype touchpoints:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationIdentifierBinding.java[`AssociationIdentifierBinding`] records association-valued identifier attributes whose columns are resolved after all root identifiers exist.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/AssociationIdentifierBinder.java[`AssociationIdentifierBinder`] resolves those association-id columns, adds them to the primary key in identifier-property order, and registers the association FK for the late FK phase.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/DerivedIdentifierBinding.java[`DerivedIdentifierBinding`] records `@MapsId` associations while member values are bound.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/DerivedIdentifierBinder.java[`DerivedIdentifierBinder`] reconciles the association value with the owner identifier component, including implicit whole-id `@MapsId`, component-valued id parts, `@IdClass` scalar parts, and embeddable-contained `@MapsId` to-ones.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/ToOneAttributeBinder.java[`ToOneAttributeBinder`] registers derived identifier bindings rather than treating `@MapsId` as an ordinary to-one.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/IdentifierBinder.java[`IdentifierBinder`] registers association-valued identifier attributes rather than resolving target identifier columns during initial identifier construction.
* link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TypeBindingPhase.java[`TypeBindingPhase.AssociationIdentifiers`] runs after `Identifiers`, and link:../../hibernate-core/src/main/java/org/hibernate/boot/mapping/internal/binders/TypeBindingPhase.java[`TypeBindingPhase.DerivedIdentifiers`] runs after members and association-target resolution.

Upstream integration should keep these concerns separate.
Association-valued identifier attributes are part of identifier construction but still depend on target identifier availability.
`@MapsId` is part of member binding but depends on identifier and association values that are only complete after earlier phases.
