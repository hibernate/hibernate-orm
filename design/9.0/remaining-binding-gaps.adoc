= Remaining Model-Binding Gaps
:toc:

This document tracks remaining gaps in the ORM 9 boot-model path: source
collection, XML processing, categorization, phased binding, and production of
`org.hibernate.mapping` metadata.

The baseline for this audit is the legacy
`org.hibernate.boot.model.process.spi.MetadataBuildingProcess` pipeline:

* `ManagedResources` preparation;
* XML pre-processing;
* metadata-complete XML application;
* annotation categorization;
* XML overlay application;
* global registrations;
* entity hierarchy binding;
* second-pass processing for collections, associations, identifiers, queries,
  secondary tables, table keys, and foreign keys;
* additional mapping contributions;
* final metadata validation/exportable processing.

The new pipeline has clearer products and phases, but the remaining risks are
not just missing single annotations.  Most test-fix cycles have exposed
composition gaps: XML plus categorization, mapped-superclass plus table context,
embeddables plus path overrides, collections plus map-key/list-index state, and
association targets plus late foreign-key creation.

See also:

* `annotation-coverage.adoc` for annotation-level descriptor/source/binding
  coverage.
* `new-contributor-model-binding-guide.adoc` for the current source,
  categorization, and binding flow.
* `orm-integration-notes.adoc` for mapping-model changes suggested by the
  phased binder.
* `mapped-superclass-model-design.adoc` for the longer-term mapped-superclass
  model.
* `aggregate-mapping-design.adoc` for aggregate/UDT work.

== Status Definitions

`source-confirmed gap`::
The current source contains an explicit unsupported branch, or a known missing
behavior is visible from the implementation.

`composition parity gap`::
Individual annotations or binders exist, but old-pipeline combinations are not
yet proven or have recently failed in focused tests.

`test-only gap`::
Binding support appears present, but no focused mapping-state test proves the
behavior.

`intentional removal`::
Legacy behavior is not expected to survive ORM 9.

`deferred design area`::
The behavior should not be completed as an isolated binder patch because it
depends on a broader model or runtime design.

== Source-Confirmed Gaps

These are the highest-signal items because the current source still contains
explicit unsupported branches or TODO-level behavior in the new pipeline.

[cols="1,1,3", options="header"]
|===
| Area
| Status
| Notes

| Inverse `@OneToOne` edge cases
| source-confirmed gap
| Normal member-level inverse one-to-one binding has a dedicated path, but
component helper paths still reject inverse one-to-one association identifiers,
and `InverseToOneAssociationBinder` still rejects `mappedBy` through a joined
table.

| Embedded-id association identifiers with `@JoinTable`
| source-confirmed gap
| `ComponentBinder` rejects association identifiers in embedded ids when the
association uses a join table.  It also rejects inverse one-to-one association
identifiers.

| IdClass identifier attributes beyond basic/to-one
| source-confirmed gap
| `IdentifierBinder` only handles basic and to-one id attributes for IdClass.
Any remaining plural, component, or special association id shape needs either
explicit support or an ORM 9 rejection decision.

| `@MapsId` over unsupported identifier value shapes
| source-confirmed gap
| `DerivedIdentifierBinder` supports basic, component, and to-one identifier
parts.  Other identifier value shapes still fail explicitly.

| Plural members inside embeddables
| source-confirmed gap
| `ComponentBinder` supports plural embeddable members only for the currently
implemented element-collection / many-to-many paths.  Unsupported plural member
shapes still throw.

| Entity-valued map keys
| source-confirmed gap
| `CollectionIndexBinder` still has unsupported paths for entity-valued map keys
requiring `@MapKeyJoinColumn` support.

| `@MapKey(name)` for non-basic target properties
| source-confirmed gap
| `CollectionIndexBinder` currently implements property map keys for basic target
properties.  Component-valued, association-valued, or otherwise complex target
properties need parity decisions.

| Implicit `@MapKey`
| source-confirmed gap
| Implicit map-key binding is only implemented for entity-valued map elements.
Other map element shapes need coverage or explicit rejection.

| Inverse map-valued `@ManyToMany`
| source-confirmed gap
| `InversePluralAssociationBinder` handles basic and to-one owning map keys, but
still rejects other inverse map-key shapes.

| `@Any` key inference
| source-confirmed gap
| `AnyValueBinder` requires `@AnyKeyJavaClass` unless explicit discriminator
target mappings allow key type inference.  Inference is limited to basic target
identifiers with matching Java types.

| Column/table `options`
| source-confirmed gap
| `ColumnBinder` still throws for column options instead of applying them.  Table
options should be checked at the same time because `TableSource` carries them.

| Constructor-level `@Instantiator`
| source-confirmed gap
| `annotation-coverage.adoc` still lists `@Instantiator` as a binding gap.
Hibernate Models does not currently expose constructor-target annotation binding
in the way this pipeline needs.

| Dynamic XML mapped-superclass
| source-confirmed gap
| XML processing still leaves `<mapped-superclass/>` in dynamic models as an open
question.
|===

== Legacy Baseline Responsibilities

The old `MetadataBuildingProcess` provides a checklist for remaining pipeline
parity.  A slice is not complete merely because a binder exists; it is complete
when the equivalent old-pipeline responsibility has either moved into the new
pipeline, been intentionally removed, or been documented as deferred.

[cols="1,3,3", options="header"]
|===
| Legacy responsibility
| New-pipeline location
| Remaining audit question

| Source inventory from `ManagedResources`
| `AvailableResources` and `MappingSourceContributions`
| Verify every supported entry point feeds the same normalized source facts:
managed classes, package metadata, mapping files, converters, query imports, and
additional contributions.  Removed cfg.xml/HBM/loaded-class setting support
should stay removed.

| XML pre-processing
| `XmlPreProcessor` called from `DomainModelCategorizer`
| Verify XML is fully ignored when XML mapping is disabled and that mapping-file
classes discovered from XML participate in categorization before hierarchy
building.

| Metadata-complete XML
| `XmlProcessor.processXml(...)`
| Verify metadata-complete XML adds/replaces annotation usages before
categorization sees managed type roles, ids, attributes, callbacks, and package
defaults.

| XML overlays
| `XmlProcessingResult.apply()`
| Verify overlay XML is applied before ordinary class categorization and before
`ManagedTypeInheritanceState` is created.

| Superclass discovery
| `ManagedTypeInheritanceState`
| Verify JPA `exclude-unlisted-classes` behavior for automatically pulled-in
persistent superclasses: default should include undeclared persistent
superclasses; excluded units should fail instead of silently using them.

| Global registrations
| `BindingCoordinator.coordinateGlobalBindings()`
| Verify all old global side effects are represented: generators, converters,
types, named queries, result-set mappings, entity graphs, fetch profiles,
database objects, filters, imports, event listeners/callbacks.

| Entity hierarchy binding
| `EntityHierarchy`, `EntityTypeBinder`, `MappedSuperTypeBinder`
| Verify inheritance strategy, root/absolute-root handling, mapped-superclass
state, subclass tables, discriminator state, and secondary table state across
mixed entity/mapped-superclass chains.

| Second-pass queues
| Typed binding phases in `BindingCoordinator`
| Verify every old second-pass category has an equivalent typed phase or explicit
removal: collection second pass, to-one FK second pass, secondary table second
pass, joined-subclass FK, discriminator second pass, fetch override second pass,
query/result-set mapping second pass, generator resolver second pass, aggregate
component second pass, and exportable/key creation.

| Additional mapping contributions
| Metadata resolver / pipeline handoff
| Verify `AdditionalMappingContributor` equivalents still contribute classes,
class details, XML mappings, tables, sequences, and auxiliary database objects

| Final metadata cleanup
| Metadata finalization path
| Verify unique property refs, delayed property refs, natural-id unique keys,
foreign keys, collection keys, imports, named objects, and exportable producers
are complete before runtime construction.

== Composition Parity Buckets

These buckets are where the remaining failures are most likely to appear.  The
presence of individual annotation coverage is useful, but not sufficient to
close these.

=== XML and Source Handling

Open parity questions:

* XML-disabled mode must not parse, class-load, or apply mapping XML.
* Metadata-complete XML should create the persistent-type role before
  categorization.
* XML overlays should mutate the Hibernate Models annotation view before
  categorization and member binding.
* XML access defaults and class/member `@Access` overlays need parity with
  annotation access resolution.
* Package metadata from explicit packages and XML defaults should be visible to
  categorization and binding.
* Additional mapping contributions need an audited new-pipeline insertion point.

Recent related failures around XML not adding `@Entity` and bad XML being read
when XML mappings were disabled belong to this bucket.

=== Categorization and Inheritance

Open parity questions:

* `ManagedTypeInheritanceState` should continue deduplicating equivalent
  `ClassDetails` by class name, not object identity.
* Persistent superclasses automatically pulled into categorization should obey
  the source policy represented by `AvailableResources#includeUnlistedPersistentSuperclasses()`.
* Entity/mapped-superclass relation enum determination should be correct before
  binders decide root/subclass/mapped-superclass behavior.
* Mapped-superclasses above a root entity may need hierarchy-specific mapping
  contributions because their table context comes from the root entity.
* Shared mapped-superclasses across multiple hierarchies need coverage for
  table-specific columns, secondary-table routing, callbacks/listeners, and
  property closure behavior.
+
Progress note: the contribution-lite bridge applies mapped-superclass declared
properties only to nearest consuming entity boundaries, using separate property
copies for each consumer.  This addresses ownership/over-propagation symptoms
without completing the full descriptor/contribution split.
+
Progress note: the single-table sibling shared-column case represented by the
`e_title` transient-override tests is addressed by applying basic `@Column`
`insertable`/`updatable` flags to both the `Property` and its `BasicValue`
selectable flags during binding.  The remaining
`EmbeddableInheritanceMappedSuperclassAndGenericsTest` failure is an embeddable
inheritance/generics metamodel gap, not a duplicate-column validation failure.
* Joined, single-table, and table-per-class inheritance need combination tests
  with secondary tables, discriminators, mapped superclasses, and generated ids.

The longer-term modeling problem is covered by
`mapped-superclass-model-design.adoc`; until that design is implemented, this
bucket should remain open even if focused tests pass.

=== Component, Override, and Access Binding

Open parity questions:

* Component member discovery must remain superclass-first and access-aware.
* Owner-type path adjustments should be collected least-specific to
  most-specific, with member-level overrides last.
* Attribute overrides, association overrides, and conversion overrides must work
  through nested embeddables, inherited embeddables, mapped-superclass
  component members, collection elements, map keys, and embedded ids.
* Implicit nested column names should always route through the configured
  `ImplicitNamingStrategy`.
* XML access handling and attribute-level `@Access` validation are still
  follow-up areas.
* `@Parent`, `@EmbeddedColumnNaming`, `@EmbeddedTable`, custom component types,
  custom instantiators, and aggregate components need combination coverage.

This is the bucket that has repeatedly exposed failures in embedded inheritance,
mapped-superclass overrides, plural embeddables, and nested override tests.

=== Identifiers and Derived Identifiers

Open parity questions:

* Basic id, embedded id, IdClass, association-valued id, and derived id paths
  need combination coverage across inheritance, mapped-superclasses,
  embeddables, quoted identifiers, property refs, and composite target ids.
* `@MapsId` should be tested for whole-id, named-id-part, nested embeddable,
  IdClass scalar part, and association-valued-id combinations.
* Identifier generator resolution should be audited against the old
  generator-resolver second passes.
* IdClass non-basic/non-to-one id attributes need an explicit ORM 9 support or
  rejection decision.
* Embedded-id association identifiers with join tables are not implemented.

=== Associations and Foreign Keys

Open parity questions:

* Owning to-one, inverse one-to-one, to-one join tables, primary-key join
  columns, join formulas, property refs, not-found, and optionality should be
  tested in combinations rather than as isolated annotations.
* Association targets that reference non-primary-key properties must remain a
  typed late phase, not an eager member-binding lookup.
* Unique property reference side effects are still a collector side channel and
  should be audited after property-ref work.
* Foreign-key metadata from `@JoinColumn`, `@JoinTable`,
  `@CollectionTable`, `@MapKeyJoinColumn`, and secondary tables should survive
  until the late FK phase.
* Case/quoting-sensitive identifier matching should use `Identifier#matches(...)`
  semantics wherever logical table/column matching is performed.

=== Collections, Indexes, and Map Keys

Open parity questions:

* `List` without `@OrderColumn` should retain LIST semantics with inferred
  order-column defaults, unless explicitly mapped as `@Bag`.
* Basic, embeddable, entity, any-valued, many-to-any, and plural-association
  collection elements need coverage across bags, sets, lists, arrays, maps, and
  id-bags.
* Basic map keys, entity map keys, property map keys, embeddable map keys,
  borrowed map keys, `@MapKeyClass`, `@MapKeyColumn`, `@MapKeyJoinColumn`,
  `@MapKeyEnumerated`, `@MapKeyTemporal`, and Hibernate map-key type overrides
  need combination coverage.
* Inverse map-valued associations need more parity coverage, especially
  non-basic/non-to-one owning map keys.
* Sort, order-by, SQL-order, filters, join-table filters, where/restriction,
  collection custom SQL, collection cache, collection type, and collection id
  need cross-feature tests.

=== Global Metadata and Named Objects

Open parity questions:

* Named queries, native queries, named mutation statements, stored procedures,
  SQL result-set mappings, query cache layout, imports, and named entity graphs
  should be verified through metadata finalization and runtime use where
  possible.
* Fetch profiles and member-level fetch-profile overrides should be tested with
  inherited attributes and XML-sourced mappings.
* Filter definitions, auto-enabled filters, parameter type inference, entity
  filters, collection filters, and join-table filters need runtime-adjacent
  coverage.
* Global type registrations, converter registrations, Java/JDBC type
  registrations, custom user types, composite user types, collection types, and
  embeddable instantiators should be verified through both annotation and XML
  source paths.

=== Table, Column, and Schema Details

Open parity questions:

* Primary, secondary, joined-subclass, union-subclass, association, and
  collection tables need identifier matching that respects quoted identifiers.
* Column options are not implemented; table options should be audited.
* Unique constraints, indexes, checks, comments, row ids, partition keys,
  generated columns, column transformers, formulas, nationalized/time-zone
  columns, temporal/enum/lob handling, and custom DDL fragments need
  combination coverage.
* Table-key creation and FK creation should remain split and ordered: table keys
  first, physical FK customization later.
* Exportable database objects and auxiliary objects need coverage through schema
  generation.

=== Aggregate and Advanced Value Mapping

Open parity questions:

* `@Struct` remains deferred to aggregate/UDT design.
* Aggregate embeddables, arrays of aggregate values, JSON/XML aggregate storage,
  nested aggregate components, and aggregate validation should be handled under
  `aggregate-mapping-design.adoc`.
* `@Any` and `@ManyToAny` support exists but has source-confirmed limits around
  key inference and non-basic target identifiers.

== Test-Only Gaps To Keep Visible

These items appear implemented but should stay visible until focused
mapping-state tests prove the behavior in the new pipeline:

* `@TimeZoneStorage`
* `@TimeZoneColumn`
* `@View`
* XML/dynamic `@Abstract`
* XML/dynamic `@Extends`
* hierarchical package metadata, especially whether metadata from `com.acme`
  should apply to `com.acme.model`

== Intentional Removals and Deferred Areas

These should not be rediscovered as binder gaps unless the ORM 9 scope changes:

* cfg.xml support;
* HBM XML support;
* `AvailableSettings.LOADED_CLASSES`;
* `AvailableSettings.ORM_XML_FILES`;
* `AvailableSettings.HBM_XML_FILES`;
* `AvailableSettings.CFG_XML_FILE`;
* public `EntityManagerFactoryBuilder`;
* public `SessionFactoryBuilder`;
* `@SelectBeforeUpdate` as a Hibernate annotation;
* broader state-management/audit support for `Audited`, `Changelog`, and
  related annotations.

Remaining XML/runtime vestiges of removed features should be cleaned up in their
own slices rather than silently reintroduced through the new pipeline.
