= Remaining Model-Binding Gaps
:toc:

This document tracks remaining gaps in the ORM 9 boot-model path: source
collection, XML processing, categorization, phased binding, and production of
`org.hibernate.mapping` metadata.

The baseline for this audit is the legacy
`org.hibernate.boot.model.process.spi.MetadataBuildingProcess` pipeline:

* `ManagedResources` preparation;
* XML pre-processing;
* metadata-complete XML application;
* annotation categorization;
* XML overlay application;
* global registrations;
* entity hierarchy binding;
* second-pass processing for collections, associations, identifiers, queries,
  secondary tables, table keys, and foreign keys;
* additional mapping contributions;
* final metadata validation/exportable processing.

The new pipeline has clearer products and phases, but the remaining risks are
not just missing single annotations.  Most test-fix cycles have exposed
composition gaps: XML plus categorization, mapped-superclass plus table context,
embeddables plus path overrides, collections plus map-key/list-index state, and
association targets plus late foreign-key creation.

See also:

* `annotation-coverage.adoc` for annotation-level descriptor/source/binding
  coverage.
* `new-contributor-model-binding-guide.adoc` for the current source,
  categorization, and binding flow.
* `orm-integration-notes.adoc` for mapping-model changes suggested by the
  phased binder.
* `mapped-superclass-model-design.adoc` for the longer-term mapped-superclass
  model.
* `identifier-correspondence-model-design.adoc` for the follow-up model for
  non-aggregated identifier correspondence across virtual id attributes,
  IdClass/embedded-id representations, target id parts, and selectable order.
* `aggregate-mapping-design.adoc` for aggregate/UDT work.

== Status Definitions

`source-confirmed gap`::
The current source contains an explicit unsupported branch, or a known missing
behavior is visible from the implementation.

`composition parity gap`::
Individual annotations or binders exist, but old-pipeline combinations are not
yet proven or have recently failed in focused tests.

`test-only gap`::
Binding support appears present, but no focused mapping-state test proves the
behavior.

`intentional removal`::
Legacy behavior is not expected to survive ORM 9.

`deferred design area`::
The behavior should not be completed as an isolated binder patch because it
depends on a broader model or runtime design.

== Source-Confirmed Gaps

These are the highest-signal items because the current source still contains
explicit unsupported branches or TODO-level behavior in the new pipeline.

|===
| Area
| Status
| Notes

| Inverse `@OneToOne` edge cases
| test-only gap
| Normal member-level inverse one-to-one binding has a dedicated path, including
inverse `mappedBy` through primary tables, association tables, and secondary
tables.  Component helper paths also accept inverse one-to-one association
identifiers.  Keep focused runtime/schema coverage visible for composite and
joined-inheritance variants.

| Embedded-id association identifiers with `@JoinTable`
| test-only gap
| `ComponentBinder` now accepts embedded-id association identifiers that use a
join table, and inverse one-to-one association identifiers.  Keep focused
runtime/schema coverage visible for more complex composite join-table variants.

| IdClass identifier attributes beyond basic/to-one/embedded-id bridge
| test-only gap
| `IdentifierBinder` now handles basic id attributes, association-valued id
attributes, entity members whose IdClass member supplies `@ManyToOne`, and the
`@EmbeddedId` plus IdClass bridge used by `@MapsId`.  Component-valued IdClass
parts, including implicit embeddable id parts and component parts used by
`@MapsId`, are now characterized as supported.  Plural and `@Any` mappings are
not supported as identifier parts.  IdClass association identifiers with join
tables are now characterized against both the new and legacy boot paths.

| `@MapsId` over unsupported identifier value shapes
| source-confirmed gap
| `DerivedIdentifierBinder` supports basic, component, and to-one identifier
parts, including IdClass component parts.  Other identifier value shapes still
fail explicitly.

| Plural members inside embeddables
| test-only gap
| `ComponentBinder` supports plural embeddable members only for the currently
implemented element-collection / one-to-many / many-to-many / many-to-any
paths when the embeddable is a value.  Embeddables used as entity identifiers
now explicitly reject plural members, including unannotated collection members
and nested plural members, with focused `EmbeddableBindingTests` coverage.
Direct and nested plural members inside non-id value embeddables now have
focused binding-model and materialization guardrails; remaining work is broader
combination coverage.

| Entity-valued map keys
| test-only gap
| `CollectionIndexBinder` now supports entity-valued map keys when explicit
`@MapKeyJoinColumn` / `@MapKeyJoinColumns` metadata is present, including
non-primary-key references through late association-target binding.  It also
supports implicit primary-key references when no explicit map-key join columns
are declared.  Keep focused coverage on composite implicit key shapes and
runtime SQL behavior.

| `@MapKey(name)` for non-basic target properties
| test-only gap
| `CollectionIndexBinder` implements property map keys for basic, to-one, and
component-valued target properties.  Remaining exotic/non-standard target value
shapes need explicit parity decisions if they are found in legacy behavior.

| Implicit `@MapKey`
| test-only gap
| Implicit map-key binding is implemented for entity-valued map elements, where
the associated entity identifier is borrowed as the map key.  Non-entity map
element shapes now fail with an explicit `MappingException` because there is no
associated entity identifier to borrow.

| Inverse map-valued `@ManyToMany`
| test-only gap
| `InversePluralAssociationBinder` handles basic, to-one, and component-valued
owning map keys, including implicit entity-valued map keys and to-one/component
`@MapKey(name)` target properties.  Remaining complex owning map-key index
shapes need explicit parity decisions if they are found in legacy behavior.

| `@Any` key inference
| test-only gap
| `AnyValueBinder` accepts explicit key type metadata from `@AnyKeyJavaClass`,
`@AnyKeyJavaType`, and XML named `@AnyKeyType`.  It can infer a key Java type
from explicit discriminator target mappings when all target identifiers are
basic values with matching Java types.  Non-basic target-id inference is
covered as an explicit unsupported shape for singular `@Any` and plural
`@ManyToAny`.

| Column/table `options`
| test-only gap
| `TableBinder` applies `TableSource` options and the resolved unique-key/index
materializers apply key/index options.  `ColumnBinder` now applies column
options from unified column sources, including map-key join columns.

== Focused Gap Refresh: 2026-06-20

This refresh was taken after the explicit key/materialization slice described
in `current-work-handoff.adoc`.

Source-confirmed refresh:

* Table-key, foreign-key, unique-key, and index creation are no longer good
  top-level gap buckets by themselves.  Current source routes the important
  new-pipeline and cheap legacy callers through resolved products such as
  `ResolvedForeignKey`, `ResolvedUniqueKey`, `ResolvedIndex`, and
  `ResolvedPrimaryTableKey`, then through materializers.
* Collection-table unique constraints and indexes now have resolved
  materialization coverage.  Collection-table owner keys still deserve focused
  parity testing because they combine implicit naming, explicit join columns,
  FK ordering, PK/UK finalization, and inverse collection handling.
* Entity-valued map keys with explicit `@MapKeyJoinColumn` metadata and
  implicit primary-key references are covered in boot-model tests.  The
  previously source-confirmed component-valued `@MapKey(name)` and inverse
  many-to-many component-index gaps are now covered by focused boot-model
  tests; keep broader combination coverage as test-only follow-up.
* Table options are applied at table binding time.  Column options now
  propagate through unified column sources, including map-key join columns.
* Identifier binding should stay narrowed to newly observed failing shapes.
  The recent identifier/materializer work covers the main simple, embedded,
  IdClass bridge, association-id, and derived-id guardrails.

=== XML and Source Handling

Open parity questions:

* XML-disabled mode must not parse, class-load, or apply mapping XML.
* Metadata-complete XML should create the persistent-type role before
  categorization.
* XML overlays should mutate the Hibernate Models annotation view before
  categorization and member binding.
* XML access defaults and class/member `@Access` overlays need parity with
  annotation access resolution.
* Package metadata from explicit packages and XML defaults should be visible to
  categorization and binding.
* Additional mapping contributions need an audited new-pipeline insertion point.

Recent related failures around XML not adding `@Entity` and bad XML being read
when XML mappings were disabled belong to this bucket.

Audit note:

* The intended ORM 9 XML/source process is:
  first resolve all referenced, available `ClassDetails`; then, for
  metadata-complete XML mappings, clear all existing annotations and apply the
  complete XML mapping; then, for non-complete XML mappings, overlay XML mapping
  annotations onto the resolved model; then categorize the final Hibernate
  Models view; then bind.
* The legacy `MetadataBuildingProcess` ordered XML work as pre-process XML,
  process metadata-complete XML, categorize known annotation classes/names, then
  apply overlay XML.  The new pipeline should not mechanically restore that
  legacy order because categorization now depends on seeing XML-created
  `@Entity`, `@MappedSuperclass`, or `@Embeddable` annotations.
* Source-adapter behavior still needs focused guardrails.  In the main pipeline
  path, XML mappings discovered as resources, files, or URLs are filtered by
  `MappingSettings.XML_MAPPING_ENABLED` before binding.  The persistence-unit
  adapter also collects visible `META-INF/orm.xml` URLs as source
  contributions; no current failure has been tied to that broadening, but it
  should be covered so persistence-unit mapping-file semantics stay explicit.
* Focused XML tests should prove the full phase boundary:
  XML-disabled resources are ignored before binding/parsing failures,
  metadata-complete XML creates persistent roles before categorization,
  overlay XML mutates annotations before categorization snapshots source roles,
