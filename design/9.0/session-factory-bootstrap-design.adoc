= SessionFactory Bootstrap Design
:toc:

This note describes the intended end-state design for building a
`SessionFactory` / `EntityManagerFactory` with the boot-model pipeline.

It is deliberately separate from
link:session-factory-bootstrap-status.adoc[`session-factory-bootstrap-status.adoc`],
which tracks current implementation status and next slices.

== Design Goal

SessionFactory bootstrap should be an explicit pipeline of named products rather
than a long inline flow hidden behind one entry point.

The immediate compatibility target is still ORM's current
`SessionFactoryImpl` constructor:

[source,java]
----
SessionFactoryImpl(
        MetadataImplementor bootMetamodel,
        SessionFactoryOptions options,
        BootstrapContext bootstrapContext)
----

The design goal is not necessarily to preserve these constructor inputs forever.
The goal is to make the products represented by these inputs explicit, testable,
and replaceable one slice at a time.

Current implementation progress and follow-up work are tracked in
link:session-factory-bootstrap-status.adoc[`session-factory-bootstrap-status.adoc`].


== Top-Level Flow

[NOTE]
====
Pipeline sketches in this document use the shared notation described in
link:flow-notation.adoc[`flow-notation.adoc`].
====

.Top-level bootstrap pipeline
[source]
----
Entry point
    -> ResolvedBootstrapSettings
    -> MappingSources
    -> PreparedMappingSources
    -> CategorizedDomainModel
    -> ResolvedMapping
    -> ResolvedSessionFactorySettings
    -> SessionFactoryImplementor
----

Each arrow should be owned by a small resolver or builder with one gross target.
The entry point chooses or creates the inputs, but should not perform all
bootstrap work itself.


== Entry Points

Supported entry points should converge quickly on the same pipeline:

* JPA SE bootstrap via `Persistence`
* JPA SE bootstrap via `HibernatePersistenceConfiguration`
* JPA EE bootstrap via persistence-unit descriptor style input
* legacy `Configuration`, preferably as a compatibility layer
* possible SPI entry accepting settings plus already-normalized source
  contributions

Entry points are responsible for collecting user intent:

* raw configuration values
* explicit managed classes, packages, mapping XML, and query imports
* persistence-unit identity and JPA bootstrap mode
* service-registry choices

Entry points should not own categorization, binding, metadata finalization, or
runtime factory construction.

Current branch status:

* `org.hibernate.boot.pipeline.internal.BootstrapPipeline` is the canonical
  internal facade for one-shot SessionFactory bootstrap.
* `PersistenceConfiguration` bootstrap goes through this facade.
* `PersistenceUnitDescriptor` bootstrap for
  `HibernatePersistenceProvider#createEntityManagerFactory(String, Map)`,
  `createContainerEntityManagerFactory(PersistenceUnitInfo, Map)`, and their
  schema-generation counterparts goes through this facade.
* The old JPA two-phase builder facade has been removed.  Metadata-only callers
  use `BootstrapPipeline.resolveMetadata(...)`; schema tooling uses
  `BootstrapPipeline.generateSchema(...)`; factory creation uses
  `BootstrapPipeline.build(...)`.
* Descriptor-based archive scanning and container class-transformer push are
  integrated into the shared pipeline.

=== Current Entry-Point Map

The branch currently has three broad SessionFactory construction families.
They should be understood separately because they represent different
compatibility pressures.

JPA provider entry points::
The public `jakarta.persistence.spi.PersistenceProvider` methods implemented by
`org.hibernate.jpa.HibernatePersistenceProvider` now delegate to
`org.hibernate.boot.pipeline.internal.BootstrapPipeline`:
+
* `createEntityManagerFactory(String, Map)` resolves a matching
  `PersistenceUnitDescriptor` from `persistence.xml`, then calls
  `BootstrapPipeline.build(PersistenceUnitDescriptor, Map)`.
* `createContainerEntityManagerFactory(PersistenceUnitInfo, Map)` wraps the
  `PersistenceUnitInfo` in `PersistenceUnitInfoDescriptor`, then calls the same
  descriptor-based bootstrap method.
* `createEntityManagerFactory(PersistenceConfiguration)` calls
  `BootstrapPipeline.build(PersistenceConfiguration)`.
* The matching schema-generation methods call the corresponding
  `BootstrapPipeline.generateSchema(...)` methods.
+
These are the canonical JPA runtime paths for this branch.

JPA pipeline operations::
The removed two-phase JPA builder facade is not replaced by another builder.
The supported operations are explicit pipeline calls:
+
* `BootstrapPipeline.build(...)` for EMF/SF creation.
* `BootstrapPipeline.generateSchema(...)` for schema tooling.
* `BootstrapPipeline.resolveMetadata(...)` for metadata inspection.
* `SettingsResolver` products for tests or tooling that assert resolved setting
  behavior.
+
Symbolic JPA integration setting names live on
`org.hibernate.jpa.boot.spi.JpaSettings`.  Custom factory creation is handled by
the SessionFactory construction producer hook, including named producer
selection.

Native Hibernate entry points::
Native bootstrap remains a first-class public API and still uses the existing
ORM boot contracts:
+
* `org.hibernate.cfg.Configuration#buildSessionFactory(...)` gathers mappings
  and convenience registrations, builds `Metadata`, collects
  `Configuration`-level factory customizations into a
  `SessionFactoryOptionsCollector`, and calls `SessionFactoryPipeline`.
* `org.hibernate.boot.Metadata` is retained as a resolved-model inspection
  surface.  Its factory-building methods have been removed.
* `org.hibernate.boot.Metadata#getSessionFactoryBuilder()` and the public
  `org.hibernate.boot.SessionFactoryBuilder` surface have been removed.
* Hibernate's test infrastructure and many native tests build factories through
  `Metadata` or `Configuration` in order
  to exercise native-specific customizers such as interceptors, tenant
  resolvers, statement inspectors, entity-name resolvers, auxiliary database
  objects, and type/function contributors.
+
These APIs are not simply aliases for JPA bootstrap.  They are part of the
native Hibernate programming model.  The new orchestration pipeline should
eventually be able to serve them, but that requires preserving the public
customization capabilities or providing a credible replacement.
+
Current branch status:
+
* The public native `org.hibernate.boot.SessionFactoryBuilder`,
  `SessionFactoryBuilderFactory`, `SessionFactoryBuilderService`, and default
  builder implementation have been removed.
* Native metadata production now routes through `MappingResolutionPipeline`.  The
  returned `MetadataImplementor` is an internal view over `ResolvedMapping`.
* `Configuration#buildSessionFactory(...)` remains as a compatibility adapter,
  but its final factory customization state is collected in
  `SessionFactoryOptionsCollector` and passed to `SessionFactoryPipeline`.
* The new pipeline should not accept finalized `MetadataImplementor`, legacy
  `SessionFactoryOptions`, or legacy `BootstrapContext` as native-entry
  handoff products.  That would move the compatibility shapes into the pipeline
  instead of replacing them.
* Native source declarations are normalized through `MappingSources`
  and resolved into `PreparedMappingSources` during source resolution.
+
Design direction:
+
* `Configuration` should become a legacy compatibility layer and, once the
  replacement path covers its supported convenience methods, should be
  deprecated in favor of `PersistenceConfiguration` /
  `HibernatePersistenceConfiguration`.
* Native bootstrap should use `HibernatePersistenceConfiguration` for direct
  SessionFactory creation.  `Metadata` remains available as a resolved-model
  inspection view.

==== Replacement Native Workflow

The replacement native workflow should preserve the useful capabilities of the
old staged native contracts without cloning their public shape.

The target workflow is:

.Replacement native bootstrap workflow
[source]
----
HibernatePersistenceConfiguration
    -> MappingSources / PreparedMappingSources
    -> ResolvedBootstrapSettings
    -> ResolvedMappingSettings
    -> ResolvedMapping
    -> ResolvedSessionFactorySettings
    -> SessionFactoryImplementor
----

`HibernatePersistenceConfiguration` is the intended user-facing programmatic
bootstrap API and the replacement for the useful native roles of the removed
staged source and builder APIs.  Its covariant
`createEntityManagerFactory()` terminal method returns `SessionFactory`, so
native users do not need a separate bootstrap type.
+
`HibernatePersistenceConfiguration` should collect source declarations and
settings, then build a `SessionFactory` or perform schema generation through
the new pipeline.  Metadata resolution remains an internal phase; the public
native API should not lead with a durable `buildMetadata()` product that
recreates the old two-phase workflow.
+
`MappingSources` is the immutable internal source product created
from `HibernatePersistenceConfiguration` and other entry-point adapters.  It is
the renamed target for the former `BootstrapSourceContributions` concept.  The
record should stay implementation-facing.
+
Native XML source handling should retain source declarations until source
resolution binds them into `PreparedMappingSources`.

Programmatic metadata-build customization should stay deliberate.  Java service
loading remains the preferred path for contributors; direct customization
methods should be added to `HibernatePersistenceConfiguration` only where there
is a clear user-facing need, rather than copying the old builder wholesale.
+
`MappingCustomizations` is the immutable internal customization product created
from entry-point adapters.  It carries programmatic metadata-build instructions
such as query imports, contributor instances, and explicit cache-region
definitions, keeping those concerns out of `MappingSources`.

The metadata inspection surface should be deliberate.  `Metadata` is now a
thin inspection contract rather than a staged factory-building surface, so there
is no strong design pressure to remove it.  Native users still need a stable
resolved-model view before building a factory, and `Metadata` can serve that
role as the public view over the internal `ResolvedMapping` product.

The factory build request is the replacement for the old public staged factory
builder role.  It should collect runtime factory customizations into
`ResolvedSessionFactorySettings` or a narrower factory-build request.  The
request should be consumed by the SessionFactory construction phase, not by
materializing legacy `SessionFactoryOptions` as the design target.

The current branch has started this migration by returning an internal
`MetadataImplementor` view over `ResolvedMapping` from the default native
metadata path.  That view keeps existing `Metadata` callers working while
preserving the full `ResolvedMapping` for any downstream factory construction,
so categorization and binding are not rebuilt.  The cleanup target is the
adapter boundary, such as `ResolvedMappingImplementor`, not the public
`Metadata` inspection contract itself.

==== Deprecation Preconditions

`MetadataSources`, `MetadataBuilder`, `Metadata#buildSessionFactory()`, and the
public `SessionFactoryBuilder` surface have been removed on this branch.
`Metadata` should be retained as the public resolved-model inspection view
unless later work finds a concrete incompatibility with that role.

The remaining public compatibility question is `Configuration`: decide whether
to keep it as a native facade over `BootstrapPipeline` or deprecate the
entry points that only exist to support staged metadata building.

Before deprecation, the branch should satisfy these conditions:

* Native source contribution has a replacement for the supported source use
  cases, including class and class-name registration, package metadata, XML
  mappings, query imports, and non-resource mapping sources that remain
  supported.
* Metadata build customization has replacements for the supported former
  builder methods, with each mapped to source contribution, mapping settings,
  metadata-build request state, or explicitly legacy-only behavior.
* Metadata inspection has either a replacement API or a documented view over
  `ResolvedMapping`.
* `Metadata -> SessionFactory` continues to reuse the same `ResolvedMapping`
  product, including the categorized model and binding state.
* Factory customizations are collected into a factory-build request or resolved
  settings product before construction.
* `Configuration` and legacy JPA builder facades adapt to the new workflow or
  have explicit deprecation guidance for unsupported compatibility behavior.
* Tests cover both direct one-shot bootstrap and staged native bootstrap through
  source collection, metadata resolution, metadata inspection, factory
  customization, and factory construction.

Two-phase bootstrap::
The old need for a two-phase JPA `EntityManagerFactory` build is gone.  This
branch no longer has active `BootstrapPipeline.prepare(...)` or
`PreparedBootstrapPipeline` call sites.
+
When tests or tooling need to assert intermediate products, they should prefer
building the specific product under test, such as metadata, source
contributions, or resolved settings, instead of preserving a general two-phase
SessionFactory bootstrap.

=== Current Test Signal

The complete `hibernate-orm` suite is green in the current worktree as of
2026-07-02.  Representative native bootstrap tests through `Configuration`,
`HibernatePersistenceConfiguration`, metadata inspection, and the pipeline
construction path remain the useful control signal for future changes.

Longer-term bootstrap cleanup is concentrated in the JPA provider paths that
now route through `BootstrapPipeline`:

* provider exception wrapping parity;
* persistence-unit setting precedence parity;
* descriptor-backed source contribution parity now that legacy `cfg.xml` support
  has been removed;
* schema-generation script handling.

Those follow-up areas are tracked in
link:session-factory-bootstrap-status.adoc[`session-factory-bootstrap-status.adoc`].

== Settings Model

`ResolvedBootstrapSettings` is the entry-point and service-setup settings
product.  It owns the immutable raw configuration map plus named values needed
early in bootstrap, such as whether the entry point is JPA bootstrap.

Specialized settings products are peers in the orchestration flow and may be
derived from its normalized configuration map:

* `ResolvedMappingSettings` for source interpretation and mapping processing
* `ResolvedSessionFactorySettings` for runtime factory construction

This keeps shared setting interpretation centralized while avoiding one
monolithic settings object where every phase sees every option.  Mapping
consumers should receive `ResolvedMappingSettings` explicitly rather than
reaching through `ResolvedBootstrapSettings`.

`SettingsResolver` is the orchestration-facing facade for the three buckets.  It
delegates to `BootstrapSettingsResolver`, `MappingSettingsResolver`, and
`SessionFactorySettingsResolver` rather than collapsing their phase-specific
rules into a single implementation.

Named settings should be added only when a concrete phase needs them.  The raw
map remains available for compatibility, integration code, and settings that
have not yet earned a named slot.

== Source Products

`MappingSources` represents entry-point source contributions after
entry-point-specific discovery has run.  It is an internal phase-boundary
product, not the public collector API.

`PreparedMappingSources` represents source material that can be categorized:

* managed class details
* package-info class details
* JPA `orm.xml` mapping bindings

Programmatic metadata customizations such as query imports are intentionally
not part of `PreparedMappingSources`; they flow through `MappingCustomizations`
and later binding/customization state.

Scanning belongs before or inside source-contribution creation, depending on the
entry point.  Categorization should receive available source material, not an
entry-point object.

Legacy `hbm.xml` support should not be treated as part of the target ORM 9 boot
model pipeline.  Removing it upstream is an enabling cleanup: it lets source
discovery, XML preprocessing, mapping defaults, and metadata binding focus on
the annotation/JPA XML model instead of preserving legacy mapping equivalence.

== Domain Model Products

`CategorizedDomainModel` is the interpreted domain model before ORM boot mapping
binding.

It owns higher-level decisions such as entity hierarchies, attributes, access
type, identifiers, callbacks, filters, and global registrations.  It should not
write to ORM's `PersistentClass` or `Collection` model directly.

XML preprocessing and XML interpretation currently live on the categorization
side.  A separate prepared-source phase can emerge later if XML or other sources
need enough interpretation to justify a dedicated product between
`PreparedMappingSources` and `CategorizedDomainModel`.

That XML path means JPA `orm.xml`.  Legacy `hbm.xml` should be removed upstream
rather than carried forward as a permanent source kind in this design.

== Metadata Target

`MappingResolutionPipeline` owns the metadata collector:

.MappingResolutionPipeline
[source]
----
ResolvedBootstrapSettings
MappingSources
ServiceRegistry
    -> PreparedMappingSources
    -> CategorizedDomainModel (*)
    -> BindingState (*)
    -> MetadataCollector
    -> MetadataImplementor (*)
    -> ResolvedMapping
----

`ResolvedMapping` should preserve both the finalized ORM metadata and the
important intermediate boot-model products:

* finalized `MetadataImplementor`
* `CategorizedDomainModel`
* `BindingState`

The binder should use `BindingState` for phase-local working state and
`MetadataCollector` for completed metadata contributions.  The current
`MetadataCollector` implementation is an adapter over ORM's
`InFlightMetadataCollector`, but that collector should be treated as a
compatibility implementation, not as a design peer of `BindingState`.  Where
upstream contracts force direct table/database creation, query binding, second
passes, or mapping constructors through ORM's legacy collector/context APIs,
those touchpoints should remain localized and documented.

Metadata finalization must include the type-contribution work normally performed
by ORM bootstrap so the `TypeConfiguration`, JDBC type registry, and DDL type
registry are usable by runtime factory construction.

== SessionFactory Target

The SessionFactory construction phase owns only the runtime factory target:

.SessionFactory construction
[source]
----
ResolvedSessionFactorySettings
ResolvedMapping
ServiceRegistry
    -> SessionFactoryImplementor
----

The current pipeline adapts `ResolvedSessionFactorySettings` to ORM's
`SessionFactoryOptions` because `SessionFactoryImpl` still requires that SPI.
That adapter is a compatibility boundary, not the desired long-term shape.

The long-term direction is to classify `SessionFactoryOptions` consumers into
smaller contracts:

* query-engine options
* cache options
* session defaults
* integration hooks
* service-registry build inputs
* SQL generation defaults
* state-management and temporal/audit options

As those consumers are audited, `SessionFactoryImpl` and its collaborators can
accept narrower resolved products instead of the whole options SPI.

Factory-scoped service-registry creation should become an explicit late
construction phase.  The current ORM path builds the
`SessionFactoryServiceRegistry` from inside `SessionFactoryImpl` with a
partially initialized factory and the full `SessionFactoryOptions` SPI.  The
target design should instead pass a narrower service-registry build context:
parent services, class-loading access, resolved settings needed by
factory-scoped services, and a factory identity/build handle.

Integrator execution should likewise be visible as a factory-integration phase.
The legacy `Integrator` SPI may need to remain as a compatibility adapter, but
new integration code should not require a broad `BootstrapContext` or a fully
exposed partially initialized `SessionFactoryImplementor`.

Runtime model creation should also be an explicit named factory-construction
phase.  The current ORM `RuntimeModelCreationContext` usefully identifies that
phase, but it exposes too much: finalized metadata, raw settings, the legacy
`BootstrapContext`, session-factory options, service registry, generator state,
and a partially initialized factory.  The target shape should keep the phase but
narrow the context to explicit runtime-model inputs:

* finalized metadata products needed for persister construction;
* the runtime mapping registry under construction;
* runtime SQL/query services;
* class-loading, model, bean, and representation-strategy services;
* resolved mapping/runtime settings;
* generator creation state;
* a completed-safe factory build handle instead of the incomplete factory.

== Bootstrap Context

`BootstrapContext` is still useful as a compatibility bridge, but it is too broad
as a permanent contract.

The end-state should split its responsibilities into narrower products where
that makes integration clearer:

* model/class-loading context
* type configuration and type contribution context
* bean/container access
* service access
* integration callbacks
* mapping defaults

Until those splits are proven, the PoC should reuse ORM's existing
`BootstrapContext` rather than inventing parallel infrastructure.

== Ownership Rules

The intended ownership boundaries are:

* entry points gather input and choose the initial services/settings
* settings resolvers interpret settings
* source contribution factories gather and normalize source inputs
* categorization interprets source metadata into boot-model metadata
* binding writes categorized metadata to the metadata collector
* metadata resolution finalizes ORM boot metadata
* SessionFactory building constructs the runtime factory

No phase should reach backward into entry-point-specific APIs once its inputs
have been normalized.

== Integration Direction

For upstream integration, the likely path is incremental:

* keep current ORM boot contracts only where they are still useful as temporary
  constructor boundaries
* replace source discovery and categorization in isolated slices
* bind to ORM boot metadata with localized compatibility bridges
* move settings interpretation into named resolved settings products
* narrow `BootstrapContext`, `MetadataBuildingContext`, and
  `SessionFactoryOptions` consumers as each use is audited
* apply upstream ORM changes whenever a narrow contract is clearer than another
  local compatibility adapter
* use deeper constructor or service-registry changes as deliberate integration
  slices, not as last-resort work

The design should stay friendly to Hibernate integrations which currently depend
on builder customization hooks.  Compatibility adapters are acceptable during
the transition, but they should not become artificial constraints on the design.
When an upstream ORM change removes a broad dependency cleanly, prefer that
change and document the migration pressure it addresses.  New long-term APIs
should expose named products rather than one all-purpose bootstrap object.
