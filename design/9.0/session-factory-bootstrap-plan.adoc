= SessionFactory Bootstrap Plan
:toc:

This note sketches the next PoC direction: use the boot-model pipeline to build
enough ORM boot state to instantiate a `SessionFactoryImpl` / `EntityManagerFactory`.

This document is the ongoing PoC work log and gap tracker.  The intended
end-state design is captured separately in
link:session-factory-bootstrap-design.adoc[`session-factory-bootstrap-design.adoc`].

The plan works backward from `org.hibernate.internal.SessionFactoryImpl` because
that constructor is the current runtime boundary.

== Constructor Boundary

`SessionFactoryImpl` currently accepts three inputs:

[source,java]
----
SessionFactoryImpl(
        MetadataImplementor bootMetamodel,
        SessionFactoryOptions options,
        BootstrapContext bootstrapContext)
----

Those inputs are not independent.  They are all products of bootstrap settings,
service-registry construction, source discovery, model categorization, and boot
model binding.

The PoC should make those products explicit rather than recreating the current
inline flow from `EntityManagerFactoryBuilderImpl` and `MetadataBuildingProcess`.

== Some Assumptions

First, a few assumptions we will be working under...

* We no longer need the 2-phase split in building the EMF from JPA container.  This was something we did for WildFly to address a gap in the EE and JPA spec which should no longer be an issue.
* We will need to come back and address other Hibernate projects (Search, Reactive, etc) concerns.  Some of them contribute specialized builders, and we need to make sure those underlying needs are addressed.
* Limit set of bootstrap entries -
** JPA "SE bootstrap" - `Persistence`
** JPA "SE bootstrap" - `(Hibernate)PersistenceConfiguration`
** JPA "EE bootstrap" - Ultimately `PersistenceUnitDescriptor`
** Legacy `Configuration`, but deprecated in favor of `HibernatePersistenceConfiguration`
** Maybe some form that accepts `AvailableResources` + settings?


== Current Bootstrap Entry-Point Status

This branch has moved the main JPA provider entry points onto
`org.hibernate.boot.orchestration.SessionFactoryBootstrap`, but several older
paths remain deliberately alive.

=== Canonical Path On This Branch

`org.hibernate.boot.orchestration.SessionFactoryBootstrap` is currently the
canonical one-shot bootstrap facade for the new pipeline.

It has entry-point adapters for:

* `jakarta.persistence.PersistenceConfiguration`
* `org.hibernate.jpa.boot.spi.PersistenceUnitDescriptor` plus integration
  settings
* schema generation for the same two source shapes
* already-resolved `SessionFactoryBootstrapRequest`

The `HibernatePersistenceProvider` public provider methods now use this facade
for:

* `createEntityManagerFactory(String, Map)`
* `createContainerEntityManagerFactory(PersistenceUnitInfo, Map)`
* `createEntityManagerFactory(PersistenceConfiguration)`
* `generateSchema(String, Map)`
* `generateSchema(PersistenceUnitInfo, Map)`
* `generateSchema(PersistenceConfiguration)`

This means the normal JPA SE, JPA EE/container, and
`PersistenceConfiguration` provider paths are no longer routed through
`EntityManagerFactoryBuilderImpl`.

=== Remaining EntityManagerFactoryBuilder / Bootstrap Usage

The following uses of `EntityManagerFactoryBuilder`,
`EntityManagerFactoryBuilderImpl`, or `org.hibernate.jpa.boot.spi.Bootstrap`
remain and need separate treatment.

[cols="2,3,3,2", options="header"]
|===
| Area
| Current use
| Why it remains
| Likely next action

| `org.hibernate.jpa.boot.spi.Bootstrap`
| Public static facade returning `EntityManagerFactoryBuilderImpl` for
  `PersistenceUnitDescriptor`, `PersistenceUnitInfo`, persistence.xml `URL`,
  and supplied class-loading variants.
| Public SPI compatibility.  Integrations and tests may still call these
  overloads directly.
| Deprecate once `SessionFactoryBootstrap` and any replacement customization
  hooks cover the documented use cases.

| `HibernatePersistenceProvider`

  `Bootstrap`.

  by downstream integrations even though the provider's public methods no
  longer call them for normal EMF creation.
| Audit known overrides, then either route through the new facade where possible
  or deprecate with a replacement.

| `hibernate-testing` JPA helpers
| `EntityManagerFactoryBasedFunctionalTest` and `EntityManagerFactoryExtension`
  still create EMFs through `Bootstrap#getEntityManagerFactoryBuilder(...).build()`.
| Test infrastructure predates the new facade and still exercises JPA-style
  descriptor bootstrap, integration settings, and extension-managed scopes.
| Convert to `SessionFactoryBootstrap.build(PersistenceUnitDescriptor, Map)`
  unless a test genuinely needs builder-specific behavior.

| Tests asserting builder metadata or implementation details
| Tests such as configuration-object/naming-strategy/cache-mode/bootstrap

  metadata-builder contributor behavior, warning behavior, or pre-build metadata

  implementation class.

=== Two-Phase Bootstrap Status

The remaining two-phase shape is the legacy JPA builder SPI:

[source,java]
----
EntityManagerFactoryBuilder builder = Bootstrap.getEntityManagerFactoryBuilder(...);
MetadataImplementor metadata = builder.metadata();
EntityManagerFactory emf = builder.build();
----

That pattern remains only because some compatibility tests and SPIs still expose
or assert pre-build metadata.  The plan is to replace those cases with narrower
product builders or assertions:

* use `MetadataResolver` when the test needs finalized metadata;
* use `BootstrapSettingsResolver` or `SessionFactorySettingsResolver` when the
  test needs setting interpretation;
* use source-contribution helpers when the test needs mapping/source discovery;
* keep explicit legacy-SPI tests for `Bootstrap` and
  `EntityManagerFactoryBuilder` until those APIs are formally deprecated or
  removed.

=== Other SessionFactory Build Paths Still In Use

The new JPA orchestration path is not the only active way to build a
`SessionFactory`.

`Configuration`::
`org.hibernate.cfg.Configuration#buildSessionFactory(...)` remains a native
Hibernate API.  It collects properties, annotated classes, XML mappings,
auxiliary database objects, type contributors, function contributors,
attribute converters, naming strategies, interceptors, statement inspectors,
tenant resolvers, observers, and other convenience registrations.  It then
builds `Metadata`, obtains `Metadata#getSessionFactoryBuilder()`, applies
factory-level customizations, and calls `SessionFactoryBuilder#build()`.
+
This path should eventually converge on the same resolved settings/source/model
products, but it must preserve native-specific convenience methods and builder
customization behavior.

`org.hibernate.boot.Metadata#buildSessionFactory()`::
This is still a public convenience method.  In `MetadataImpl` it delegates to
`getSessionFactoryBuilder().build()`.  `InFlightMetadataCollectorImpl`
deliberately rejects this path, which is why the PoC finalizes metadata before
factory construction.

`org.hibernate.boot.SessionFactoryBuilder`::
This public builder remains the central native customization point after
metadata is built.  `MetadataImpl#getSessionFactoryBuilder()` creates the
default builder through `SessionFactoryBuilderService` and still honors
`SessionFactoryBuilderFactory` service-provider replacement.
+
The new `org.hibernate.boot.orchestration.SessionFactoryBuilder` is an internal
pipeline component with the same conceptual target but not the same API role.
The public `org.hibernate.boot.SessionFactoryBuilder` cannot be folded into it
without a compatibility plan for public builder methods and builder-factory
replacement.

Hibernate test infrastructure::
The JUnit 5 `SessionFactoryExtension`, JUnit 4 core test bases, and many native
tests still use `Metadata`, `Configuration`, or public
`SessionFactoryBuilder`.  Those paths are useful coverage for native bootstrap
features and should be migrated selectively, not mechanically.

=== Recent Decisions Applied To Tests

`PostgreSQLFunctionProcedureTest` is the current concrete example of how tests
should move away from two-phase bootstrap pressure.  The test previously used
builder metadata access to add PostgreSQL functions as auxiliary database
objects before building the factory.  That was the wrong pressure on
`SessionFactoryBootstrap`: the functions are mapping/source contributions, not
a reason to preserve a general prepared factory bootstrap.

The test now contributes the functions through Hibernate `mapping.xsd` XML using
`entity-mappings/database-object`.  The resource uses the current mapping XML
format, not legacy `hbm.xml`, because this branch is intentionally not carrying
legacy HBM mappings forward in the boot-model path.

This pattern should guide similar tests:

* if the thing being asserted is mapping metadata, express it as a mapping
  contribution or build metadata directly;
* if the thing being asserted is settings interpretation, test the settings
  resolver;
* if the thing being asserted is runtime factory behavior, use one-shot
  `SessionFactoryBootstrap.build(...)`;
* avoid adding public or semi-public bootstrap phases just to let a test mutate
  metadata between discovery and factory construction.

== Current Constructor Responsibilities

The constructor still orchestrates a large amount of factory assembly:

* builds the `SessionFactoryServiceRegistry`
* creates the `EventEngine`
* initializes the boot metamodel with the factory
* resolves factory name, JNDI name, UUID, and masked settings
* creates `SqlStringGenerationContext`
* registers integrators and applies integration callbacks
* orders columns and validates the boot metamodel
* primes second-level cache regions
* creates `RuntimeMetamodelsImpl`
* creates `QueryEngineImpl` before runtime metamodel initialization
* creates `SqlTranslationEngine`
* creates and initializes `MappingMetamodelImpl`
* applies fetch profiles after mapping metamodel creation
* creates default session and stateless-session builders
* resolves current-session context
* extracts frequently used services
* scopes `TypeConfiguration` to the completed factory

That sequence is delicate.  The PoC should avoid changing all of it at once, but
it should use named bootstrap products so the handoff into this constructor is
clear.

== Proposed Flow

[source]
----
Bootstrap inputs
    -> ResolvedBootstrapSettings
    -> AvailableResources
    -> CategorizedDomainModel
    -> MetadataImplementor
    -> ResolvedSessionFactorySettings
    -> SessionFactoryBuildContext
    -> SessionFactoryImpl
----

The immediate goal is not to replace `SessionFactoryImpl`.  The immediate goal
is to produce its three required inputs through the new pipeline.

== Phase Products

=== ResolvedBootstrapSettings

Already prototyped.

Owns early boot settings that affect source collection and mapping processing.
It should not be a dumping ground for all session-factory behavior.

=== AvailableResources

Already prototyped.

Owns normalized source inventory:

* managed class details
* package-info class details
* already-bound XML mapping documents

It should not own XML interpretation, categorization, binding, or settings
resolution.

=== CategorizedDomainModel

Already prototyped.

Owns interpreted domain metadata before binding to ORM's boot mapping model.
This is currently where XML source interpretation lives because there is no
standalone prepared-source artifact.

=== MetadataImplementor

This is the first major bridge into existing ORM runtime bootstrap.

The PoC binder should produce a normal ORM boot mapping model that can satisfy
`SessionFactoryImpl` expectations:

* persistent classes
* collections
* database / namespace / table state
* type registrations and converter registrations
* filters, fetch profiles, named queries, imports, and global registrations
* cache region configuration

This is the main proving ground for the binding work.

=== ResolvedSessionFactorySettings

This is now prototyped as
link:../../hibernate-core/src/main/java/org/hibernate/boot/settings/ResolvedSessionFactorySettings.java[`ResolvedSessionFactorySettings`]
and
link:../../hibernate-core/src/main/java/org/hibernate/boot/settings/SessionFactorySettingsResolver.java[`SessionFactorySettingsResolver`].

`SessionFactoryOptions` is very large and mixes resolved values, service
references, JPA/native defaults, and raw setting interpretation.  A resolver
similar to `BootstrapSettingsResolver` makes the factory-build handoff more
explicit without immediately copying ORM's full options surface.

Potential responsibilities:

* apply JPA-vs-native defaults
* resolve factory name, UUID, observer list, interceptor, statement observer,
  statement inspector, and session defaults
* resolve tenant, cache, JDBC, transaction, statistics, query, and compliance
  options
* preserve a raw settings map for components that still require one
* expose the `StandardServiceRegistry` used for factory construction

The current prototype carries configuration values, JPA bootstrap mode, and the
first constructor-facing values needed by `SessionFactoryImpl`: standard service
registry, UUID, factory name/JNDI-name flag, statement observer, interceptor,
factory observers, Bean Validation `ValidatorFactory` reference, multi-tenancy
flag, current-tenant resolver, cache settings, default tenant id Java type,
default catalog, and default schema.  It should continue to grow one setting at
a time as `SessionFactoryBuilder` needs them.

The PoC also has a temporary
link:../../hibernate-core/src/main/java/org/hibernate/boot/orchestration/internal/SessionFactoryOptionsAdapter.java[`SessionFactoryOptionsAdapter`].
It adapts `ResolvedSessionFactorySettings` to ORM's existing
`SessionFactoryOptions` interface, but only for audited methods.  Unsupported
methods throw immediately so new dependencies are documented before being added.
The design point is to move option resolution out of the final constructor path
without quietly copying the full legacy options builder.

=== SessionFactoryBuildContext

This is proposed, not yet prototyped.

`BootstrapContext` currently spans early source collection, model building, and
runtime metamodel creation.  That makes it easy for early bootstrap concepts to
leak into late factory construction.

A later build context could hold only the values needed after boot model binding:

* `BootstrapContext` or a narrowed replacement for runtime model creation
* `MetadataImplementor`
* resolved session-factory options
* service registry
* `TypeConfiguration`
* resolved settings map
* `SqlStringGenerationContext`

This does not need to replace ORM's `BootstrapContext` immediately.  It can
start as a local planning object used by the PoC factory bootstrap.

== Improvements Worth Prototyping

=== Resolve SessionFactoryOptions Explicitly

Build a resolver that produces an adapter implementing ORM's
`SessionFactoryOptions` from resolved settings and services.

This makes the PoC path testable without forcing all option interpretation to
remain hidden in existing builders.

=== Split Early BootstrapContext From Late Build Context

Keep `BootstrapContext` for early model-building concerns.  Introduce a late
`SessionFactoryBuildContext` only when preparing to instantiate the factory.

The aim is to make the handoff visible:

[source]
----
MetadataImplementor + ResolvedSessionFactorySettings + BootstrapContext
    -> SessionFactoryBuildContext
    -> SessionFactoryImpl
----

=== Make Runtime Model Creation a Named Phase

`SessionFactoryImpl` currently performs runtime metamodel creation inline.  The
sequence is constrained by query-engine and SQL-translation dependencies.

For now, leave that logic in `SessionFactoryImpl`, but document it as a future
phase:

[source]
----
validated MetadataImplementor
    -> RuntimeMetamodels
    -> QueryEngine / SqlTranslationEngine
    -> fetch profile application
----

=== Treat Integrators as an Explicit Phase

Integrators currently run inside `SessionFactoryImpl` before boot metamodel
ordering, validation, cache priming, and runtime metamodel creation.

That phase should eventually be visible:

[source]
----
bound boot model
    -> integrators applied
    -> boot model ordered and validated
    -> runtime model created
----

=== Avoid Requiring an Incomplete SessionFactory

`SessionFactoryImpl` builds the session-factory service registry by passing
`this` before the factory is fully initialized.

That may remain unavoidable in the short term, but it is a useful design smell.
An eventual factory-build handle could allow services that need factory identity
to avoid depending on a partially initialized `SessionFactoryImpl`.

=== BootstrapContext

Initial inspection target: `SessionFactoryImpl` and the first pass-through level
(`Integrator` and `RuntimeModelCreationContext` consumers).

Direct `SessionFactoryImpl` usage:

* `bootstrapContext.getTypeConfiguration()` is read at constructor start and
  stored as the factory `TypeConfiguration`.
* the full `BootstrapContext` is passed to `Integrator#integrate`.
* the full `BootstrapContext` is passed through
  `RuntimeModelCreationContext#getBootstrapContext()` during mapping/JPA
  runtime model creation.

This means the constructor itself does not appear to need the whole contract.
The broad need comes from extension and runtime-model pass-through.

[cols="1,2,2,2", options="header"]
|===
| Consumer area
| Observed access
| Likely narrower dependency
| Notes

| `SessionFactoryImpl`
| `getTypeConfiguration()`
| `TypeConfiguration`
| Direct constructor need.  This can be made explicit in a later
  `SessionFactoryBuildContext`.

| Integrators
| `Integrator#integrate(Metadata, BootstrapContext, SessionFactoryImplementor)`
| Compatibility adapter for existing `Integrator` contract
| This is an extension compatibility concern.  Existing integrators may use any
  `BootstrapContext` method, so narrowing probably requires a staged SPI change
  or adapter.

| Runtime model creation
| `RuntimeModelCreationContext#getBootstrapContext()`
| Runtime model creation context with explicit values
| Known downstream uses include class-loading access, bean registry/access,
  representation strategy selection, metadata building options, service registry,

| Runtime JPA metamodel

Preliminary conclusion:

`BootstrapContext` should not be treated as a single required constructor
dependency.  For the PoC, keeping it as an adapter may be acceptable for
integrator compatibility and current runtime model creation, but the design
target should be to split it into:

* explicit direct constructor inputs such as `TypeConfiguration`;
* a narrower runtime model creation context;
* an integrator compatibility adapter until the extension SPI can be revisited;
* early boot-only services that should not reach `SessionFactoryImpl`.

==== Usage Buckets

The constructor-path audit classifies observed `BootstrapContext` uses into
three buckets:

Early boot only::
Should not reach `SessionFactoryImpl`.
+
Likely examples include source discovery, XML processing, metadata binding
helpers, `ModelsContext`, temporary classloader access, and scanning artifacts.

Runtime model creation::
Should move onto, or remain available through, a narrower
`RuntimeModelCreationContext`.
+
Likely examples include `ClassLoaderService`, `ManagedBeanRegistry`,
representation strategy selection, and any mapping/runtime options that are
still proven necessary.

Integrator compatibility::
May require a legacy `BootstrapContext` adapter in the short term.
+
`Integrator#integrate(Metadata, BootstrapContext, SessionFactoryImplementor)` is
the strongest compatibility reason to preserve a `BootstrapContext` view until a
new extension SPI exists.

Future upstream integration work should assign any newly discovered
`BootstrapContext` use to one of these buckets before introducing or widening a
replacement contract.

==== PoC Decision

The PoC should not treat `BootstrapContext` as a primary bootstrap product.
Where existing ORM integration points still require the legacy contract, the PoC
may create or carry an adapter.  New PoC phases should request explicit values
instead of accepting `BootstrapContext`.

==== Next Slice Guardrails

The next SessionFactory bootstrap slice may introduce a temporary
`BootstrapContext` adapter in order to call ORM's current `SessionFactoryImpl`.
That adapter is a compatibility device, not the target architecture.

Rules for that slice:

* support a `BootstrapContext` method only after identifying the concrete
  consumer that asked for it;
* keep unsupported methods failing loudly so new dependencies become audit
  items instead of accidental design;
* prefer explicit values or a narrower `SessionFactoryBuildContext` for new PoC
  code;
* preserve a compatibility view for existing ORM extension points such as
  `Integrator#integrate(Metadata, BootstrapContext, SessionFactoryImplementor)`;
* document every supported adapter method in this audit section.

The desired upstream shape is still expected to be narrower than
`BootstrapContext`: likely explicit factory-build inputs plus a runtime model
creation context, with a legacy adapter only where extension compatibility
requires it.

==== Existing Context Reuse

The first attempt did not need a new `BootstrapContext` adapter.  The
`MetadataImplementor` produced by the PoC is currently still an
`InFlightMetadataCollector`, and that collector exposes the ORM
`BootstrapContext` that was used during metadata binding.  Reusing that existing
context is a better compatibility step than inventing a second context view.

The next boundary was therefore not `BootstrapContext` construction.  It was the
handoff from in-flight metadata to finalized metadata:

* `SessionFactoryBuilder` can create the minimal `SessionFactoryOptions` adapter.
* `SessionFactoryBuilder` can locate the existing `BootstrapContext` from the
  in-flight metadata collector.
* `SessionFactoryImpl` should not be called with the in-flight collector because
  `InFlightMetadataCollectorImpl#initSessionFactory` deliberately rejects that
  path.
* `MetadataResolver#finalizeMetadata` now builds the finalized `MetadataImpl`
  using `InFlightMetadataCollectorImpl#buildMetadataInstance`.
* Finalized metadata runs ORM second-pass/exportable processing and can change
  finalized property/column ordering compared to the earlier in-flight collector
  view.

With finalized metadata in place, the current SessionFactory construction
boundary moved into runtime service creation.  The first newly observed
unsupported options method was `SessionFactoryOptions#isQueryCacheEnabled`,
asked for while creating cache services.  The cache-related options bucket has
now been added to `ResolvedSessionFactorySettings` and
`SessionFactoryOptionsAdapter`.

The next boundary after cache was `SessionFactoryOptions#getInterceptor`, which
belongs to the session defaults / integration hooks bucket rather than cache.
That method is now resolved from `SessionEventSettings.INTERCEPTOR`.

With the interceptor hack in place, `SessionFactoryImpl` reached metadata
validation and exposed a duplicate identifier-column binding gap.  The PoC fixed
that for basic identifiers by keeping the basic id as the identifier property
only, rather than also adding it to the normal declared-property list.

The query-engine and runtime-default options needed by the simple
SessionFactory smoke test are now resolved.  The old
`SessionFactoryOptions#getCustomSqlFunctionMap` boundary has been crossed.

The binding test helper now finalizes the in-flight collector with
`InFlightMetadataCollectorImpl#buildMetadataInstance`, then runs
`MetadataImplementor#orderColumns(false)` and `MetadataImplementor#validate()`
before invoking test assertions.  This makes the binder suite exercise the same
mapping validation pressure that exposed the duplicate identifier-column bug.
Tests for known unsupported validated shapes, such as composite-key `@Any` and
association-id join-table mappings, assert the validation failure explicitly.

Follow-up questions:

* Can `RuntimeModelCreationContext` expose the needed values directly and
  deprecate `getBootstrapContext()`?
* Does runtime model creation still need `MetadataBuildingOptions`, or only a
  smaller set of resolved mapping/runtime options?
* Can `TypeConfiguration` be produced/scoped without retaining an early
  `BootstrapContext` reference?
* Should a replacement integrator SPI receive a factory-integration context
  instead of `BootstrapContext` and a partially initialized
  `SessionFactoryImplementor`?


==== RuntimeModelCreationContext Usages

Runtime model creation is the phase where finalized boot metadata becomes the
runtime mapping/JPA metamodel used by the `SessionFactory`.  In current ORM this
phase is driven from `SessionFactoryImpl` by creating a
`MappingMetamodelImpl`, then calling `MappingMetamodelImpl#finishInitialization`
with an inner `ModelCreationContext`.

The existing `RuntimeModelCreationContext` is a useful phase boundary, but it is
also very broad.  It exposes:

* the partially initialized `SessionFactoryImplementor`;
* the legacy `BootstrapContext`;
* the finalized `MetadataImplementor` boot model;
* the `MappingMetamodelImplementor` being initialized;
* `TypeConfiguration` and the derived `JavaTypeRegistry`;
* graph planning options;
* function registry, raw settings, dialect, cache, JDBC services, SQL string
  generation context, service registry, session-factory options, and generator
  state.

`SessionFactoryImpl.ModelCreationContext#getSessionFactory()` explicitly notes
that returning the enclosing factory is "bad" because the factory is not fully
initialized yet.  That makes this phase an important target for a narrower build
handle or runtime-model context that can supply the few factory-like services
needed before the factory is complete.

[cols="1,2,2,2", options="header"]
|===
| Consumer area
| Observed access
| Likely narrower dependency
| Notes

| `MappingMetamodelImpl#finishInitialization`
| boot entity/collection/component/mapped-superclass bindings, cache, graph
  planning options, SQL string generation context, raw settings, named entity
  graphs

Conclusion:

`RuntimeModelCreationContext` is worth keeping as a named phase concept, but the
current contract should be treated as a compatibility bridge.  The first
upstream pressure point is `getBootstrapContext()`: the observed runtime
consumers can be described with explicit dependencies such as `ModelsContext`,
class-loading access, managed-bean access, service access, representation
strategy selection, persistent-collection representation resolution, and resolved
mapping/runtime options.

The second pressure point is `getSessionFactory()`.  Runtime model creation
currently needs a factory-like handle for SQL/filter/helper construction, but the
actual `SessionFactoryImpl` is not yet complete.  A later design pass should
separate the pieces that genuinely require a completed factory from the pieces
that only need SQL generation, wrapper options, services, query/function access,
or mapping lookup.


==== BeanValidationIntegrator Usages

`BeanValidationIntegrator` does not use the `BootstrapContext` argument
directly.  It reads the `SessionFactoryServiceRegistry` from the partially
constructed `SessionFactoryImplementor`, resolves validation modes from
`ConfigurationService`, checks `ClassLoaderService` for the Bean Validation API,
and passes an internal `ActivationContext` to `TypeSafeActivator`.

The `ActivationContext` exposes:

* validation modes;
* `Metadata`;
* the partially constructed `SessionFactoryImplementor`;
* the `SessionFactoryServiceRegistry`.

`TypeSafeActivator` then uses that context to:

* apply DDL constraints from `Metadata#getEntityBindings()`;
* resolve settings, `JdbcServices`, and `ClassLoaderService` from the
  `SessionFactoryServiceRegistry`;
* register validation event listeners through
  `sessionFactory.getEventListenerRegistry()`;
* add the validation listener as a `SessionFactoryObserver`;
* call `SessionFactoryOptions#setCheckNullability(false)` when Bean Validation
  callback validation replaces core nullability checking.

This means Bean Validation is not a strong reason to keep
`BootstrapContext` in the integrator signature, but it is a strong example of
integrators needing a late build/integration context that can still mutate event
listeners, observers, and a few runtime settings before the factory is fully
usable.

==== CollectionCacheInvalidator Usages

`CollectionCacheInvalidator` also ignores the `BootstrapContext` argument.  It
checks `SessionFactoryOptions#isSecondLevelCacheEnabled()` and
`SessionFactoryOptions#isAutoEvictCollectionCache()`, then appends itself to the
factory event listener registry for post-insert, post-delete, and post-update
events when needed.

This is another signal that the current `Integrator` signature is broader than
the built-in integrators require.  The real dependency is a factory integration
context with options and event-listener registration access.

=== SessionFactoryOptions

Initial inspection target: `SessionFactoryImpl`,
`org.hibernate.boot.SessionFactoryBuilder`, `SessionFactoryOptionsBuilder`,
`SessionFactorySettings`, and the runtime session/JDBC contexts that read
options through `SessionFactoryImplementor`.

`SessionFactoryOptions` is different from the earlier constructor dependencies.
It is accepted by the `SessionFactoryImpl` constructor, but it is also stored and
returned by `SessionFactoryImplementor#getSessionFactoryOptions()`.  That means
the PoC cannot treat it purely as a temporary construction object unless the
runtime SPI is also changed upstream.

Direct `SessionFactoryImpl` constructor usage:

* stores the full options reference as the factory's runtime options view;
* resolves `StatementObserver`, `uuid`, `SessionFactoryObserver[]`, factory
  name/JNDI name, tenant id Java type, default catalog/schema, and
  multi-tenancy enablement;
* obtains the `StandardServiceRegistry` from the options to build the
  `SessionFactoryServiceRegistry`;
* passes the options to `QueryEngineImpl`;
* passes the options through `RuntimeModelCreationContext` during runtime model
  creation.

The constructor itself only dereferences a modest subset, but retaining the
object makes the full SPI available after construction.  Runtime consumers still
read many individual settings through the factory, including session defaults,
JDBC behavior, query/cache behavior, JPA compliance, dirty checking, tenant
resolution, JSON/XML format mappers, and graph parser behavior.

Preliminary conclusion:

The closest conceptual corollary to `SessionFactorySettingsResolver` is ORM's
`org.hibernate.boot.SessionFactoryBuilder`: it represents the factory-build
customization surface.  That distinction is not very helpful mechanically,
however, because ORM's builder currently delegates most of that state into
`SessionFactoryOptionsBuilder`.  For the audit, treat the two together:
`SessionFactoryBuilder` identifies externally supported customization points,
while `SessionFactoryOptionsBuilder` shows how those points are currently
resolved and accumulated.

`SessionFactoryOptionsBuilder` should still not become the design target.  It
mixes several concerns:

* normalized raw configuration values;
* service/dialect/JDBC-metadata-derived decisions;
* query engine options;
* cache options;
* session defaults;
* integration object references;
* the legacy runtime `SessionFactoryOptions` SPI view.

The practical first step is still likely an adapter implementing ORM's
`SessionFactoryOptions`, backed by `ResolvedSessionFactorySettings`.  The
important design point is that `ResolvedSessionFactorySettings` should grow by
consumer bucket, not by blindly mirroring every method in the legacy interface.

That first adapter slice now exists as
link:../../hibernate-core/src/main/java/org/hibernate/boot/orchestration/internal/SessionFactoryOptionsAdapter.java[`SessionFactoryOptionsAdapter`].
It supports only the methods used by the direct `SessionFactoryImpl`
constructor path identified above.  The PoC `SessionFactoryBuilder` creates the
adapter and can find the existing `BootstrapContext` from the in-flight metadata
collector.  It still stops before constructing the factory because the metadata
product must be finalized first.

==== PoC Decision

For the next factory-bootstrap slice:

* keep the PoC `SessionFactoryBuilder` returning `SessionFactoryImplementor`;
* have `SessionFactorySettingsResolver` produce `ResolvedSessionFactorySettings`
  from `ResolvedBootstrapSettings` and a `ServiceRegistry`;
* keep the raw immutable configuration map on `ResolvedSessionFactorySettings`;
* add named fields only for values required by the immediate
  `SessionFactoryImpl` construction path;
* compare the growing settings surface against ORM's
  `org.hibernate.boot.SessionFactoryBuilder` customization methods, while also
  checking the delegated `SessionFactoryOptionsBuilder` behavior;
* introduce a temporary `SessionFactoryOptions` adapter if needed to instantiate
  ORM's current `SessionFactoryImpl`;
* avoid copying query/cache/session-default resolution until a real consumer
  requires it.

==== Minimal Adapter Slice

Implemented in the PoC:

* `SessionFactorySettingsResolver#resolve(ResolvedBootstrapSettings,
  ServiceRegistry)` requires a `StandardServiceRegistry` and resolves the
  constructor-facing settings listed above.
* `ResolvedSessionFactorySettings` keeps those values as named immutable fields
  alongside the raw configuration map.
* `SessionFactoryOptionsAdapter` exposes the audited subset through ORM's
  `SessionFactoryOptions` SPI.
* `SessionFactoryBuilder` now creates that adapter, verifies that an existing
  `BootstrapContext` is reachable from the current metadata product, and can
  build a `SessionFactoryImplementor` for the current simple-entity smoke test.

==== Cache Options Slice

Implemented in the PoC:

* `SessionFactorySettingsResolver` resolves cache settings using the configured
  `RegionFactory`, `CacheSettings`, and `StrategySelector`.
* `ResolvedSessionFactorySettings` now carries second-level cache enablement,
  query cache enablement, query cache layout, timestamps cache factory, cache
  region prefix, minimal puts, structured entries, direct-reference entries, and
  collection-cache auto-eviction.
* `SessionFactoryOptionsAdapter` exposes those values through the corresponding
  `SessionFactoryOptions` cache methods.

This moved the SessionFactory construction boundary past cache service creation
to the session-defaults/integration-hooks and query-engine options buckets,
which are now represented as named values in `ResolvedSessionFactorySettings`.
The subsequent duplicate identifier-column validation failure was fixed in the
basic-id binder.

==== Runtime Defaults and Query Options Slice

Implemented in the PoC:

* `ResolvedSessionFactorySettings` now carries the query-engine options needed
  by `QueryEngineImpl`, including custom function/translator hooks, JPA
  compliance, criteria value handling, immutable-entity update handling, JSON/XML
  function flags, portable integer division, and native JDBC parameter handling.
* `ResolvedSessionFactorySettings` also carries runtime default options needed
  while creating persisters, loaders, and default session builders, including
  statement inspector, session cache mode, connection handling, JDBC time zone,
  auto-close/flush-before-completion, identifier rollback, SQL comments, batch
  and subselect fetch defaults, maximum fetch depth, temporal table strategy,
  audit strategy, JPA callbacks, and collections-in-default-fetch-group.
* `MetadataResolver` now performs the dialect type-contribution step before
  binding so the bootstrap `TypeConfiguration` has DDL type descriptors needed by
  temporary table strategy creation.
* `SessionFactoryBuilder` re-scopes `TypeConfiguration` to the metadata building
  context before calling `SessionFactoryImpl`, preserving the boot-time access
  expected by dialect function registration.

The simple `SessionFactoryBuilderTests` smoke test now builds and closes a real
`SessionFactoryImplementor` and verifies that the runtime mapping metamodel
contains the simple entity descriptor.

The adapter is intentionally incomplete.  A failure from an unsupported
`SessionFactoryOptions` method is not just a missing implementation; it is a
prompt to classify that method into one of the buckets above before growing
`ResolvedSessionFactorySettings`.

Follow-up questions:

* Should `QueryEngineOptions` become a separate resolved slice before building
  `QueryEngineImpl`?
* Which runtime users truly need access through
  `SessionFactoryImplementor#getSessionFactoryOptions()` and which could use
  narrower factory methods?
* Should Bean Validation's mutation of `setCheckNullability` be replaced by a
  build-time validation decision rather than a mutable options setter?
* Can `SessionFactoryServiceRegistryFactory#buildServiceRegistry` and
  `SessionFactoryServiceInitiatorContext` accept a narrower build context
  instead of full `SessionFactoryOptions` plus a partially constructed factory?

==== Service Registry and Integrator Slice

Audited construction order:

. `SessionFactoryImpl` stores `sessionFactoryOptions` and obtains
  `TypeConfiguration` from `BootstrapContext`.
. It creates the `SessionFactoryServiceRegistry` by calling
  `options.getServiceRegistry().requireService(SessionFactoryServiceRegistryFactory.class)
  .buildServiceRegistry(this, options)`.  The `this` reference is not fully
  initialized yet.
. The factory service registry is available before `bootMetamodel.initSessionFactory(this)`.
. `EventEngine`, `SchemaManager`, class-loading service, JDBC-values mapping
  producer provider, and changeset coordinator are initialized before
  integrators run.
. Integrators run before boot metadata column ordering, metadata validation,
  second-level cache region priming, runtime metamodel creation, query-engine
  creation, SQL-translation engine creation, default session builders, and final
  `TypeConfiguration#scope(SessionFactory)`.
. On successful construction, `IntegratorObserver` calls
  `Integrator#disintegrate` during `sessionFactoryClosed`.
. On construction failure, already-run integrators are disintegrated immediately,
  and disintegration failures are suppressed onto the startup exception.

`SessionFactoryServiceRegistryFactoryImpl` currently:

* obtains `ClassLoaderService` from `options.getServiceRegistry()`;
* creates `SessionFactoryServiceRegistryBuilderImpl` with the standard service
  registry as parent;
* loads `SessionFactoryServiceContributor` implementations via Java
  `ServiceLoader`;
* lets contributors add factory-scoped initiators or provided services;
* builds `SessionFactoryServiceRegistryImpl` with the partially initialized
  factory and the full `SessionFactoryOptions`.

`SessionFactoryServiceRegistryBuilderImpl` starts with the standard
session-factory service initiators:

* `StatisticsInitiator`;
* `CacheInitiator`;
* `NativeQueryInterpreterInitiator`;
* `ExtensionIntegrationServiceInitiator`.

Parent service bindings take precedence over these standard initiators.  The
factory-scoped registry also configures `Configurable` services with
`ConfigurationService#getSettings()`.

The standard factory-scoped initiators currently need:

[cols="1,2,2", options="header"]
|===
| Initiator
| Observed needs
| Narrower dependency

| `StatisticsInitiator`
| `ConfigurationService`, `ClassLoaderService`, `SessionFactoryImplementor`,
  `SessionFactoryOptions#isStatisticsEnabled()`
| statistics settings, class-loading access, factory identity/build handle

| `CacheInitiator`
| `RegionFactory`, `SessionFactoryImplementor`
| cache services plus factory identity/build handle

| `NativeQueryInterpreterInitiator`
| `SessionFactoryOptions#getNativeJdbcParametersIgnored()`
| native-query option

| `ExtensionIntegrationServiceInitiator`
| `SessionFactoryImplementor#getClassLoaderService()`
| class-loading access
|===

Preliminary conclusion:

`SessionFactoryServiceRegistry` itself can remain a compatibility boundary for
now, but upstream should consider splitting service-registry creation into a
named late construction phase with a narrower context:

* parent service registry;
* class-loading access;
* resolved factory/query/cache/statistics settings;
* factory identity or a factory-build handle;
* integration/event-listener registration access where needed.

The full `SessionFactoryOptions` SPI is mainly required because the existing
`SessionFactoryServiceInitiatorContext` exposes it wholesale.  The full
`SessionFactoryImplementor` is mainly required because the registry is built
inside the constructor with `this`.  Neither shape is clearly the desired
long-term contract.

| Investigate a narrower service-registry build input that does not require the

| `BindingState` and binder-local context hide some usage, while remaining
  constructor requirements continue to use the legacy context.
| Isolate mapping-object construction behind local factories or adjust upstream
  constructors to accept narrower dependencies.

| Schema export for the runtime smoke entity
| Calling `SchemaManager#exportMappedObjects(true)` for the minimal runtime
  smoke entity generated an empty table column list, even though the runtime
  persister could persist/query the manually created table.
| The smoke test creates and drops the tiny table manually through JDBC so it
  can focus on SessionFactory runtime behavior.
| Track this as a binding/schema-export gap.  Runtime smoke tests should
  eventually use Hibernate schema tooling once the exported table columns are
  correct.

| Fetch profiles

| Fetch profiles are converted after runtime persisters exist, so this is a late

| Schema tooling

| database/schema model, contributors, entity/collection/table/sequence state

|===

`MetadataImpl` itself contains the following distinct groups of state:

* metadata-building options and `BootstrapContext`-derived type/function
  registries;
* entity, collection, component, mapped-superclass, type, and identifier
  mappings;
* filter, fetch-profile, import, named query/native query/procedure/result-set,
  and named entity-graph definitions;
* SQL function definitions;
* the database/namespace/table/sequence model;
* helper maps used by copying and generic/embeddable discriminator resolution.

Preliminary conclusion:

`MetadataImplementor` should remain the PoC's factory-construction
compatibility target.  It is the current handoff accepted by ORM's runtime
metamodel, query engine, schema tooling, cache priming, and integrator SPI.

However, the audit makes it clear that the target design should not treat one
large `MetadataImplementor` as the only long-term product.  `ResolvedMetadata`
should be allowed to expose named sub-products as they become useful:

* finalized boot mapping model;
* database/schema model;
* named object definitions;
* filter and fetch-profile definitions;
* cache-region descriptors;
* type/function registries;
* metadata finalization/validation result.

The likely upstream path is incremental:

* keep producing ORM `MetadataImplementor` until runtime metamodel creation and
  schema tooling can consume narrower products;
* move `orderColumns(false)` and `validate()` out of `SessionFactoryImpl` and
  into metadata resolution/finalization;
* keep `PersistentClass`, `Collection`, and `Component` as the runtime mapping
  model inputs until upstream mapping creation APIs are changed;
* introduce narrower consumers first around query named objects, cache-region
  descriptors, fetch profiles, and schema tooling because those are easier to
  isolate than persister construction.

Follow-up questions:

* Can `MetadataResolver#finalizeMetadata` own `orderColumns(false)` and
  `validate()` permanently, leaving `SessionFactoryImpl` to consume already
  finalized metadata?
* Should `ResolvedMetadata` expose named-object definitions separately from the
  ORM `MetadataImplementor` bridge?
* Should cache-region priming consume cache descriptors produced during binding
  instead of re-reading `PersistentClass` / `Collection` cache settings?
* Can schema tooling consume a narrower schema/database model, or is
  `MetadataImplementor` still the right boundary there until upstream changes?
* Which third-party integrator use cases require the full metadata view?

==== InFlightMetadataCollector versus BindingState

`BindingState` should not be viewed as a local clone of
`InFlightMetadataCollector`.  The clean design separates three jobs:

* `BindingState` is the PoC's typed binding-phase state, used to coordinate
  binders and remember local decisions.
* `MetadataCollector` is the structured write target for completed metadata
  contributions.
* `InFlightMetadataCollectorAdapter` is the current compatibility implementation behind
  `MetadataCollector`, not a long-term peer of `BindingState`.

The clean target is therefore not immediate collector removal.  The clean target
is to make every collector touchpoint fall into one of a few explicit buckets.

|===
| Responsibility
| Best current owner
| Why
| Possible upstream direction

| Creating/finalizing ORM metadata
| `InFlightMetadataCollector`

| Create a first-class metadata collector/finalizer contract.  Until then, keep the

| Prefer named typed phases in new boot-model binding.  Keep collector second
  passes only as an adapter for legacy ORM binders.

| Explicit services via `BindingState`, `BindingOptions`, or binding context

Practical rule for the PoC:

* if the state is needed to coordinate PoC binding phases, put it on
  `BindingState`;
* if the write contributes completed metadata, send it through `MetadataCollector`;
* if ORM needs the side effect during the same bind run, let the current
  `MetadataCollector` adapter write through immediately and document why;
* if ORM owns object creation or second-pass/query machinery, avoid local
  rewrites unless we are ready to change the upstream API.

The likely eventual shape is a pair of first-class contracts plus a compatibility
adapter:

Binding phase state::
Typed, phase-local state like today's `BindingState`.  It coordinates binders
and exposes explicit services/options, but does not pretend to be
`MetadataImplementor`.

Metadata collector::
A narrow target for finalized ORM boot metadata contributions.  Initially this is an
adapter over `InFlightMetadataCollector`; longer term it can replace the broad
collector exposure with named operations for entity, collection, table, type,
query, filter, fetch-profile, and cache metadata.

Collector compatibility adapter::
`InFlightMetadataCollectorAdapter` is the current adapter from `MetadataCollector` to
`InFlightMetadataCollector`.  It exists to keep using ORM's metadata finalizer
while the PoC proves the cleaner boundary.

=== RuntimeModelCreationContext

Initial audit done against ORM's `RuntimeModelCreationContext`,
`SessionFactoryImpl.ModelCreationContext`, `MappingMetamodelImpl`, entity and
collection persisters, JPA metamodel creation, representation strategies, and
mapping-model creation helpers.

This contract is both a good concept and currently too broad.  It represents a
real construction phase, but the values it exposes fall into several different
ownership buckets:

* finalized boot metadata: `MetadataImplementor`, entity/collection bindings,
  components, mapped superclasses, named entity graphs, metadata-building
  options still reachable from metadata;
* runtime model under construction: `MappingMetamodelImplementor`, persisters,
  JPA metamodel, constraint model, loaders, embeddable mappings;
* factory-like runtime services: dialect, JDBC services, SQL string generation,
  function registry, cache, graph planning options, type configuration;
* broad compatibility views: `SessionFactoryOptions`, raw settings,
  `BootstrapContext`, service registry, and the partially initialized
  `SessionFactoryImplementor`;
* generator state: generator cache plus `GeneratorSettings`.

The most important observation is that current ORM explicitly passes a
partially initialized `SessionFactoryImpl` into this phase.  The inner
`ModelCreationContext#getSessionFactory()` comment calls out that this is bad
because the factory is not fully initialized.  Several runtime model helpers use
that factory handle for SQL generation, wrapper options, filter helpers, mapping
lookups, or service access, but those needs are narrower than the completed
factory contract.

== Current PoC Slice

The current metadata collector resolver is:

* link:../../hibernate-core/src/main/java/org/hibernate/boot/settings/BootstrapSettingsResolver.java[`BootstrapSettingsResolver`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/orchestration/MetadataResolver.java[`MetadataResolver`]
* link:../../hibernate-core/src/main/java/org/hibernate/boot/orchestration/ResolvedMetadata.java[`ResolvedMetadata`]

Settings resolution is a separate target:

.BootstrapSettingsResolver
[source]
----
configuration values / JPA inputs
    -> BootstrapSettingsResolver
    -> ResolvedBootstrapSettings
----

`MetadataResolver#resolve` owns only the metadata collector phase order:

.MetadataResolver
[source]
----
ResolvedBootstrapSettings + BootstrapSourceContributions + ServiceRegistry
    -> MetadataBuildingContext
    -> AvailableResources
    -> CategorizedDomainModel (*)
    -> BindingState (*)
    -> MetadataCollector
    -> MetadataImplementor (*)
    -> ResolvedMetadata
----

See
link:session-factory-bootstrap-design.adoc#_flow_notation[`session-factory-bootstrap-design.adoc`]
for the notation used by these pipeline sketches.  In short, the settings,
source contributions, and service registry are peer inputs to the resolver; the
arrowed lines are the ordered products/steps owned by that resolver.  A `(*)`
marker means the product is retained by the final product.  Inputs and unmarked
intermediate products do not automatically become part of the final product.

`ResolvedMetadata` carries the finalized `MetadataImplementor`, the
`CategorizedDomainModel`, and the `BindingState` used as local binding-phase
state.  It intentionally does not carry `ResolvedBootstrapSettings`; settings
remain input to metadata resolution and to later settings resolvers, not part of
the metadata result.

At the moment `MetadataResolver#finalizeMetadata` builds ORM's finalized
`MetadataImpl` through `InFlightMetadataCollectorImpl#buildMetadataInstance`.
Column ordering and validation are exercised by binding tests, but they are not
yet permanently owned by `MetadataResolver`; that remains one of the follow-up
questions above.

Later slices should add one resolver per gross target rather than growing a
single SessionFactory-wide orchestrator.

`SessionFactoryBuilder` is the next target shell.  It accepts
`ResolvedSessionFactorySettings`, `ResolvedMetadata`, and a `ServiceRegistry`,
and its intended return type is the runtime `SessionFactoryImplementor` itself.
There is no separate build-result object unless the factory construction path
proves it needs additional materialized products.

== Concrete Forward Plan

The constructor audit is complete enough to move from discovery to incremental
prototype slices.  The next work should keep the current compatibility path
running while externalizing one constructor responsibility at a time.

This plan is not limited to local PoC adapters.  Because the target is upstream
ORM integration, each slice should explicitly decide whether the better move is:

* a local compatibility adapter to keep the PoC moving;
* an upstream ORM signature/context change that directly supports the design;
* both, with the adapter serving only as the migration bridge.

Adapters should therefore be treated as pressure-point markers.  When an
upstream change is small, clarifies ownership, and avoids preserving a broad
legacy contract, prefer making that upstream change over layering another local
bridge.

=== 1. Remove Legacy hbm.xml Support Upstream

Goal: simplify the upstream boot surface before carrying the PoC design deeper
into ORM.

Legacy `hbm.xml` support is a major source of bootstrap coupling.  It pushes ORM
toward source-neutral XML preprocessing, broad mapping defaults, legacy mapping
constructors, second-pass style coordination, and equivalence paths that are not
part of the ORM 9 target.  Removing it should make several current dependencies
disappear or become easier to narrow.

Tasks:

* remove `hbm.xml` as an accepted mapping source in upstream ORM 9 bootstrap;
* keep JPA `orm.xml` support as the remaining XML mapping source;
* remove or quarantine settings that exist only for `hbm.xml` compatibility,
  such as explicit legacy HBM resource lists and HBM-specific XML assumptions;
* revisit XML preprocessing and processing after the removal, because some
  currently necessary abstractions may only exist to preserve HBM behavior;
* document any temporary migration guardrails or failure messages needed for
  users attempting to configure HBM mappings.

Success criteria:

* source discovery no longer needs an HBM source bucket;
* `AvailableResources` XML handling clearly means JPA `orm.xml`;
* categorization and binding no longer preserve behavior solely for HBM parity;
* follow-up bootstrap slices can treat removed HBM dependencies as gone rather
  than adapting around them.

=== 2. Stabilize the Current Handoff

Goal: make the current metadata-to-factory path intentional and well tested.

Tasks:

* keep `MetadataResolver` producing `ResolvedMetadata`;
* keep `ResolvedMetadata` limited to the products it actually retains:
  `MetadataImplementor`, `CategorizedDomainModel`, and `BindingState`;
* keep `ResolvedBootstrapSettings` and `ResolvedSessionFactorySettings` as
  separate inputs to later phases;
* keep `MetadataCollector` as the explicit completed-metadata write target;
* add smoke tests which prove the same `ResolvedMetadata` can support both
  metadata inspection and `SessionFactoryBuilder`.

Success criteria:

* a simple model, a model with composite identifiers/values, and at least one
  cache/named-query-oriented model can build metadata and bootstrap a usable
  `SessionFactory`;
* column ordering and metadata validation always run through
  `MetadataResolver`;
* tests do not reach through `ResolvedMetadata` to recover bootstrap settings.

=== 3. Make SessionFactoryBuilder Own the Compatibility Adapter Boundary

Goal: keep `SessionFactoryBuilder` returning `SessionFactoryImplementor`, while
making the current adapter work explicit.

.SessionFactoryBuilder
[source]
----
ResolvedSessionFactorySettings
ResolvedMetadata
ServiceRegistry
    -> SessionFactoryOptions adapter
    -> existing BootstrapContext adapter/reuse
    -> SessionFactoryImplementor
----

Tasks:

* document every supported `SessionFactoryOptionsAdapter` method in the audit;
* add named fields to `ResolvedSessionFactorySettings` only when the builder or
  constructor path needs them;
* keep unsupported adapter methods failing loudly;
* for each unsupported method encountered, decide whether to add a named
  resolved setting, keep a documented compatibility default, or change the
  upstream consumer to request a narrower contract;
* keep reusing ORM's current `BootstrapContext` from the metadata product until
  a concrete replacement slice needs otherwise.

Success criteria:

* every newly supported option method maps to either
  `ResolvedSessionFactorySettings`, `ResolvedBootstrapSettings`, or a documented
  compatibility default;
* no broad `SessionFactoryOptionsBuilder` dependency returns as the design
  target;
* the builder remains a small runtime factory target, not a second metadata
  resolver.

=== 4. Externalize Factory-Scoped Service Registry Creation

Goal: identify whether `SessionFactoryServiceRegistry` can be built from a
narrow factory-build context instead of a partially initialized
`SessionFactoryImpl`.

.Factory service registry slice
[source]
----
ResolvedSessionFactorySettings
ResolvedMetadata
ServiceRegistry
Factory build handle
    -> SessionFactoryServiceRegistry
----

Tasks:

* prototype a narrow factory-build handle with factory identity, parent service
  access, class-loading access, resolved settings, and metadata access;
* compare that handle to what `SessionFactoryServiceRegistryFactoryImpl` and its
  initiators actually consume;
* decide whether to adapt the existing service-registry implementation or change
  ORM's service-registry factory signature to accept the narrow build handle;
* keep any adapter local and temporary if the upstream signature change is not
  part of the current slice.

Success criteria:

* the service-registry creation inputs are named and testable outside the
  `SessionFactoryImpl` constructor;
* any need for the partially initialized factory is either removed or documented
  as a specific upstream pressure point.

=== 5. Make Integrator Execution an Explicit Compatibility Phase

Goal: keep legacy `Integrator` behavior working while making the callback shape
visible.

.Integrator compatibility slice
[source]
----
ResolvedMetadata
SessionFactoryServiceRegistry
SessionFactoryImplementor / factory build handle
BootstrapContext adapter
    -> legacy Integrator callbacks
----

Tasks:

* isolate creation/reuse of the `BootstrapContext` view passed to legacy
  integrators;
* document which built-in integrators use the context, options, service
  registry, or metadata;
* identify a narrower future integration callback shape;
* decide whether built-in ORM integrators can move to the narrower shape before
  the public `Integrator` SPI changes.

Success criteria:

* existing integrators still run;
* the legacy `BootstrapContext` argument is clearly treated as adapter state;
* built-in integrator needs are captured as candidates for a future SPI.

=== 6. Name Runtime Model Creation as a Builder Subphase

Goal: stop treating runtime model creation as incidental constructor work.

.Runtime model creation slice
[source]
----
ResolvedMetadata
ResolvedSessionFactorySettings
SessionFactoryServiceRegistry
Factory build handle
    -> RuntimeModelCreationContext
    -> runtime mapping/model products
----

Tasks:

* keep `RuntimeModelCreationContext` as the named phase boundary;
* replace observed `getBootstrapContext()` use with explicit services/settings
  where possible;
* avoid exposing a partially initialized `SessionFactoryImplementor` to new code;
* change upstream runtime-model consumers where reasonable so they request
  explicit services/settings instead of `BootstrapContext`;
* keep `MetadataImplementor` as the compatibility metadata input only until
  persister construction can consume narrower metadata products upstream.

Success criteria:

* runtime model creation can be described as inputs plus products, not just as
  constructor side effects;
* new code asks for explicit services/settings instead of the broad
  `BootstrapContext`;
* any remaining need for `MetadataImplementor` is documented as a persister or
  runtime-model dependency.

=== 7. Revisit ResolvedMetadata Shape After Factory Construction Works

Goal: replace transitional retained products with narrower products only after
their consumers are known.

Tasks:

* keep `BindingState` on `ResolvedMetadata` while it is useful for proving the
  handoff and auditing consumers;
* when a later phase reads `BindingState`, decide whether that value should
  become a named immutable `ResolvedMetadata` sub-product;
* avoid making `ResolvedMetadata` a general bag of bootstrap state.

Success criteria:

* `ResolvedMetadata` remains a metadata result, not a settings or entry-point
  carrier;
* `BindingState` either proves its long-term value or is replaced by narrower
  retained products.

=== Suggested Immediate Next Slice

Start with slice 1 in upstream ORM if practical: remove legacy `hbm.xml` support
before preserving abstractions that only exist for it.  In parallel or
immediately after that, tighten `SessionFactoryBuilder` around the compatibility
adapter boundary.  That keeps the PoC booting a real `SessionFactory` while
forcing each remaining constructor dependency to be named, adapted, documented,
or replaced by a targeted upstream ORM change.

== Open Questions

* Should `ResolvedSessionFactorySettings` later produce an adapter implementing
  ORM's `SessionFactoryOptions`, or should the adapter itself be the resolved
  object?
* Which settings belong in `ResolvedBootstrapSettings` versus
  `ResolvedSessionFactorySettings`?
* Can `BootstrapContext` be narrowed before integration, or should the PoC simply
  reuse ORM's current implementation until the factory path works?
* Should JPA `orm.xml` source interpretation remain in categorization internals,
  or should a prepared-source phase emerge if more source interpretation steps
  appear?
* Can service-registry creation be moved before `SessionFactoryImpl` without
  breaking services that expect a concrete factory reference?
