*Extracted from:* `AntlrBasedSQLFormatterTest` test fixtures +
*Initial Date:* 2026-05-08 +
*Last Updated:* 2026-07-03

'''

== 1. Casing Rules

=== 1.1 SQL Keywords → LOWERCASE

* Examples: `select`, `from`, `where`, `join`, `left join`, `inner join`, `on`, `and`, `or`, `as`, `with`, `group by`, `having`, `order by`, `offset`, `fetch`, `first`, `rows`, `exists`, `in`, `all`, `any`, `desc`, `asc`, `distinct`, `interval`, `lateral`, `union`, `except`, etc.

=== 1.2 SQL Functions → LOWERCASE

* Examples: `count`, `max`, `min`, `sum`, `avg`, `upper`, `lower`, `trim`, `round`, `nullif`, `coalesce`, `cast`, `substring`, etc.
* Includes function-specific keywords: `both`, `leading`, `trailing` (in trim)

=== 1.3 Boolean Literals → LOWERCASE

* `true`, `false`, `null`

=== 1.4 Identifiers (tables, columns, aliases) → PRESERVE ORIGINAL CASE

* Examples: `customers`, `customer_id`, `full_name`, `c`, `rca`, `o2`
* Do NOT change user-defined names

=== 1.5 Quoted Identifiers → PRESERVE EXACTLY AS-IS

* Double quotes: `"o".`customer_id``
* Backticks: ```column_name```
* Square brackets: `[oi].description`
* Preserve both the quotes AND the content case

'''

== 2. Indentation Rules

=== 2.1 Indentation Character

* *Use 4 SPACE characters, NOT tabs*
* Each indentation level = 4 spaces (defined as `INDENT = "    "` in implementation)

=== 2.2 Indentation is Relative to Containing Context

* Not absolute levels (0, 1, 2)
* Each nesting level adds +1 indent level (4 spaces) from its parent context
* No maximum depth (could be 27 levels in complex queries)

=== 2.3 Indentation Levels (relative)

*Current context level (base):*

* Major clauses: `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY`, `OFFSET`, `FETCH`
* `JOIN` clauses (all types)
* `WITH` clause and CTE names

*Current level + 1 tab:*

* SELECT list items (when on separate lines)
* ON clause conditions (when on separate lines)
* WHERE clause conditions (when on separate lines)
* GROUP BY items (when on separate lines)
* ORDER BY items (when on separate lines)
* CTE contents (inside `AS (...)`)
* Subquery contents
* Operator list items when >3 items (IN, ALL, ANY with value lists)
* EXISTS/IN/ALL/ANY subquery contents

*Nested contexts:*

* Each subquery creates new context with its own relative levels
* Example: subquery inside WHERE clause inside CTE inside main query could be at indent level 3+ depending on nesting

'''

== 3. Line Break Rules

=== 3.1 New Line BEFORE

* Each major query clause: `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY`, `OFFSET`, `FETCH`, `LIMIT`, `UNION`, `INTERSECT`, `EXCEPT`
* Each major DML clause: `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `VALUES`, `SET`, `RETURNING`
** Exception: `FROM` after `DELETE` stays inline (no newline)
* DML-specific clauses: `ON CONFLICT`, `ON DUPLICATE KEY`, `DO UPDATE`, `DO NOTHING`, `USING` (in MERGE/DELETE)
* MERGE conditionals: `WHEN MATCHED THEN`, `WHEN NOT MATCHED THEN`
* Each `JOIN` clause (`JOIN`, `LEFT JOIN`, `INNER JOIN`, `RIGHT JOIN`, `FULL JOIN`, `CROSS JOIN`, `LATERAL JOIN`)
* Each list item when list has >3 elements (see section 4)
* Each `AND`/`OR` in ON/WHERE clauses (after first condition)
* Closing `)` of CTE definitions

=== 3.2 New Line AFTER

* CTE name and `AS (`
* Opening `(` of subqueries (e.g., `EXISTS (SELECT ...`, `LATERAL (SELECT ...`)
* Opening `(` of operator lists when >3 items (e.g., `IN (` with >3 values)
* Major clause keywords when their content is on separate lines

=== 3.3 Same Line

* First condition stays on same line as `ON` keyword
* First condition stays on same line as `WHERE` keyword
* Lists with ≤3 elements stay on same line (see section 4)

'''

== 4. List Formatting Rules

=== 4.1 Lists with ≤3 Elements

* *CAN* stay on same line as their owning element/keyword
* Examples:
** `SELECT col1, col2, col3 FROM ...`
** `COUNT(*)`
** `ROUND(value, 2)`
** `GROUP BY a, b, c`
** `ORDER BY x, y`

=== 4.2 Lists with >3 Elements

* *MUST* put each element on separate line
* Each element indented +1 tab from current context level
* Examples:
** SELECT list with 11 items → each on new line, indented
** GROUP BY list with 9 items → each on new line, indented
** ORDER BY list with 5 items → each on new line, indented

=== 4.3 Applies To

* SELECT column/expression lists
* GROUP BY lists
* ORDER BY lists
* SET assignment lists (UPDATE/MERGE statements)
* Function argument lists (when >3 args)
* Operator value lists: `IN (...)`, `ALL (...)`, `ANY (...)` value lists

*Note:* Parenthesized lists (INSERT column lists, VALUES tuples) stay on single line regardless of item count - they are treated as atomic units.

'''

== 5. Comma Placement

=== 5.1 Trailing Commas (comma at END of line)

* Used in all multi-line lists
* Examples:
** SELECT list items (except last)
** GROUP BY items (except last)
** ORDER BY items (except last)
** Between CTE definitions
** Function arguments when multiline

=== 5.2 No Comma After Last Item

* Last item in any list has no trailing comma

'''

== 6. Spacing Rules

=== 6.1 Function Calls

* *NO* space between function name and opening `(`
* Examples: `COUNT(*)`, `MAX(value)`, `TRIM(BOTH ' ' FROM col)`

=== 6.2 Operators

* Space *BEFORE* and *AFTER*: `=`, `>`, `<`, `>=`, `<=`, `!=`, `<>`, `+`, `-`, `*`, `/`
* Examples:
** `o.status = 'COMPLETED'`
** `rca.total_spent / NULLIF(rca.order_count, 0)`
** `age > 18`

=== 6.3 Keywords

* Space *BEFORE* and *AFTER*: `AND`, `OR`, `AS`, `IN`, `NOT`, `BETWEEN`, `LIKE`

=== 6.4 Commas

* *NO* space before comma
* Space *AFTER* comma (when on same line)
* Example: `col1, col2, col3`

=== 6.5 Parentheses

* *NO* space after opening `(`
* *NO* space before closing `)`
* Examples: `(SELECT ...)`, `COUNT(*)`, `(col1 + col2)`

=== 6.6 Special Operators - Space BEFORE Opening Parenthesis

The following operators require a space before the opening `(`:

* `LATERAL (...)` - for lateral joins
* `EXISTS (...)` - for existence checks
* `IN (...)` - for membership tests
* `ALL (...)` - for quantified comparisons
* `ANY (...)` - for quantified comparisons
* `OVER (...)` - for window functions

Examples:

* `WHERE EXISTS (SELECT ...)` ✓ (not `WHERE EXISTS(SELECT ...)`)
* `WHERE col IN ('a', 'b')` ✓ (not `WHERE col IN('a', 'b')`)
* `WHERE value = ALL (1, 2, 3)` ✓
* `WHERE value > ANY (10, 20)` ✓
* `RANK() OVER (PARTITION BY ...)` ✓

*Note:* Regular functions do NOT have a space: `COUNT(*)`, `UPPER(name)`, etc.

'''

== 7. Special Constructs

=== 7.1 CTE (Common Table Expressions)

Structure:

[source,sql]
----
WITH cte_name AS (
	[indented content]
),
next_cte AS (
	[indented content]
)
[main query]
----

=== 7.2 Subqueries

* Indented relative to parent context
* Follow same formatting rules recursively
* Closing `)` is dedented to parent context level

*Example in WHERE with simple condition:*

[source,sql]
----
WHERE condition
	AND EXISTS (
		SELECT ...
		FROM ...
	)
----

*Example in WHERE with multiple conditions:*

[source,sql]
----
WHERE c.is_active = TRUE
	AND EXISTS (
		SELECT 1
		FROM customer_tags ct
		WHERE ct.customer_id = c.customer_id
			AND ct.tag_name = 'VIP'
	)
----

*Note:* When EXISTS contains a subquery with its own WHERE clause and AND conditions, those inner conditions get additional indentation. The closing `)` is placed at the same indentation level as the line that opened the subquery (the `AND EXISTS` line in this case).

=== 7.3 LATERAL Joins

Structure:

[source,sql]
----
LEFT JOIN LATERAL (
	[indented SELECT query]
) alias
	ON condition
----

*Note:* The alias appears on the same line as the closing `)`, then the `ON` clause starts on a new line with indentation.

=== 7.4 FETCH Clauses

Structure:

[source,sql]
----
OFFSET n ROWS
FETCH FIRST n ROWS
WITH TIES
----

* Each major clause (`OFFSET`, `FETCH`, `WITH`) on separate line at current context level
* The `WITH TIES` (or `WITH TIES ONLY`) qualifier appears on its own line after `FETCH FIRST n ROWS`

=== 7.5 CASE Expressions

*Standalone CASE:*

[source,sql]
----
CASE
	WHEN condition THEN value
	WHEN condition THEN value
	ELSE value
END
----

*CASE in SELECT list (with other items):*

[source,sql]
----
SELECT
	col1,
	col2,
	CASE
		WHEN condition THEN value
		ELSE value
	END AS alias,
	col3
----

*CASE in SELECT list (single item):*

[source,sql]
----
SELECT
	CASE
		WHEN condition THEN value
		ELSE value
	END
FROM table1
----

*CASE in assignment (UPDATE/SET):*

[source,sql]
----
UPDATE table1
SET column_name =
	CASE
		WHEN condition THEN value
		ELSE value
	END
----

* CASE keyword indented at +1 from context (same level as list items or assignment values)
* WHEN/ELSE clauses indented +1 from CASE
* END keyword indented at same level as CASE

=== 7.6 Operator Lists (IN, ALL, ANY, EXISTS)

The operators `IN`, `ALL`, `ANY`, and `EXISTS` all follow consistent formatting rules:

*Space before opening parenthesis:*

* Always include a space: `IN (`, `ALL (`, `ANY (`, `EXISTS (`

*With ≤3 items (single-line):*

[source,sql]
----
WHERE status IN ('ACTIVE', 'PENDING', 'APPROVED')
WHERE value = ALL (1, 2, 3)
WHERE price > ANY (10, 20)
----

*With >3 items (multi-line):*

[source,sql]
----
WHERE status IN (
	'ACTIVE',
	'PENDING',
	'APPROVED',
	'COMPLETED',
	'CANCELLED'
)
WHERE value = ALL (
	100,
	200,
	300,
	400,
	500
)
----

*With subquery:*

[source,sql]
----
WHERE EXISTS (
	SELECT 1
	FROM other_table
	WHERE condition
)
WHERE id IN (
	SELECT id
	FROM other_table
)
----

'''

== 8. Identifier Quoting

=== 8.1 Preserve All Quote Styles

* Double quotes: `"identifier"`
* Backticks: ```identifier```
* Square brackets: `[identifier]`

=== 8.2 Preserve in Qualified Names

* Examples: `"schema".table`, `table.`column``, `[database].table`

'''

== 9. String Literals

=== 9.1 Preserve Exactly As-Is

* Single quotes: `'value'`
* Content and case unchanged
* Escaped quotes unchanged

'''

== 10. Comments

=== 10.1 Preserve Comments

* Single-line: `-- comment`, terminated by the line break token (`'\n'`)
** Always placed on its own line (newline before and after)
** Inherits indentation from current context
* Multi-line: `/* comment */`
** Always placed on its own line (newline before and after)
** If the comment contains line breaks, the indentation of the first line is carried over to continuation lines
** Leading whitespace is stripped from continuation lines before applying base indentation
** Special case: `/*+ text content */` (hint comments) are preserved exactly as-is, including line breaks
* Comments in WHERE/ON/HAVING clauses between conditions are indented at the condition level (context indent + 1 tab)

'''

== 11. DML Statement Formatting

=== 11.1 INSERT Statements

*Basic INSERT with column list:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2, col3, col4)
VALUES
	(val1, val2, val3, val4);
----

* Column list in parentheses stays on single line (indented +1 from INSERT)
* Column list treated as atomic unit regardless of item count
* VALUES keyword on new line at INSERT context level
* Value tuple in parentheses stays on single line (indented +1 from VALUES)

*INSERT without column list:*

[source,sql]
----
INSERT INTO table_name
VALUES
	(val1, val2, val3, val4);
----

*Multi-row INSERT:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2, col3)
VALUES
	(val1a, val2a, val3a),
	(val1b, val2b, val3b),
	(val1c, val2c, val3c);
----

* Each value tuple on separate line with +1 indent
* Trailing commas on all rows except last

*INSERT...SELECT:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2, col3)
SELECT col1, col2, col3
FROM source_table
WHERE condition;
----

* SELECT formatted according to standard SELECT rules
* No VALUES keyword

=== 11.2 UPDATE Statements

*UPDATE with SET clause:*

[source,sql]
----
UPDATE table_name
SET column1 = value1
WHERE condition;
----

* Single assignment stays inline with SET keyword (space after SET)

*UPDATE with multiple assignments (>3):*

[source,sql]
----
UPDATE table_name
SET
	column1 = value1,
	column2 = value2,
	column3 = value3,
	column4 = value4
WHERE condition;
----

* SET clause follows SELECT list rules: ≤3 assignments inline, >3 multiline
* Multiline: each assignment on separate line at +1 indent from UPDATE context
* Trailing commas on all assignments except last

*UPDATE with FROM clause:*

[source,sql]
----
UPDATE table_name t
SET column1 = s.value
FROM source_table s
WHERE t.id = s.id;
----

*UPDATE with subquery assignment:*

[source,sql]
----
UPDATE table_name
SET column_name = (
	SELECT value
	FROM other_table
	WHERE condition
)
WHERE condition;
----

* Newline after `=`, opening `(` on same line as `=`
* Subquery indented +1 from UPDATE context
* Closing `)` dedented to UPDATE context level

*UPDATE with CASE assignment:*

[source,sql]
----
UPDATE table_name
SET column_name =
	CASE
		WHEN condition THEN value
		ELSE other_value
	END
WHERE condition;
----

* Newline after `=`
* CASE indented +1 from UPDATE context
* See section 7.5 for CASE formatting details

=== 11.3 PostgreSQL ON CONFLICT

*ON CONFLICT with DO NOTHING:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2)
VALUES
	(val1, val2)
ON CONFLICT (column)
DO NOTHING;
----

*ON CONFLICT with DO UPDATE:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2)
VALUES
	(val1, val2)
ON CONFLICT (column)
DO UPDATE
SET column = excluded.column
RETURNING *;
----

* ON CONFLICT at INSERT context level
* Constraint specification in parentheses inline: `(column)` or `ON CONSTRAINT name`
* DO UPDATE/DO NOTHING at INSERT context level
* SET follows standard UPDATE rules
* RETURNING at INSERT context level

=== 11.4 MySQL ON DUPLICATE KEY

*ON DUPLICATE KEY UPDATE:*

[source,sql]
----
INSERT INTO table_name
	(col1, col2)
VALUES
	(val1, val2)
ON DUPLICATE KEY
UPDATE col1 = VALUES(col1), col2 = VALUES(col2);
----

* No SET keyword in this syntax variant
* Assignment list follows same ≤3/inline, >3/multiline rules as SET clause
* `VALUES(col)` is a function reference (not a major VALUES clause), stays inline after `=`

=== 11.5 MERGE Statements

*MERGE with conditional actions:*

[source,sql]
----
MERGE INTO target t
USING source s
ON
	(t.id = s.id)
WHEN MATCHED THEN
	UPDATE
	SET t.column = s.column
WHEN NOT MATCHED THEN
	INSERT (col1, col2)
	VALUES
		(s.col1, s.col2);
----

* USING at MERGE context level
* ON at MERGE context level (not indented), condition in parentheses indented +1 from ON
* WHEN MATCHED THEN / WHEN NOT MATCHED THEN at MERGE context level
* UPDATE, INSERT, SET, VALUES indented +1 from MERGE context (inside THEN block)
* Column lists and value tuples stay inline (parenthesized lists)

=== 11.6 DELETE Statements

*Simple DELETE:*

[source,sql]
----
DELETE FROM table_name;
----

* FROM immediately follows DELETE with space (inline)

*DELETE with WHERE:*

[source,sql]
----
DELETE FROM table_name
WHERE condition;
----

* FROM inline with DELETE
* WHERE and subsequent clauses at DELETE context level

*DELETE with subquery in WHERE:*

[source,sql]
----
DELETE FROM table_name
WHERE column IN (
	SELECT id
	FROM other_table
	WHERE condition
);
----

* Subquery formatted according to standard subquery rules (section 7.2)

*PostgreSQL DELETE with USING:*

[source,sql]
----
DELETE FROM employees e
USING departments d
WHERE e.department_id = d.id
	AND d.closed = TRUE;
----

* FROM inline with DELETE
* USING at DELETE context level (newline before USING)
* WHERE at DELETE context level
* Multiple conditions follow standard AND/OR formatting rules

=== 11.7 TRUNCATE Statements

*Simple TRUNCATE:*

[source,sql]
----
TRUNCATE TABLE table_name;
----

* TABLE keyword follows TRUNCATE with space (inline)
* All keywords stay on single line for simple TRUNCATE

*TRUNCATE with options:*

[source,sql]
----
TRUNCATE TABLE table_name RESTART IDENTITY CASCADE;
----

* RESTART, IDENTITY, CASCADE stay inline with TRUNCATE TABLE
* All keywords on single line (no newlines)

'''

== 12. Window Functions

Window functions use the `OVER` clause with optional `PARTITION BY`, `ORDER BY`, and frame specification.

=== 12.1 OVER Clause Structure

*Basic window function:*

[source,sql]
----
SELECT
    department,
    name,
    salary,
    rank() over (
        partition by department
        order by name, salary desc
    ) as salary_rank
FROM employees
----

* OVER keyword requires space before opening `(` (see section 6.6)
* Opening `(` triggers newline and indent (+1 from current context)
* Contents of OVER clause are indented relative to the opening
* Closing `)` dedented to match the line that opened the OVER clause

=== 12.2 PARTITION BY

[source,sql]
----
rank() over (
    partition by department
    ...
)
----

* PARTITION BY follows same formatting as other BY clauses
* List items follow standard rules: ≤3 inline, >3 multiline

=== 12.3 ORDER BY Inside OVER

**Context-aware terminators:**

* ORDER BY inside OVER clause has different terminators than query-level ORDER BY
* Inside OVER: terminators are `ROWS`, `RANGE`, `GROUPS`, `RPAREN`
* Query-level: terminators are `OFFSET`, `FETCH`, `LIMIT`

*Example with ≤3 items (inline):*

[source,sql]
----
rank() over (
    partition by department
    order by name, salary desc
)
----

*Example with >3 items (multiline):*

[source,sql]
----
dense_rank() over (
    partition by region
    order by
        year desc,
        quarter desc,
        month desc,
        day desc
)
----

=== 12.4 Frame Clauses

Frame clauses (`ROWS`, `RANGE`, `GROUPS`) follow the ORDER BY:

[source,sql]
----
sum(amount) over (
    partition by account_id
    order by transaction_date
    rows between unbounded preceding and current row
)
----

* Frame clause keywords at same indent level as PARTITION BY and ORDER BY
* Frame specification stays on same line (BETWEEN ... AND ...)
