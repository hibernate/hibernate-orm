The term bootstrapping refers to initializing and starting a software component.
In Hibernate, we are specifically talking about the process of building a fully functional `SessionFactory` / `EntityManagerFactory` instance.

[TIP]
====
This chapter mentions a number of specific configuration settings.  Be sure to check out the
<<appendices/SettingsReference.adoc#settings>> section as well for documentation of each available setting.
====

[[bootstrap-process]]
=== Bootstrap Process

Every supported bootstrap entry point follows the same basic shape:

* define mapping information;
* define configuration settings;
* apply integrations; and
* build a `SessionFactory` / `EntityManagerFactory`.

Mapping information may come from:

* annotated classes;
* managed class names;
* package-level annotations;
* XML mapping documents;
* `persistence.xml` declarations; or
* classpath scanning, where available.

Settings may come from the selected bootstrap API, persistence-unit
descriptors, integration-provided settings, `hibernate.properties`, system
properties, or other environment-specific sources.  The exact precedence
depends on the entry point and deployment environment.

Hibernate also discovers a number of optional integrations using Java service
loading.  See <<bootstrap-java-service-loadable>> for the service-loadable
contracts that can affect bootstrap.

The supported entry points collect mapping information, settings, and integrations, and
Hibernate handles the internal bootstrap pipeline.


[[bootstrap-entrypoints]]
=== Supported Entry Points

Hibernate supports several bootstrap entry points for building the factory.
Since the Hibernate `SessionFactory` is also an implementation of the
Jakarta Persistence `EntityManagerFactory`, these approaches are interchangeable.

* <<PersistenceUnitInfo>>
* <<Persistence>>
* <<PersistenceConfiguration>>
* <<HibernatePersistenceConfiguration>>


[NOTE]
====
`org.hibernate.cfg.Configuration` remains available (though deprecated) as a
native compatibility facade for existing applications.  Existing
`Configuration` usage should be migrated to
`PersistenceConfiguration` or `HibernatePersistenceConfiguration`.
====

[[PersistenceUnitInfo]]
=== PersistenceUnitInfo

`jakarta.persistence.spi.PersistenceUnitInfo` is the standard way to bootstrap
Hibernate in a container.  Jakarta Persistence calls this approach "EE
bootstrapping".

The application provides `META-INF/persistence.xml` for configuration.  The
container parses the configuration and builds a `PersistenceUnitInfo`, which it
passes to Hibernate (as the `PersistenceProvider`).  Hibernate then builds a
factory based on the `PersistenceUnitInfo` and integration settings provided by
the container.
In these container environments, an `EntityManager` may be dependency injected
via `@PersistenceContext` or located by JNDI.  In most cases, the lifecycle of such an injected
`EntityManager` is managed by the container.

[[HibernatePersistenceConfiguration]]
=== HibernatePersistenceConfiguration

`org.hibernate.jpa.HibernatePersistenceConfiguration` is Hibernate's extension
of `jakarta.persistence.PersistenceConfiguration`.  It produces a Hibernate
`SessionFactory`, which is also a Jakarta Persistence `EntityManagerFactory`,
and exposes Hibernate-specific conveniences for settings, mapping sources,
scanning, schema tooling, caching, and other native features.

Use this entry point when the application wants programmatic Jakarta
Persistence bootstrap with Hibernate-specific conveniences, or when the
application wants to work with a native Hibernate `SessionFactory` directly.

[[bootstrap-java-service-loadable]]
=== Java Service Loading

Many Hibernate bootstrap integration contracts are discoverable using the
standard Java `ServiceLoader` facility.  A contract annotated with
`org.hibernate.service.JavaServiceLoadable` may be implemented by a library or
application and listed in a `META-INF/services/<fully-qualified-contract-name>`
file.  The service file contains the fully-qualified names of implementation
classes, one per line.

The service-loadable contracts most directly affecting the bootstrap process include:

.`@JavaServiceLoadable` bootstrap contracts
[cols="35,65", options="header"]
|===
| Contract
| Purpose

| `org.hibernate.boot.model.TypeContributor`
| Contributes custom basic types, Java type descriptors, JDBC type descriptors, or related type-system registrations.

| `org.hibernate.boot.model.FunctionContributor`
| Contributes HQL/SQL functions to the function registry.

| `org.hibernate.boot.spi.AdditionalMappingContributor`
| Contributes additional mapping metadata during metadata building.

| `org.hibernate.boot.scan.spi.ScanningProvider`
| Provides classpath scanning support for discovered managed classes and mapping files.

| `org.hibernate.resource.beans.container.spi.BeanContainer`
| Supplies managed-bean integration for CDI or another bean container.

| `org.hibernate.boot.pipeline.spi.SessionFactoryProducer`
| Advanced internal extension point for replacing the SessionFactory production pipeline.
|===

[NOTE]
====
Some service-loadable contracts are intended for ordinary application or
library integration, while others are low-level replacement points for
containers, modules, or Hibernate extensions.  This section only lists the
contracts that directly affect the bootstrap process; other service-loadable
contracts affect runtime services, dialect selection, schema tooling,
statistics, or event monitoring.  Prefer the most specific contract for the
customization you need, and use settings for ordinary configuration choices.

See `@JavaServiceLoadable` and its uses to identify which services are loadable.
====
