= Hibernate ORM Jakarta Data TCK runner module

This module pulls the Jakarta Data TCK tests, generates repository implementations
via `hibernate-processor`, and runs them in standalone mode using Weld CDI and a database
(need to start it with the `db.sh` script as in other cases).

The companion classes (`$` classes from https://github.com/jakartaredhat/hibernate-data-tck/tree/main/tools[jakarta.data-tools])
are committed as sources, while `_Repo` and `Repo_` classes are generated by annotation processing at compile time.

== Running the Data TCK

With H2 (default, no database setup required):

[source,bash]
----
./gradlew :hibernate-tck-data-runner:test -Prundatatck=true
----

With PostgreSQL:

[source,bash]
----
./db.sh postgresql
./gradlew :hibernate-tck-data-runner:test -Prundatatck=true -Pdb=postgresql
----

For any other DB -- just lookup the supported `db` values in the script.

== Testing with a locally built Jakarta Data

If you are developing changes to the Jakarta Data spec and want to test against the TCK
without publishing upstream, install the Jakarta Data artifacts to your local Maven repository
(`mvn clean install -DskipTests` in the Jakarta Data project) and then run:

[source,bash]
----
./gradlew :hibernate-tck-data-runner:test -Prundatatck=true -DenableMavenLocalRepo=true
----

== Testing from the IDE

* Go to `org.hibernate.orm.jakarta.data.tck.runner.JakartaDataTckTestRunner`
* Run it to create a run configuration. IDE might prevent test execution unless `rundatatck=true` is set.
Do it either in the run configuration or in the `gradle.properties`
* The tests will use H2 by default, but can also be changed in the same `gradle.properties` by setting the `db` to some
other supported value.

== Test structure

Each test class uses the `@DataTck` annotation to declare its domain and repository classes:

[source,java]
----
@DataTck(
    domainClasses = { Country.class },
    repositoryClasses = { _Countries.class }
)
public class StandaloneFirstTests extends FirstTests {
}
----

The `DataTckExtension` (which extends Weld's JUnit extension) handles:

* Creating an `EntityManagerFactory` from the declared domain classes
* Registering repository implementations as CDI beans
* Providing persistence unit/context/agent factories to Weld

== Challenge exclusions

To exclude (challenge) a test, add its pattern to `@ExcludeClassNamePatterns`
in `JakartaDataTckTestRunner`.

== Limiting executed tests

To run a single test class, modify `@IncludeClassNamePatterns` in `JakartaDataTckTestRunner`:

[source,java]
----
@IncludeClassNamePatterns({".*StandaloneEntityTests"})
----

Or to run a single test method, use `@SelectMethod`:

[source,java]
----
@SelectMethod(type = StandaloneEntityTests.class, name = "testFindFirst")
----
